# Publish graphitron-tree-sitter-natives jar; retire vendored C build in graphitron-lsp

`graphitron-lsp` today compiles the vendored tree-sitter runtime (`v0.26.0`) plus the bkegley tree-sitter-graphql grammar (`5e66e96`, both upstreams dormant) into a per-platform shared library on every `mvn install`. Three host-activated Maven profiles in `graphitron-lsp/pom.xml:86-197` invoke `src/main/native/build-native.sh` to produce `target/classes/lib/<os>-<arch>/libtree-sitter-graphql.{so,dylib}`; the build script rejects non-Linux non-Darwin hosts, so Windows is unimplemented (`BundledLibraryLookup.java:91-100` throws `UnsupportedOperationException`). The shape costs every developer and CI a C toolchain plus a non-trivial recompile on each build, ships only the platform the build is currently running on (CI builds `linux-x86_64` only; macOS rides on hope), and leaves R89 ("macOS / Windows CI verification") open as the multi-platform gap.

The chosen shape is a new published Maven artifact `no.sikt:graphitron-tree-sitter-natives` whose jar carries one prebuilt **grammar** library per supported platform (`tree-sitter-graphql.{so,dylib,dll}`, four binaries total) under the `lib/<os>-<arch>/...` layout `BundledLibraryLookup` already resolves. The artifact's version stream tracks the tree-sitter ABI the grammar is built against (`<runtime-version>-<build-n>`, e.g. `0.26.0-1`); built by a GitHub Actions matrix on native runners that run upstream's `tree-sitter build` CLI against the vendored grammar; published to Maven Central via `central-publishing-maven-plugin` configuration inlined into the module's standalone pom (matching the shape the rewrite parent's `release` profile uses at `graphitron-rewrite/pom.xml:262-337`, but not inherited from it). `graphitron-lsp` consumes the published jar as one `<dependency>`; `libtree-sitter` itself is **not** bundled. Instead, graphitron-lsp relies on jtreesitter's documented library-loading contract to find an OS-installed `libtree-sitter` (`brew install tree-sitter` on macOS, distro package or source build on Linux, `vcpkg install` or upstream build on Windows); the system dependency is documented in the LSP's setup docs along with the per-platform loader-path wiring each install layout needs, and the loader fails fast with an actionable error when it is missing. The on-build C compile, the three `build-native-*` profiles, and the vendored `src/main/native/` tree all delete from graphitron-lsp; the grammar source moves to the natives module, the runtime sources are dropped entirely.

## Design rationale

Two earlier drafts of this plan died on the same misunderstanding: that this module is in the business of shipping `libtree-sitter` itself.

The **first attempt** ("unity build") committed the natives module + a release workflow that ran `tree-sitter build` and treated its output as a single library containing both the grammar entry point and the tree-sitter runtime. The workflow's symbol-export assertion caught it: `tree-sitter build` in 0.26.x compiles the grammar's `parser.c` only and does not link the runtime, so the binaries had `tree_sitter_graphql` but no `ts_*` symbols. The spec was reverted to draft.

The **second attempt** ("download upstream runtime + compile grammar; ship two libraries per platform") proposed a Phase 1 workflow step that downloads `libtree-sitter-{platform}.{tar.gz,zip}` from upstream's GitHub release assets, SHA256-pins it, and merges it with the locally-compiled grammar into a ten-entry jar (two binaries per platform). On first investigation of Phase 1 the asset never materialized: upstream's tree-sitter release manifests (verified against `v0.26.0`, `v0.25.10`, `v0.26.9`) contain the CLI binary (`tree-sitter-{platform}.gz`), CLI zip variants, the WASM `web-tree-sitter` bundle, and source tarballs — no prebuilt `libtree-sitter` runtime archive. The "we'll just download it" premise was false.

The **corrected understanding** comes from reading jtreesitter's `package-info.java` directly. It documents three loading mechanisms for the runtime: (1) `libtree-sitter` on the OS search path or `java.library.path`, picked up automatically by `SymbolLookup.libraryLookup`; (2) `java.library.path` only, picked up by `SymbolLookup.loaderLookup`; (3) a custom `NativeLibraryLookup` SPI registered under `META-INF/services/`, "for example, to load libraries from inside a JAR file". The disassembled `internal.ChainedLibraryLookup` confirms the chain: SPI services composed with `.or`, then a `System.loadLibrary("tree-sitter")` / `SymbolLookup.libraryLookup(System.mapLibraryName("tree-sitter"), arena)` fallback, then `Linker.defaultLookup`. **jtreesitter already knows how to find `libtree-sitter` from the system; we do not need to ship it.** Per-language grammar libraries are caller-provided, passed to `Language.load(symbolLookup, "tree_sitter_<lang>")` explicitly. The ecosystem's standing convention is to split: the runtime is generic and lives once on the system, the grammar lib is per-language and ships with the language's binding.

Both prior drafts conflated these two libraries. The "we have to ship `libtree-sitter` ourselves" framing — and all the matrix complexity that followed (5×2 binaries, SHA-pinned runtime downloads, two-lookup composition logic in `BundledLibraryLookup`) — was solving a problem jtreesitter already solves. The correct shape is **grammar-only**: the natives module ships exactly the per-platform grammar library jtreesitter consumers cannot get from anywhere else, and the runtime is the OS's problem (the way every other jtreesitter consumer treats it).

The remaining design choices fall out cleanly. The bonede route stays rejected for the same Windows-DLL reason as before (verified against `io.github.bonede:tree-sitter:0.26.6` — Linux exports `ts_*` symbols, the Windows DLL exports only `Java_org_treesitter_*` JNI wrappers), so we still own the grammar build. Committing binaries directly in graphitron-lsp couples the artifact's release cadence to graphitron's, which doesn't match how often the grammar's C source actually changes. A separate repo adds cross-repo PR coordination for source updates that happen roughly never. Same-repo module with detached `<version>` is the middle.

## Design decisions

- **Module location:** `graphitron-rewrite/graphitron-tree-sitter-natives/`.
- **Maven coordinates:** `no.sikt:graphitron-tree-sitter-natives`.
- **Version scheme:** `<tree-sitter-runtime-version>-<build-n>`. First release `0.26.0-1`. The `<runtime-version>` portion is the tree-sitter ABI the grammar was built against (currently `0.26.0`); a consumer with an OS-installed `libtree-sitter` older than this will fail at `Language.load` with a clear ABI-mismatch error. Bump `<build-n>` for any change to the grammar binaries (grammar update, build-flag change, recompile). Bump `<runtime-version>` when we re-target the grammar to a newer tree-sitter ABI (which is when the `tree-sitter` CLI's bundled runtime moves under us). Snapshots not used.
- **Standalone pom, not in parent reactor, no parent inheritance.** The module's pom is fully self-contained: explicit `<groupId>no.sikt</groupId>`, explicit `<version>0.26.0-1</version>`, no `<parent>` reference. The release plumbing (`maven-gpg-plugin`, `central-publishing-maven-plugin`, `maven-source-plugin`, `maven-javadoc-plugin`, `maven-deploy-plugin`) is inlined into the module's pom. Rationale: the rewrite parent on trunk runs at `10-SNAPSHOT`, which is not deployed to Maven Central; inheriting would publish a natives jar whose parent reference is unresolvable for any Central consumer and lock the natives release cadence behind the next 10.x rewrite release. Standalone shape costs ~60 lines of inlined plugin config and removes the coupling. The parent pom's `<modules>` list also does *not* include the natives module, keeping `mvn install -f graphitron-rewrite/pom.xml -Plocal-db` zero-cost. `graphitron-lsp` depends on the published artifact via Maven Central (or local m2 cache), not on a reactor-built sibling.
- **Build tool:** the upstream `tree-sitter` CLI (`tree-sitter build`) compiles the vendored grammar's `parser.c` on each matrix runner, producing `tree-sitter-graphql.{so,dylib,dll}`. The CLI handles MSVC + POSIX compiler invocation uniformly, including Windows `__declspec(dllexport)` for the grammar's single exported function. Parser-only output is exactly what we want here — this is the dimension where the first attempt was wrong (it wanted a unity build) and the corrected understanding makes it right.
- **Matrix entries:** four — `linux-x86_64`, `linux-aarch64`, `macos-aarch64`, `windows-x86_64`. Exact GitHub-hosted runner labels are picked at implementation time. `linux-aarch64` and `windows-x86_64` are gains over current state (graphitron-lsp ships `linux-x86_64` only today and throws on Windows). `macos-x86_64` was in the original matrix but dropped during Phase 1 dry-run: Sikt graphitron-lsp developers all run M1 or newer, the macos-13 (Intel) GitHub-hosted runner is the slowest queue on the platform, and the LSP is internal-only per the "Out of scope" list — so Intel-Mac coverage costs significant CI time without a known consumer.
- **Jar layout:** one grammar binary per platform under `lib/<os>-<arch>/`. Four entries total: `tree-sitter-graphql.{so,dylib,dll}` (POSIX files prefixed `lib`, Windows file unprefixed per platform convention — `libtree-sitter-graphql.so`, `libtree-sitter-graphql.dylib`, `tree-sitter-graphql.dll`). Jar also includes `META-INF/UPSTREAM.md` documenting the grammar commit + the tree-sitter ABI version it was built against.
- **Grammar source:** vendored at `graphitron-tree-sitter-natives/src/main/native/grammars/graphql/` (`parser.c` + `tree_sitter/parser.h` + `LICENSE` + `UPSTREAM.md`, moved verbatim from graphitron-lsp). Pinned to bkegley commit `5e66e96`.
- **Tree-sitter runtime (`libtree-sitter`):** not bundled, not built, not vendored. Documented as a system dependency of `graphitron-lsp`. The LSP setup docs list the install commands per platform; the loader's failure path on missing-libtree-sitter surfaces the same instructions in the runtime error message.
- **Workflow trigger:** `workflow_dispatch` only (manual). No automatic builds on push. Cutting a release is a deliberate human-driven action: bump the version in the pom, push, trigger the workflow. Tagging is optional; the artifact is identified by Maven version.
- **Bootstrap path:** four phases. Phase 1 amends the natives module + release workflow so the published artifact is grammar-only, then lands the change without touching graphitron-lsp. Phase 2 triggers the workflow and verifies `0.26.0-1` resolves from Central. Phase 3 swaps graphitron-lsp to depend on the published jar, deletes the vendored sources + Maven profiles, and lands the documentation that names `libtree-sitter` as a setup prerequisite. Phase 4 (rework after the first In Review pass) adds the spec-named error-translation unit test, distinguishes "missing libtree-sitter" from "found but too old" by probing for `ts_language_abi_version`, and aligns the operator-facing install hints with the docs table.

## Implementation

### Phase 1 — natives module + release workflow (shipped)

Workflow amendments: dropped the `ts_parser_new` symbol-export check on both POSIX `nm` and Windows `dumpbin` branches (grammar lib is parser-only by design; the runtime symbol is intentionally absent). Added `libtree-sitter` install steps to the post-deploy-verify job (`apt install libtree-sitter0` on Linux, `brew install tree-sitter` on macOS, `vcpkg install tree-sitter:x64-windows` on Windows with the resulting `bin` directory appended to `GITHUB_PATH` so the verifier JVM can load it). The inlined `Verify.java` was already grammar-only-load shape; dropped one unused import. Header and build-step comments rewritten to drop the unity-build framing. Module `UPSTREAM.md`, grammar `UPSTREAM.md`, module `pom.xml` description + leading comment, and `graphitron-rewrite/docs/tree-sitter-natives-release.adoc` rewritten to match the grammar-only + OS-runtime story.

Open follow-up for Phase 2: the Windows `vcpkg install tree-sitter:x64-windows` path needs validation on the actual `windows-latest` runner during the first release. If vcpkg's registry doesn't carry it or the install fails, the fallback per the spec is a pinned upstream source-tarball build cached out-of-band. The implementation took the vcpkg branch as the simpler default.

### Phase 2 — first release (shipped)

`no.sikt:graphitron-tree-sitter-natives:0.26.0-1` was published to Maven Central via the manually-triggered `tree-sitter-natives-release` workflow. The build matrix and the post-deploy load+parse matrix succeeded on the four supported platforms (`linux-x86_64`, `linux-aarch64`, `macos-aarch64`, `windows-x86_64`). The artifact resolves into a fresh local m2 without unresolvable-parent failures, including the Windows `vcpkg install tree-sitter:x64-windows` path. Several workflow refinements landed during this phase: dry-run input, gunzip-suffix fix, Windows asset name (`.zip` → `.gz`), Node 24 artifact-action bumps, jar-layout assert pinned to the main artifact (not the source/javadoc classifier jars), exclusion of bare directory entries from the assert, dropping `macos-x86_64` from the matrix.

### Phase 3 — graphitron-lsp cutover (shipped)

`graphitron-lsp` now depends on `no.sikt:graphitron-tree-sitter-natives:0.26.0-1` and the vendored runtime + grammar tree is gone. `graphitron-lsp/pom.xml` lost the three `build-native-*` profiles and the `exec-maven-plugin` plumbing; `src/main/native/` was removed wholesale. `BundledLibraryLookup.resolveResourcePath` switched to the four-platform set (`linux-x86_64`, `linux-aarch64`, `macos-aarch64`, `windows-x86_64`) with the Windows DLL unprefixed per convention; the previous `UnsupportedOperationException` branch is gone, replaced by a generic unsupported-platform message. `BundledLibraryLookup.get` is unchanged (single-library lookup; jtreesitter's chain handles the runtime). `GraphqlLanguage` now wraps `Language.load` in a helper that detects the "missing libtree-sitter" failure shape and surfaces a startup error naming the per-platform install command. `NativeBuildSmokeTest` was renamed to `NativeLibraryBundleTest` with one `@EnabledOnOs` method per supported platform (four total). `IncrementalParseTest` and `TreeSitterSmokeTest` docstrings dropped the `bonede` references. `rewrite-build.yml` now builds `libtree-sitter v0.26.9` from source so the linux-x86_64 method runs green in CI. `graphitron-rewrite/docs/getting-started.adoc` gained a "Native runtime dependency" section with a per-platform table covering install command AND the loader-path wiring each platform needs. `actions/upload-artifact@v4` and `actions/download-artifact@v4` in `rewrite-build.yml` and `preview-docs.yml` were bumped off Node 20 (`v7` upload, `v8` download), aligning with the tree-sitter-natives-release workflow.

Discovered during Phase 3 dry-run: the spec's working assumption ("jtreesitter's chain finds the OS-installed `libtree-sitter` automatically") only holds on traditional FHS layouts. Two gaps:

- **Debian/Ubuntu LTS apt is incompatible.** `libtree-sitter0` is pinned to `0.20.x` on bookworm / 24.04, predates `ts_language_abi_version` that jtreesitter 0.26 needs. Resolution: source-build from upstream (`v0.26.9`). CI does this; user docs include the same snippet for Debian/Ubuntu developers.
- **Modern package managers install outside the JVM's default loader path.** Apple-silicon Homebrew lands `libtree-sitter.dylib` in `/opt/homebrew/lib` (not in dyld's default fallback). vcpkg lands `tree-sitter.dll` under `<VCPKG_ROOT>/installed/x64-windows/bin` (only on `PATH` if the user wired it). NixOS lands it under `/nix/store/.../lib` (not on any default path; content-hashed so not statically probable).

The Linux gap is a CI-config concern; the modern-package-manager gap is partly a code concern. `BundledLibraryLookup` was extended to probe well-known install prefixes (Homebrew `/opt/homebrew/lib` and `/usr/local/lib` on macOS; vcpkg's `<VCPKG_ROOT|VCPKG_INSTALLATION_ROOT>/installed/x64-windows/bin` and the default `C:\vcpkg\...` layout on Windows; `/usr/local/lib` on Linux) and composes a found `libtree-sitter` onto the grammar SPI lookup via `SymbolLookup.or`. The probe is best-effort and silent on miss: jtreesitter's normal loader chain still runs, NixOS still needs the shellHook, non-default install locations still need `JAVA_TOOL_OPTIONS`, but a vanilla `brew install tree-sitter` on macOS or `vcpkg install tree-sitter:x64-windows` on Windows now works with no env-var wiring.

The design itself (grammar-only natives jar, jtreesitter chain handles the runtime) is sound and was verified end-to-end against `libtree-sitter 0.25.10` on NixOS and `0.26.x` from Apple-silicon Homebrew.

### Phase 4 — In Review → Ready rework (shipped)

Three review findings on the first In Review pass: spec-named unit test for the libtree-sitter-missing error path did not ship; the Debian/Ubuntu install hint in `GraphqlLanguage.missingRuntimeMessage` pointed at the broken apt `libtree-sitter0` package; phase 3 also missed the implementer-final `In Progress → In Review` status flip. All three addressed:

- `GraphqlLanguage` was refactored to make its error-translation surface unit-testable: `looksLikeRuntimeFailure`, `missingRuntimeMessage`, `tooOldRuntimeMessage`, and the new `classifyInstalledRuntime` (returning `MODERN` / `TOO_OLD` / `MISSING`) are all package-private static. `loadOrExplain` now classifies the runtime on failure and picks the message accordingly. `classifyInstalledRuntime` walks a probe-path superset (the `BundledLibraryLookup` set plus apt's `/usr/lib/<arch>-linux-gnu/libtree-sitter.so.0` locations) and opens each candidate via `SymbolLookup.libraryLookup` to check for `ts_language_abi_version` — present means modern, absent means too old (the apt 0.20.x signature). The probe is best-effort: when the user's runtime lives somewhere we don't probe, the classifier returns `MISSING` (or `MODERN` for a probe hit elsewhere) and the message defaults to install-instructions, which is still actionable.
- `tooOldRuntimeMessage` names the apt package by name, names the `ts_language_abi_version` symbol, and points at the per-platform upgrade path; `missingRuntimeMessage` now matches the docs table (pacman/dnf for the easy distros, source-build snippet for Debian/Ubuntu). Both messages link `https://graphitron.sikt.no/getting-started.html#native-runtime-dependency`; an explicit `[#native-runtime-dependency]` anchor was added in `getting-started.adoc` so the URL doesn't drift if the heading is reworded.
- `GraphqlLanguageErrorTranslationTest` pins all of the above: the classifier walks cause chains, recognises the ABI-symbol failure shape, and ignores unrelated errors; per-OS `@EnabledOnOs` assertions pin the install + upgrade hints; the apt 0.20.x case is named explicitly on Linux.

## Tests

- Natives release workflow asserts jar layout in the package stage: exactly four `lib/<os>-<arch>/` entries (one grammar lib per platform), expected file names per platform. Each build-matrix runner asserts the grammar lib exports `tree_sitter_graphql`. Each post-deploy-matrix runner asserts the published jar load+parses on its platform, after installing the OS `libtree-sitter` (apt on Linux, brew on macOS, vcpkg or pinned download on Windows).
- `NativeLibraryBundleTest.java` in graphitron-lsp covers each of the four platforms with `@EnabledOnOs`, loads the bundled grammar lib through `BundledLibraryLookup`, parses a trivial GraphQL document, asserts root node + no errors. The test implicitly exercises the jtreesitter chain finding `libtree-sitter` on the OS. In regular CI (`rewrite-build.yml`, currently `ubuntu-latest`) only the linux-x86_64 method executes per PR; the other three `@EnabledOnOs` methods run on their respective platforms in the natives release workflow's post-deploy matrix. `rewrite-build.yml`'s ubuntu-latest step source-builds `libtree-sitter v0.26.9` (apt's `libtree-sitter0` is 0.20.x and predates `ts_language_abi_version`); without it the test fails with the LSP's own libtree-sitter-missing error message, which is itself a useful negative signal but not what we want as the green-path CI behaviour. `rewrite-build.yml` does not gain a multi-OS matrix as part of this item: the binaries don't change per PR (they're a published artifact), so per-PR multi-OS coverage would mostly re-verify a stable input.
- Windows verification has no pre-existing test fixture; the new `windows-x86_64` `NativeLibraryBundleTest` method establishes it, exercised at natives-release time and the first time a developer runs the LSP on Windows with a system `libtree-sitter`.
- `GraphqlLanguageErrorTranslationTest` (graphitron-lsp) pins the surfaces graphitron-lsp owns around jtreesitter's runtime failure modes: cause-chain classification, the missing-runtime install hint per platform, and the too-old upgrade hint per platform (the latter naming Debian/Ubuntu's apt `libtree-sitter0` 0.20.x explicitly). The load-and-parse coverage above exercises jtreesitter's chain, not our error handling; this test pins the wrapper text directly so the docs-link and install-command contract can be reasoned about without a native dependency on the test runner.
- No pipeline-tier or execution-tier coverage changes: the natives jar is a build-time substitution, everything downstream sees identical behaviour.

## Roadmap entries

- **R89** (`lsp-native-build-multiplatform-ci`) is subsumed and deleted as part of this item's `In Review → Done`. Its core concern (verify macOS / Windows actually work) survives as the three additional `@EnabledOnOs` methods on `NativeLibraryBundleTest`; the "stand up a CI matrix on every build" framing is moot once the grammar binary comes from a separate release pipeline.

## Acceptance

- `no.sikt:graphitron-tree-sitter-natives:0.26.0-1` exists on Maven Central with exactly four `lib/<os>-<arch>/tree-sitter-graphql.{so,dylib,dll}` entries (POSIX `lib` prefix on Linux/macOS, no prefix on Windows). The artifact resolves into a fresh local m2 without unresolvable-parent failures.
- `mvn install -f graphitron-rewrite/pom.xml -Plocal-db` from a clean checkout no longer invokes `cc`, `cl.exe`, or any C toolchain. The build completes without `src/main/native/` existing in `graphitron-lsp/`. The `rewrite-build.yml` CI pipeline installs `libtree-sitter0` via apt as a one-line setup step.
- The natives release workflow's post-deploy load+parse matrix passes on all four platforms for the `0.26.0-1` release; the matrix exercises the published grammar binary against an OS-installed `libtree-sitter` (apt on Linux runners, brew on macOS runners, vcpkg or pinned download on Windows).
- A developer machine running `mvn graphitron:dev` without `libtree-sitter` installed sees a startup error that names the missing library and lists the per-platform install command, not an opaque `UnsatisfiedLinkError`.
- R89 is deleted from `graphitron-rewrite/roadmap/`.

## Non-goals

- GraalVM native image support for graphitron-lsp.
- Bundling, building, or otherwise shipping `libtree-sitter` ourselves. Earlier drafts treated this as in-scope; it is explicitly out. The runtime is the consumer OS's responsibility, per jtreesitter's package-info contract.
- Runtime auto-download of the grammar binary from Maven Central at first LSP use; the jar dependency ships it on the classpath in the standard way.
- Supporting tree-sitter grammars other than the LSP's current single GraphQL grammar.
- Consuming the natives jar from any module other than graphitron-lsp.
- Adding `linux-arm` (32-bit), `powerpc64`, or `macos-x86_64`: no consumer. Intel-Mac coverage was in the original matrix but dropped during Phase 1 dry-run for the reasons noted in **Matrix entries** above.
- Solving the Windows `libtree-sitter` install story upstream of our docs. We document the install path; we do not own a Windows package for it. If users hit friction, the answer is to upstream a vcpkg / scoop / chocolatey port of `libtree-sitter`, not to walk back the design here.
