# Publish graphitron-tree-sitter-natives jar; retire vendored C build in graphitron-lsp

`graphitron-lsp` today compiles the vendored tree-sitter runtime (`v0.26.0`) plus the bkegley tree-sitter-graphql grammar (`5e66e96`, both upstreams dormant) into a per-platform shared library on every `mvn install`. Three host-activated Maven profiles in `graphitron-lsp/pom.xml:86-197` invoke `src/main/native/build-native.sh` to produce `target/classes/lib/<os>-<arch>/libtree-sitter-graphql.{so,dylib}`; the build script rejects non-Linux non-Darwin hosts, so Windows is unimplemented (`BundledLibraryLookup.java:91-100` throws a pointed `UnsupportedOperationException`). The current shape costs every developer and CI a C toolchain plus a non-trivial recompile on each build, ships only the platform the build is currently running on (CI builds `linux-x86_64` only; macOS and Windows ride on hope), and leaves R89 ("macOS / Windows CI verification for graphitron-lsp native build") open as the multi-platform gap.

The proposed shape is a new published Maven artifact `no.sikt:graphitron-tree-sitter-natives` whose jar carries prebuilt binaries for every supported platform under the existing `lib/<os>-<arch>/...` layout `BundledLibraryLookup` already resolves. The artifact is produced by a separate workflow with its own version stream tracking upstream tree-sitter (e.g. `0.26.0-1`, `0.26.0-2`), built by a GitHub Actions matrix on native runners (`ubuntu-latest`, `macos-latest`, `windows-latest`, plus arm64 where available) using the upstream `tree-sitter build` CLI rather than ad-hoc `cc` invocations. Each matrix job produces one merged runtime+grammar shared library; a downstream job collects the five workflow artifacts into a single jar and publishes via the existing Sonatype central-publishing-maven-plugin configuration (`graphitron-rewrite/pom.xml:262-337`). `graphitron-lsp` then depends on the artifact as a single line, deleting `src/main/native/` and the three `build-native-*` profiles.

Why this shape over alternatives explored in design discussion: (a) committing binaries directly in `graphitron-lsp` works mechanically but ties the artifact's version to graphitron's release cadence, which doesn't match how often the C sources actually change (roughly never); a separate version stream pinned to upstream tree-sitter is the honest version semantics. (b) The bonede prebuilt-binaries route (`io.github.bonede:tree-sitter:0.26.6` + `io.github.bonede:tree-sitter-graphql:master-a`) was probed end-to-end with jtreesitter 0.26 and works for Linux x86_64; symbol-table inspection confirms it would work on macOS x86_64/aarch64 and Linux aarch64 too, but the bonede Windows DLL exports only `Java_org_treesitter_*` JNI wrappers (zero bare `ts_*` runtime symbols) because MSVC defaults to no-export and the upstream tree-sitter C sources don't carry `__declspec(dllexport)` annotations. Building ourselves with `tree-sitter build` produces correctly-exported DLLs on Windows. (c) A separate repo (`sikt-no/graphitron-tree-sitter-natives`) was considered for clean isolation but adds cross-repo PR coordination for a C source update that happens roughly never; a same-repo module with detached `<version>` and a `workflow_dispatch`-triggered release pipeline captures the same versioning independence without the second repo.

Scope:

- New module `graphitron-rewrite/graphitron-tree-sitter-natives/` (or similar) with its own `<version>` not inheriting from the rewrite parent. Module produces a jar containing `lib/<os>-<arch>/libtree-sitter-graphql.{so,dylib,dll}` for `linux-x86_64`, `linux-aarch64`, `macos-x86_64`, `macos-aarch64`, `windows-x86_64`. No Java code; resources-only.
- New GitHub Actions workflow `.github/workflows/tree-sitter-natives-release.yml`, triggered on `workflow_dispatch` (and on tag push `natives-v*`). Matrix: five native runners, each downloads the upstream `tree-sitter` CLI from `https://github.com/tree-sitter/tree-sitter/releases`, invokes `tree-sitter build` against the bkegley grammar (vendored into the new module, or fetched at a pinned commit), uploads the resulting binary as a workflow artifact. A package job downloads all five, runs `mvn deploy -Prelease` from the natives module, the existing GPG signing + central-publishing path picks up automatically.
- `graphitron-lsp/pom.xml`: drop the three `build-native-*` profiles and the `exec-maven-plugin` invocations; add a single `<dependency>` on `no.sikt:graphitron-tree-sitter-natives` at the published version.
- Delete `graphitron-lsp/src/main/native/` (vendored tree-sitter runtime + grammar + `build-native.sh`).
- `BundledLibraryLookup.resolveResourcePath` extended to recognise `linux-aarch64` and `windows-x86_64` (the latter resolving to `tree-sitter-graphql.dll`). The merged-lib assumption (`SymbolLookup.libraryLookup(extractOnce(), arena)` returns one lookup for both runtime + grammar) holds because `tree-sitter build` produces a unified binary; no split-lib refactor needed.
- Update tests that reference the old build path: `NativeBuildSmokeTest.java` rephrases from "the build produced a library" to "the bundled binary loads and parses"; existing `@EnabledOnOs` gating extends to Windows. Docstrings in `IncrementalParseTest.java:15` and `TreeSitterSmokeTest.java:11` no longer need to mention bonede.
- Subsumes R89: with binaries built in a controlled matrix once per upstream bump, the "CI matrix on every build" framing R89 imagines is moot. Delete R89 as part of this item's landing; its substantive concerns (verify macOS/Windows actually work) survive as smoke-test methods on `NativeBuildSmokeTest`.
- Documentation: a short section in `graphitron-rewrite/docs/` covering how to cut a new natives release (when to bump the version, what triggers the workflow, what the binary smoke-test confirms). The `UPSTREAM.md` files in `src/main/native/` migrate into the new natives module as the source-of-truth for what the published binaries were built from.

Non-goals: GraalVM native image support for graphitron-lsp; runtime auto-download of native libs from Maven Central at first use (the jar dep ships them on the classpath, which is the standard pattern); supporting tree-sitter grammars other than the LSP's current single GraphQL grammar; consuming the natives jar from any module other than graphitron-lsp.
