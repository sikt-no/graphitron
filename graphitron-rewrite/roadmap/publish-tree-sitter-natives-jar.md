# Publish graphitron-tree-sitter-natives jar; retire vendored C build in graphitron-lsp

`graphitron-lsp` today compiles the vendored tree-sitter runtime (`v0.26.0`) plus the bkegley tree-sitter-graphql grammar (`5e66e96`, both upstreams dormant) into a per-platform shared library on every `mvn install`. Three host-activated Maven profiles in `graphitron-lsp/pom.xml:86-197` invoke `src/main/native/build-native.sh` to produce `target/classes/lib/<os>-<arch>/libtree-sitter-graphql.{so,dylib}`; the build script rejects non-Linux non-Darwin hosts, so Windows is unimplemented (`BundledLibraryLookup.java:91-100` throws `UnsupportedOperationException`). The shape costs every developer and CI a C toolchain plus a non-trivial recompile on each build, ships only the platform the build is currently running on (CI builds `linux-x86_64` only; macOS rides on hope), and leaves R89 ("macOS / Windows CI verification") open as the multi-platform gap.

The chosen shape is a new published Maven artifact `no.sikt:graphitron-tree-sitter-natives` whose jar carries two prebuilt binaries for every supported platform, `libtree-sitter` (the runtime) and `libtree-sitter-graphql` (the grammar), under the `lib/<os>-<arch>/...` layout `BundledLibraryLookup` already resolves. The artifact has its own version stream tracking the bundled tree-sitter runtime (`<runtime-version>-<build-n>`, e.g. `0.26.0-1`); built by a GitHub Actions matrix on native runners that download the upstream `libtree-sitter` release binary for the runner's platform and run `tree-sitter build` against the vendored grammar to produce the grammar lib; published to Maven Central via `central-publishing-maven-plugin` configuration inlined into the module's standalone pom (matching the shape the rewrite parent's `release` profile uses at `graphitron-rewrite/pom.xml:262-337`, but not inherited from it). `graphitron-lsp` consumes the published jar as one `<dependency>`. The on-build C compile, the three `build-native-*` profiles, and the vendored `src/main/native/` tree all delete from graphitron-lsp; the grammar source moves to the natives module, the runtime sources are dropped entirely (replaced by upstream's prebuilt binaries), and the per-platform compile runs once per release rather than on every consumer's `mvn install`.

Design rationale was explored against three alternatives (committing binaries directly in graphitron-lsp; depending on bonede's prebuilt jars; standing up a separate repo). The bonede route was probed end-to-end with jtreesitter 0.26 and works on four of five platforms but the bonede Windows DLL exports only `Java_org_treesitter_*` JNI wrappers, zero bare `ts_*` runtime symbols, because MSVC defaults to no-export and the upstream tree-sitter C sources carry no `__declspec(dllexport)` annotations. Producing the binaries ourselves on a per-release matrix gives us all five platforms with a known-good symbol surface. Within that choice, the further question is *how* to produce them; the path picked is the standard tree-sitter ecosystem pattern of two libraries per platform: the host (jtreesitter, in our case) loads `libtree-sitter` once and resolves grammar libraries against its runtime symbols via separate `SymbolLookup`s composed at load time. Upstream tree-sitter publishes prebuilt `libtree-sitter` release binaries for all five target platforms (linux x64/arm64, macos x64/arm64, windows x64) on every tagged release; we consume those directly and use `tree-sitter build` (the upstream CLI) to compile the vendored grammar's `parser.c` into `libtree-sitter-graphql` on each matrix runner. The CLI handles MSVC + POSIX compiler differences uniformly, and upstream owns the runtime artifact end-to-end; no bespoke compile scripts, no Windows `.def` file extraction, no re-vendored runtime tree. (An earlier draft of this spec proposed a single unity-built library produced by `tree-sitter build`; empirical check showed the 0.26.x CLI compiles parser-only and does not link the runtime, so the unity-build claim was wrong. A follow-up draft proposed re-vendoring the runtime and hand-rolling per-platform compile scripts to restore the unity build; the realization that the standard ecosystem pattern is two libraries, and that upstream publishes the runtime binary we'd otherwise be re-compiling ourselves, collapsed both detours. The workflow's symbol-export assertion would have caught the parser-only build before publishing either way.) Committing binaries directly in graphitron-lsp couples the artifact's version to graphitron's release cadence, which doesn't match how often the C sources actually change. A separate repo adds cross-repo PR coordination for source updates that happen roughly never. Same-repo module with detached `<version>` is the middle.

## Design decisions

- **Module location:** `graphitron-rewrite/graphitron-tree-sitter-natives/`.
- **Maven coordinates:** `no.sikt:graphitron-tree-sitter-natives`.
- **Version scheme:** `<runtime-version>-<build-n>`. First release `0.26.0-1`. Bump `<build-n>` for any change to the bundled binaries (grammar update, build flag change, recompiled artifact for any reason). Bump `<runtime-version>` when moving the bundled tree-sitter runtime to a new upstream version. Snapshots not used; this artifact ships releases only.
- **Standalone pom, not in parent reactor, no parent inheritance.** The module's pom is fully self-contained: explicit `<groupId>no.sikt</groupId>`, explicit `<version>0.26.0-1</version>`, no `<parent>` reference. The release plumbing (`maven-gpg-plugin`, `central-publishing-maven-plugin`, `maven-source-plugin`, `maven-javadoc-plugin`, `maven-deploy-plugin`) is inlined into the module's pom rather than picked up from `graphitron-rewrite-parent`'s `release` profile. Rationale: the rewrite parent's version on trunk is `10-SNAPSHOT`, which is not deployed to Maven Central (the parent declares no `<snapshotRepository>`); inheriting via `<parent>graphitron-rewrite-parent:10-SNAPSHOT</parent>` would publish a natives jar whose parent reference is unresolvable for any consumer, locking the natives release cadence behind the next 10.x rewrite release. Standalone shape costs ~60 lines of inlined plugin config (a near-empty sources jar + javadoc jar are still produced; Maven Central requires both for releases even when there are no Java sources) and removes the coupling. The parent pom's `<modules>` list also does *not* include the natives module, keeping `mvn install -f graphitron-rewrite/pom.xml -Plocal-db` zero-cost. `graphitron-lsp` depends on the published artifact via Maven Central (or local m2 cache), not on a reactor-built sibling.
- **Build tool:** the upstream `tree-sitter` CLI (`tree-sitter build`, pinned to the same release tag as the runtime; `v0.26.0` for the first release) compiles the vendored grammar's `parser.c` on each matrix runner, producing `libtree-sitter-graphql.{so,dylib,dll}` per platform. The CLI handles MSVC + POSIX compiler invocation uniformly, including Windows `__declspec(dllexport)` for the grammar's single exported function. The runtime (`libtree-sitter.{so,dylib,dll}`) is *not* built locally; the workflow downloads it from upstream's GitHub release for the matrix runner's platform (`libtree-sitter-{platform}.{tar.gz,zip}` from `https://github.com/tree-sitter/tree-sitter/releases/download/<runtime-version>/...`) and verifies the downloaded archive against a SHA256 pinned in the workflow YAML. No bespoke compile scripts, no Windows `.def` file extraction, no `cc`/`cl.exe` invocation we maintain. Bumping `<runtime-version>` in the natives module's pom is a coordinated bump of the upstream tag *and* the per-platform SHA256s.
- **Matrix entries:** five: `linux-x86_64`, `linux-aarch64`, `macos-x86_64`, `macos-aarch64`, `windows-x86_64`. Exact GitHub-hosted runner labels picked at implementation time (the available labels shift); use whatever native arm64 runners GitHub offers for public repos at release time. The matrix is constrained to the intersection of (a) GitHub-hosted runners we can rent and (b) upstream tree-sitter's published `libtree-sitter` release binary set. Upstream's 0.26.x releases cover all five; if a future runtime version ever drops a platform, the matrix shrinks to match. `linux-aarch64` is a gain over current state (we ship only `linux-x86_64` today).
- **Jar layout:** binaries at `lib/<os>-<arch>/libtree-sitter.{so,dylib,dll}` (the runtime) and `lib/<os>-<arch>/libtree-sitter-graphql.{so,dylib,dll}` (the grammar). Windows files drop the `lib` prefix per platform convention: `tree-sitter.dll`, `tree-sitter-graphql.dll`. Matches `BundledLibraryLookup.resolveResourcePath`'s existing directory format with two new platform branches added (`linux-aarch64`, `windows-x86_64`) and a second filename to resolve per platform (the runtime, alongside the grammar). Jar also includes `META-INF/UPSTREAM.md` documenting the bundled tree-sitter runtime version + per-platform SHA256s + grammar commit.
- **Grammar source:** vendored at `graphitron-tree-sitter-natives/src/main/native/grammars/graphql/` (`parser.c` + `tree_sitter/parser.h` + `LICENSE` + `UPSTREAM.md`, moved verbatim from graphitron-lsp). Pinned to bkegley commit `5e66e96`.
- **Tree-sitter runtime source:** not vendored. Sourced at workflow-run time from upstream's GitHub release artifacts (`libtree-sitter-{platform}.{tar.gz,zip}` from `https://github.com/tree-sitter/tree-sitter/releases/tag/v<runtime-version>`), one archive per matrix runner. SHA256s for each platform's archive are pinned in the workflow YAML and verified on download; mismatch fails the runner. The runtime sources currently vendored in `graphitron-lsp/src/main/native/tree-sitter/` are deleted in Phase 3 and never moved into the natives module; we depend on upstream's published binary, not on their C sources.
- **Workflow trigger:** `workflow_dispatch` only (manual). No automatic builds on push. Cutting a release is a deliberate human-driven action — bump the version in the pom, push, trigger the workflow. Tagging is optional; the artifact is identified by Maven version, not git tag.
- **Bootstrap path:** the first release is published before any change to `graphitron-lsp` lands. Phase 1 lands the natives module + the workflow file without touching graphitron-lsp; phase 2 triggers the workflow and verifies the published `0.26.0-1` artifact resolves from Central; phase 3 swaps graphitron-lsp's pom to depend on the published version and deletes the vendored sources. Three phases, three observable seams.

## Implementation

A first Phase 1 attempt landed the natives module skeleton + workflow skeleton on `claude/graphitron-rewrite` and treated `tree-sitter build`'s parser-only output as the whole library; this spec was reverted to draft once that came to light. Nothing has been published to Maven Central: the workflow's symbol-export assertion catches the missing `ts_*` symbols and fails-closed at the build-matrix stage. The corrected direction keeps the CLI for the grammar compile (which is what it actually does well) and adds an upstream-runtime-download step; the standalone pom, the placeholder Java package, and the release plumbing stay as-is. The amendment list is in Phase 1 below; the existing files get *amended*, not rewritten.

### Phase 1 — natives module + release workflow

Land all of the following before triggering any release:

- Keep the vendored grammar at `graphitron-tree-sitter-natives/src/main/native/grammars/graphql/` (already in place from the first Phase 1 attempt). Do *not* move the tree-sitter runtime tree from graphitron-lsp into the natives module; the runtime comes from upstream's release binaries, not from source.
- Add a new matrix step in `.github/workflows/tree-sitter-natives-release.yml`, before the existing Build step: download `libtree-sitter-{platform}.{tar.gz,zip}` from `https://github.com/tree-sitter/tree-sitter/releases/download/v<runtime-version>/...` for the runner's platform, verify SHA256 against a pinned value in the workflow YAML (one entry per matrix runner), extract the shared library to a known path, and copy it into the staging directory the Package step assembles into the jar.
- Keep the existing `tree-sitter build` invocation for the grammar compile; that step is correct as-is for the grammar lib. The `tree-sitter-cli-version` workflow input stays (pinned to the same upstream tag as the runtime by convention; documented in the workflow).
- Split the matrix Symbol-Export Assertion (`.github/workflows/tree-sitter-natives-release.yml:119-127`) into two checks: (a) grammar lib (`libtree-sitter-graphql.{so,dylib,dll}`) exports `tree_sitter_graphql`; (b) runtime lib (`libtree-sitter.{so,dylib,dll}`) exports `ts_parser_new` (and a small set of representative `ts_*` symbols). Both must pass; either failure fails the runner.
- Update the Package step to assemble two libraries per platform into `target/classes/lib/<os>-<arch>/` rather than one. The jar-layout assertion bumps from expecting five jar entries (one per platform) to ten (two per platform); per-runner artifact upload includes both files.
- Update `BundledLibraryLookup.java`'s class docstring to reflect the actual shape: the natives jar carries two libraries per platform (runtime + grammar), composed at load time via separate `SymbolLookup`s. (Code change to `BundledLibraryLookup.get(arena)` lives in Phase 3; it's coupled to the lsp-side cutover, not the natives module itself.)
- Update `graphitron-rewrite/docs/tree-sitter-natives-release.adoc` with the corrected build-tool story: CLI for grammar, upstream release binary for runtime, two libs per platform, SHA256-pinned runtime download.

The standalone pom, the `no.sikt.graphitron.natives` placeholder Java package, and the existing release plumbing (gpg/source/javadoc/central-publishing) stay as-is; they were correct in the first attempt and are independent of the build-step direction change. The parent's `<modules>` list still omits the natives module; `mvn -f graphitron-rewrite/pom.xml ...` does not touch it.

### Phase 2 — first release

- Trigger the workflow manually. Verify the build matrix succeeds on all five platforms; the per-runner symbol-export assertion (stage 1, split form) fails loudly if the grammar lib lacks `tree_sitter_graphql` or the runtime lib lacks `ts_parser_new`.
- Verify the SHA256 check on the upstream-runtime download passes on all five runners. A mismatch (upstream re-published a tag, or our pin is wrong) fails the runner before any compile happens.
- Verify the package+deploy stage succeeds. Verify `no.sikt:graphitron-tree-sitter-natives:0.26.0-1` appears on Maven Central within the central-publishing-maven-plugin's typical sync window.
- Verify the post-deploy load+parse matrix (stage 3) succeeds on all five platforms: this is the authoritative "the jar works end-to-end on every supported platform" gate, and it specifically exercises the two-lib composition (runtime lookup + grammar lookup, composed). If any platform's load+parse fails after a successful deploy, increment `<build-n>` to `0.26.0-2`, fix, and re-release; `0.26.0-1` stays on Central as a tombstone but `graphitron-lsp` pins to the working build.

### Phase 3 — graphitron-lsp cutover

- `graphitron-lsp/pom.xml`: add `<dependency>no.sikt:graphitron-tree-sitter-natives:0.26.0-1</dependency>`. Delete the three `build-native-*` profiles (lines `86-197`) and the `org.codehaus.mojo:exec-maven-plugin` plumbing.
- Delete `graphitron-lsp/src/main/native/` entirely (grammar already lives in the natives module after Phase 1; the runtime sources are not needed anywhere in graphitron's tree once the natives jar is the source of truth).
- `BundledLibraryLookup.resolveResourcePath` (`graphitron-lsp/src/main/java/.../parsing/BundledLibraryLookup.java:77-101`): add two new platform branches (`linux-aarch64`, `windows-x86_64` matched on `os.name` containing `windows`, `os.arch` ∈ {`amd64`, `x86_64`}) and broaden each branch to return a *pair* of resource paths (runtime + grammar) rather than a single path. The map becomes `lib/<os>-<arch>/{libtree-sitter,libtree-sitter-graphql}.{so,dylib,dll}` with the Windows variants dropping the `lib` prefix.
- `BundledLibraryLookup.get(arena)`: change the shape to extract *both* library resources to temp files, build two `SymbolLookup.libraryLookup(...)` against them, and return a composed lookup. The standard composition is `(name) -> runtime.find(name).or(() -> grammar.find(name))`; this matches jtreesitter's expectation that grammar symbols (`tree_sitter_graphql`) and runtime symbols (`ts_parser_new`, `ts_parser_parse`, …) both resolve through the lookup it's handed. Library extraction order matters on Windows (extract + load the runtime first, then the grammar) because grammar code on Windows is linked against the runtime DLL by name; on POSIX the order is irrelevant. Encode the order in the extraction loop, not by relying on map iteration.
- Update test docstrings: `IncrementalParseTest.java:15` ("proving it works through the bonede binding closes one of the biggest …") and `TreeSitterSmokeTest.java:11` ("Verifies that the bonede tree-sitter Java binding loads its native libraries") drop the bonede references; both now exercise the natives-jar path.
- `NativeBuildSmokeTest.java` (`graphitron-lsp/src/test/java/.../native_build/`): rename to something like `NativeLibraryBundleTest.java` (the test no longer asserts "the build produced a library"; it asserts "the bundled binaries load + compose + parse"), and add a fifth `@EnabledOnOs(value = OS.WINDOWS, architectures = "amd64")` method. The existing three methods stay; the fourth (`linux-aarch64`) and fifth (`windows-x86_64`) get added. Each method exercises the two-lib composition path through `BundledLibraryLookup`.

### Documentation

- `graphitron-rewrite/docs/` gets one new short doc (`tree-sitter-natives-release.adoc` or fold into the existing LSP-internals doc): how to cut a new natives release, when to bump `<runtime-version>` vs `<build-n>`, what the smoke-test confirms, where the published artifact lives. Internal to the rewrite; not a user-facing page.
- The `UPSTREAM.md` files moved from `graphitron-lsp/src/main/native/` are the source-of-truth pin and update with each runtime/grammar version change.

## Tests

- Natives release workflow asserts jar layout in the package stage: exactly ten `lib/<os>-<arch>/` entries (two libraries per platform, runtime + grammar), expected file names per platform. Each build-matrix runner asserts symbol-export on both binaries separately (grammar exports `tree_sitter_graphql`; runtime exports `ts_parser_new`). Each post-deploy-matrix runner asserts the published jar load+parses on its platform, which specifically covers the two-lib composition.
- SHA256 verification of the upstream-runtime download runs before any compile on every matrix runner. A mismatch fails the runner; this gates against an upstream tag silently re-uploaded under us.
- `NativeLibraryBundleTest.java` in graphitron-lsp covers each of the five platforms with `@EnabledOnOs`, loads both bundled libraries through `BundledLibraryLookup`, parses a trivial GraphQL document, asserts root node + no errors. In regular CI (`rewrite-build.yml`, currently `ubuntu-latest`) only the linux-x86_64 method executes per PR; the other four `@EnabledOnOs` methods run on their respective platforms in the natives release workflow's post-deploy matrix. `rewrite-build.yml` does not gain a multi-OS matrix as part of this item: the binaries don't change per PR (they're a published artifact), so per-PR multi-OS coverage would mostly re-verify a stable input.
- Windows verification has no pre-existing test fixture; the new fifth `NativeLibraryBundleTest` method establishes it, exercised at natives-release time.
- No pipeline-tier or execution-tier coverage changes: the natives jar is a build-time substitution, everything downstream sees identical behaviour.

## Acceptance

- `no.sikt:graphitron-tree-sitter-natives:0.26.0-1` exists on Maven Central with two binaries (runtime + grammar) for all five platforms; the artifact resolves into a fresh local m2 without errors (no unresolvable-parent failures: the natives pom has no `<parent>` reference).
- `mvn install -f graphitron-rewrite/pom.xml -Plocal-db` from a clean checkout no longer invokes `cc`, `cl.exe`, or any C toolchain. The build completes without `src/main/native/` existing in `graphitron-lsp/`.
- The natives release workflow's post-deploy load+parse matrix passes on all five platforms for the `0.26.0-1` release, specifically exercising the two-lib composition through `BundledLibraryLookup`. Regular CI (`rewrite-build.yml`) on `ubuntu-latest` runs the `linux-x86_64` `@EnabledOnOs` method on every PR; the other four are gated to their respective hosts and run on the release workflow's matrix runners.
- R89 is deleted from `graphitron-rewrite/roadmap/`.

## Non-goals

- GraalVM native image support for graphitron-lsp.
- Runtime auto-download of native libs from Maven Central at first use (the jar dep ships them on the classpath; the standard pattern).
- Supporting tree-sitter grammars other than the LSP's current single GraphQL grammar.
- Consuming the natives jar from any module other than graphitron-lsp.
- Adding `linux-arm` (32-bit) or `powerpc64` despite the upstream CLI shipping for them: no consumer.
- Re-vendoring the tree-sitter runtime sources in the natives module or compiling `libtree-sitter` ourselves. Upstream publishes the runtime as a release artifact for every target platform; we consume those directly and stay out of the runtime-build business.
