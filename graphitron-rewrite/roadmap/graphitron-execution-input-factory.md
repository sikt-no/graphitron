# Graphitron.newExecutionInput factory for single-tenant ergonomics

Every app that runs a graphitron-generated schema repeats the same two lines of boilerplate on every request: `.graphQLContext(b -> b.put(GraphitronContext.class, ctx))` to thread the per-request context where generated fetchers look it up, and `.dataLoaderRegistry(new DataLoaderRegistry())` to satisfy graphql-java's requirement that one always be present (the sakila example's `GraphqlResource.execute` at `graphitron-rewrite/graphitron-sakila-example/src/main/java/no/sikt/graphitron/sakila/example/app/GraphqlResource.java:62-67` is the canonical demonstration, with the javadoc apologising for the registry requirement). The context wiring is a silent foot-gun: forget it and generated fetchers look up `GraphitronContext.class` in the request's `GraphQLContext`, get `null`, and crash deep in a resolver. The registry boilerplate is pure ceremony because generated fetchers populate the registry lazily via `env.getDataLoaderRegistry().computeIfAbsent(name, k -> DataLoaderFactory.newDataLoader(...))` (e.g. `QueryNodeFetcherClassGenerator.java:163`, `DataLoaderFetcherEmitter.java:116`), so an empty fresh registry is exactly what every app needs. The legacy codebase recognised half this gap in `graphitron-common`'s `DefaultGraphitronContext` (`graphitron-common/src/main/java/no/sikt/graphql/DefaultGraphitronContext.java`), which lets single-tenant users construct a context from a single `DSLContext`; the rewrite has no analog.

Add two static factory overloads to the already-generated `Graphitron` facade (`GraphitronFacadeGenerator`, `<outputPackage>.Graphitron`): `newExecutionInput(GraphitronContext ctx)` and `newExecutionInput(DSLContext dsl)`. Both return an `ExecutionInput.Builder` pre-wired with the supplied context under `GraphitronContext.class` in `graphQLContext` and a fresh empty `DataLoaderRegistry`; the `DSLContext` overload threads the value via the lambda `env -> dsl` against the generated functional-interface form of `GraphitronContext`. Make this the canonical entry point by also pushing `getContextArgument` from abstract to a `default` method on the generated interface (default body: `return env.getGraphQlContext().get(name);`, matching the legacy default), which leaves `getDslContext` as the only abstract method and makes the interface a de-facto functional interface usable as a lambda anywhere. Do not annotate the interface `@FunctionalInterface` (the annotation is a permanent contract that would block ever adding another abstract method); the lambda usability is independent of the annotation. Update the sakila example's `GraphqlResource.execute` to use the factory, since it is the canonical "how do I use this" reference and the boilerplate it currently models is exactly what's being removed. Custom `DataLoaderRegistry` users (shared across requests, pre-populated, instrumented subclass) remain supported because graphql-java's `ExecutionInput.Builder.dataLoaderRegistry(...)` *replaces* rather than merges — a chained `.dataLoaderRegistry(custom)` after the factory wins.

Considered and rejected during design discussion: shipping a generated `DefaultGraphitronContext` class alongside the interface (the lambda `env -> dsl` is shorter than `new DefaultGraphitronContext(dsl)` and removes one name from the user's import list); shipping an `ExecutionInput`-wrapping builder (the friction is the `graphQLContext` and `dataLoaderRegistry` calls specifically, not the input plumbing; wrapping the full builder surrenders flexibility for one map entry); a `Graphitron.newExecutionInput(DSLContext, DataLoaderRegistry)` three-arg overload (`.dataLoaderRegistry(custom)` already covers the custom-registry case without expanding the API surface).
