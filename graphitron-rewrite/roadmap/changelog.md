New `ValuesJoinRowBuilder` helper at `graphitron/src/main/java/no/sikt/graphitron/rewrite/generators/util/ValuesJoinRowBuilder.java` owns the typed `Row<N+1>` / `Record<N+1>` type-args (with the arity-22 cap and a per-call-site directive context — `"@lookupKey"` / `"@key"` — in the cap-violation message), the `@SuppressWarnings({"unchecked", "rawtypes"})` typed-row array declaration, per-cell `DSL.val(value, table.COL.getDataType())` construction, the `DSL.values(rows).as(alias, "idx", "<sqlName1>", …)` alias-args list, and the `USING(table.C1, …)` column-list helper consumed by the lookup root path; `Row<N+1>[]` and `Table<Record<N+1>>` type helpers fall out of the same parameterisation. Helper API is generic over the caller's slot type (`<S>`) plus a `Function<S, ColumnRef>` projection: lookup callers keep their rich `Slot` record (argName + `RootSource` + decode bindings) and pass `Slot::targetColumn`; the dispatcher passes `Function.identity()` against `List<ColumnRef>` directly. The reviewer pass dropped an earlier helper-internal `Slot(ColumnRef)` record that had forced callers to maintain a parallel-list bridge between the helper's bare slot and their richer slot type; the value-expression callback now receives the caller's slot back through the callback. Caller-local: the for-loop body that fills `rows[i]` (lookup site has the composite-key extraction and per-row `DecodedRecord` NodeId decode; dispatcher reads `binding[0]` / `binding[1]`); the `idx` cell expression (`DSL.inline(i)` for compile-time int vs `DSL.val(idx, Integer.class)` for binding-derived — same SQL, different javapoet); any extra projections beyond the join (e.g. dispatcher's `DSL.inline("<TypeName>").as("__typename")`); the `.where(condition).orderBy(idxCol)` chain. **Dispatcher kept on `.on(…)`, not switched to `.using(…)`** (deviation from the original spec, recorded in the spec's "Reviewer pass deltas"): the dispatcher's SELECT projection comes from `<TypeName>.$fields(env.getSelectionSet(), t, env)` which references `t.<col>` directly, and `USING` collapses joined columns at render time, which can interact poorly with `$fields`-emitted projections that include the joined key columns themselves. The lookup root path stays on `.using(…)` (its FROM side is the entity's own table only — no FK chain — and its projection does not reuse joined columns). The helper still exposes `usingArgs` for the lookup root path; the dispatcher's class Javadoc on `SelectMethodBody` calls out the asymmetry. `Condition condition = DSL.noCondition()` is declared before the dispatcher's join body so the chain stays symmetric with the lookup site and gives R36 Track B (per-typename interface filters) a hook to AND into; jOOQ folds `noCondition()` away at render time. **Coverage**: 14 new `ValuesJoinRowBuilderTest` cases (UnitTier) pin arity behaviour at 1 / 5 / 21, the arity-22 cap with directive-context error message, the empty-slots guard (defensive against the silent `Row1<Integer>` shape — both upstream classifiers already enforce non-empty key columns), alias args, USING args, both idx-cell shapes, the `Row<N+1>[]` / `Table<Record<N+1>>` type helpers, and the rich-slot callback contract via a local `RichSlot` record. The f-E SQL-shape regression test (`GraphQLQueryTest.nodes_perTypeIdBatch_emitsValuesJoinOrderByIdxShape`) continues to pass; its substring pin (`values (` + `join ` + `order by`) is loose enough to accept either ON or USING shape across both call sites. `FederationEntitiesDispatchTest`'s 16 dispatch cases all pass against the post-collapse emission. **Net code reduction**: lookup emitter 505 → 458 lines; dispatcher's row-shape emission consolidated; new helper at 185 lines + 196 lines of unit tests; net diff is -68 lines of generator code with one new file carrying the shared core. **Unblocks**: R36 Track B's stage-2 emission becomes the third consumer of `ValuesJoinRowBuilder` rather than forking a third copy; R36's `depends-on` cleared.
