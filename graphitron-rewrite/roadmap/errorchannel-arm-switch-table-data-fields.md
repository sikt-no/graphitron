# Error-channel arm-switch for @table-bound DataLoader data fields under Outcome types

R244's `@service` flip (commit 3b) hands graphql-java a typed `Outcome<X>` source for every root `@service` outcome field and adds the `validateOutcomeChildArmSwitch` pass that rejects any immediate child of a flipped outcome type whose leaf is off the `OUTCOME_TYPE_ARM_SWITCHED_DATA_CHANNEL_VARIANTS` allow-list. That allow-list deliberately omits `ChildField.RecordTableField`: R244's arm-switch emitter (`FetcherEmitter.armSwitchValueExpr`) only handles inline-readable success projections (Java-backed accessor reads, constructor/nesting passthrough), and a `@table`-bound child resolved through a batch DataLoader is not inline-readable the same way. R244's commit-3b inventory ("no in-scope payload has a nested data child") held only over graphitron's own sakila fixtures; real consumers (e.g. `opptak-subgraph`) put a `@table`-bound data field next to the errors field in nearly every `@service` mutation payload (`...Payload.data` / `.sak` / `.kvotetype` / `.opptak` / ...), so the flip turned those schemas from *compiling* into *rejected at schema validation* with the "RecordTableField is not on the Outcome arm-switch allow-list" author error. This is the deferred follow-up R244 named explicitly ("Nested ... children under an outcome type ... are narrowed out ... so the arm-switch validator rejects them with a clear message until a follow-up supports them"). It is also the build-time face of a latent runtime bug: pre-flip, the same shape silently NPE'd on the error arm (`Cannot invoke SakRecord.into(...) because "__elt" is null`), because the generated split-query key extraction reads the nested jOOQ record off the parent and calls `.into(<PK>)` on it with no null guard. This slice teaches the `@table`-bound DataLoader data-channel field to arm-switch, which both lifts the rejection and removes the NPE.

## Vocabulary

Reuses R244's terms exactly (error type / errors field / outcome type / outcome field / the `Outcome<T>` witness with its `Success` and `ErrorList` arms / data-channel field / arm-switch). One term added for this slice:

- **Loaded data field** ; a data-channel field of an outcome type whose value is resolved through a batch DataLoader keyed by parent correlation rather than read inline off the source object. In the model these are the `BatchKeyField` permits that can sibling an `ErrorsField` under a root-`@service` payload: `ChildField.RecordTableField` (the observed `opptak-subgraph` case) and `ChildField.RecordLookupTableField` (its `@lookupKey`-argument sibling). Both are in scope: `RecordLookupTableField` can classify as an immediate child of a root-`@service` `OutcomeType` (a table-bound payload field whose argument, or a nested input field, carries `@lookupKey`; constructed at `FieldBuilder.classifyChildFieldOnResultType` via `hasLookupKeyAnywhere`), and both implement `BatchKeyField` with a `SourceKey` + `LoaderRegistration`, so the arm-switch contract is substrate-identical for the two. (The variant set is fixed here, not deferred to In Progress, because the allow-list edit and its pinning fixtures must be specifiable at Spec time.) The DML `SingleRecord*` carriers are *not* loaded data fields in this sense; they ride the `LocalContext`/sentinel transport and are out of scope (see below).

## Why this is not slice 1's inline read

R244's arm-switch for an inline shape is a one-liner: `env.getSource() instanceof Success<?> success ? <read off success.value()> : null` (`FetcherEmitter.armSwitchedDataFetcher`, line 82-86). A loaded data field does not reach that path. `RecordTableField` falls through `dataFetcherValueRaw` to the catch-all method-reference (`FetcherEmitter.java:289`, `$T::$L`); its real fetcher is a generated method (`TypeFetcherGenerator.buildSplitQueryDataFetcher`) that reads the parent key off `env.getSource()`, dispatches a DataLoader, and returns a `CompletableFuture`. The key extraction is `GeneratorUtils.buildAccessorKeySingle` / `buildAccessorKeyMany` (lines 249-296):

```java
ElementRecord __elt = ((Backing) env.getSource()).accessor();   // null on the error arm
RecordN<...> key = __elt.into(Tables.T.PK1, Tables.T.PK2);       // NPE: __elt is null
```

So the arm-switch for a loaded data field must live *inside* the generated DataLoader method, not in the inline-ternary emitter. On the `ErrorList` arm the method must return `CompletableFuture.completedFuture(null)` *before* touching the key (never dispatch the loader); on the `Success` arm it must extract the key off `success.value()` (not `env.getSource()`) and dispatch as today. This is the same null-on-the-error-arm contract R244's `Outcome` javadoc states, expressed for a CompletableFuture-shaped fetcher rather than a scalar one.

**The error-arm short-circuit is the whole fix for the reported failures.** Because the `ErrorList` arm returns before key extraction, the generated method never reaches `buildAccessorKeySingle`/`buildAccessorKeyMany` on the error arm, so the `__elt`-null NPE that `opptak-subgraph` hit (`sakFinnesIkke`, `oppdaterIkkeFunnet`) cannot recur. The model already proves the short-circuit shape: the sibling table-parent path `GeneratorUtils.buildKeyExtractionWithNullCheck` (lines 324-356) returns `CompletableFuture.completedFuture(null)` when a key component is null ("a NULL FK can never match ... dispatching to the DataLoader is a wasted round-trip").

A second, *independent* NPE lives at the same code site: the accessor key-extraction reads the nested record off `success.value()` and calls `.into(...)` on it unguarded, so a legitimately-null nullable to-one relation NPEs on the **success** arm, with no `@error` payload involved. That is not an error-channel concern and is **out of scope here**; it is filed as **R269** (null-guard the accessor key-extraction helpers, pinned by a non-error-parent fixture). R268 changes the *source* the helpers read from (`success.value()` vs `env.getSource()`); R269 changes whether they tolerate a null read. Keeping them separate keeps each change paired with its own pin (R268's allow-list entry + error-arm round-trip; R269's success-arm null-resolves-not-throws fixture).

## Implementation seams

1. **`FetcherRegistrationsEmitter` → split-query method threading.** R244 threads `sourceIsOutcome` (via `hasWrapperArmErrors`) into `FetcherEmitter.dataFetcherValue` only. A loaded data field's fetcher is generated by `TypeFetcherGenerator`, so the same per-type "parent is a flipped outcome type" signal must reach `buildSplitQueryDataFetcher` (and any sibling DataLoader-method builder a `RecordTableField` routes through). Thread the flag, do not re-derive it by sibling-scanning at emit time (R244's settled stance: arm-switch is a generation-time decision).

2. **`TypeFetcherGenerator.buildSplitQueryDataFetcher`.** When the parent is a flipped outcome type: narrow the source to `Outcome.Success<X>` first; on the `ErrorList` arm `return CompletableFuture.completedFuture(null)`; on the `Success` arm bind the backing from `success.value()` and run the existing key-extraction + DataLoader dispatch unchanged. Generated-code style follows R244 commit 3b (readable locals: `source`, `success`; no dunder-prefixed names in new code; `if`/`return` over nested ternaries for the CompletableFuture arms).

3. **`GeneratorUtils.buildAccessorKeySingle` / `buildAccessorKeyMany`.** The backing must come from `success.value()` under the wrapper instead of `(($T) env.getSource())`. Carry this as *one* threaded value, not a per-helper `if (sourceIsOutcome)`: compute the source binding once at the `buildSplitQueryDataFetcher` seam (the `Outcome.Success<X>` narrowing prelude plus the `success.value()` expression, or `env.getSource()` + empty prelude for the non-wrapper case) and pass it into both helpers (and `buildKeyExtractionWithNullCheck`) as a small `SourceBinding` value or `CodeBlock` the helper interpolates via `$L`. The narrowing prelude is emitted once, not re-decided in three places; this also makes the helpers the natural unit-addressable seam (seam below / Tests). The null short-circuit on a null nested record is **R269**, not this slice. While editing these helpers, rename the pre-existing `__elt` / `__k` dunder locals to readable names (`elt`, `key`) per R244's generated-code style and its own loop-var retrofit; leaving them dunder while touching the lines would read against the style note in the same diff.

4. **`GraphitronSchemaValidator` allow-list.** Add `RecordTableField` and `RecordLookupTableField` to `OUTCOME_TYPE_ARM_SWITCHED_DATA_CHANNEL_VARIANTS` *in the same change* as seams 1-3. The validator javadoc makes this lockstep mandatory: "Adding a variant to this set requires the matching emitter site to arm-switch." Adding the variant without the emitter support would replace a clear author error with a generated NPE; shipping the emitter without the allow-list entry leaves the schema rejected.

## Latent allow-list / emitter mismatch (call out, do not silently widen)

The R244 allow-list already lists four variants the emitter does *not* implement: `ServiceTableField`, `ServiceRecordField`, `TableMethodField`, `RecordTableMethodField` are in `OUTCOME_TYPE_ARM_SWITCHED_DATA_CHANNEL_VARIANTS` but `armSwitchValueExpr` throws `IllegalStateException` on them. A schema with one of those as a loaded data sibling would pass the validator and then crash code generation. That mismatch is pre-existing and out of scope here; it is filed as **R270**. This slice must not widen the gap: every variant it adds to the allow-list (`RecordTableField`, `RecordLookupTableField`) must have a working emitter site landed in the same change, pinned by the execution-tier fixture below.

## Design fork for In Progress (flag to principles review)

R244's commit 3a already rejected a delegation-wrapper arm-switch (rebuild the env with `source(value())` and delegate to the inner fetcher) on the explicit-fetcher principle, on three concrete costs: re-instantiating the inner fetcher per request, asserting env-rebuild transparency on the least-tested fetchers, and hiding the read behind opaque delegation. The analogous fork here is: wrap the whole generated DataLoader method body in a `Success`-narrow shim, versus thread the source expression through key extraction so the method reads `success.value()` directly. The shim recurs two of those same costs structurally, it hides the arm decision behind a narrowing layer rather than emitting it as part of the explicit method, and it asserts transparency over the DataLoader dispatch (among the least-tested paths). So the recommendation follows from the explicit-fetcher principle itself, not merely from consistency with 3a: one generated method, source bound from `success.value()` on the `Success` arm, `completedFuture(null)` on the `ErrorList` arm, no shim and no env rebuild. The `principles-architect` confirmation before the flip is then a formality, not a re-derivation.

## Tests

- **Execution (`@ExecutionTier`) is the primary safety net, and R268 adds the fixture.** No sakila fixture today pairs a `@table`-bound data field with an errors field under a root `@service` payload (that gap is exactly why R244's inventory missed this). R268 adds one: a `@service` mutation whose payload type has a loaded data field (`RecordTableField`) sibling to an `[SomeError]` errors field, with the data field nullable per R244's success-projection rule. The test asserts both arms round-trip against the real generated fetchers: the mapped-error path returns `{data: null, errors: [...]}` and does **not** raise the `__elt`-null NPE; the success path returns `{data: {...}, errors: []}`. This reproduces the `opptak-subgraph` `sakFinnesIkke` / `oppdaterIkkeFunnet` failures in-tree. A `RecordLookupTableField` data field (a payload field with a `@lookupKey` argument) is covered by a classification assertion at minimum; an execution case if a lookup-under-payload fixture is cheap to express.
- **Pipeline (`@PipelineTier`)**: the loaded data field under a flipped `@service` payload classifies as `RecordTableField` / `RecordLookupTableField` and `validateOutcomeChildArmSwitch` no longer rejects it (structural assertion on the classified model and the absence of the rejection, not a fetcher body-string assertion).
- **Unit (`@UnitTier`)**: the threaded source-binding (the `success.value()` source vs. `env.getSource()`, and the once-emitted `Success`-narrow prelude) at the helper seam from seam 3. The null-short-circuit itself is R269's test, not this slice's.
- **Compilation (`@CompilationTier`)**: `mvn install -Plocal-db` end-to-end-green over `graphitron-sakila-example`, which compiles the generated `CompletableFuture`-returning arm-switch and catches type-narrowing mismatches across the erased `env.getSource()` seam.

## Out of scope

- **DML `SingleRecord*` data fields.** The four `DmlTableField` permits and the `SingleRecord*` carriers stay on the `LocalContext`/sentinel transport (`LOCAL_CONTEXT_GUARDED_DATA_CHANNEL_VARIANTS` / `validateLocalContextErrorsFieldGuards`), retired by R244's deferred DML-migration slice, not here. Their arm-switch onto `Outcome` belongs with that migration.
- **`@tableMethod` / child `@service` data fields.** R244 scoped `@tableMethod` out of the flip (returns classify `NoChannel`); child `@service` likewise. Their outcome-channel migration (and any arm-switch they need) is a separate slice.
- **The four allow-listed-but-unimplemented variants** (`ServiceTableField`, `ServiceRecordField`, `TableMethodField`, `RecordTableMethodField`): **R270**.
- **The general nullable-to-one `.into(...)` null-guard** on the success arm: **R269**.
- **Carrier collapse / `Optional<ErrorChannel>` → non-Optional**, the `MutationField` interface split: bundled with the DML migration per R244.
