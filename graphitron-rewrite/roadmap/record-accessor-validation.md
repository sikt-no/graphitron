# Validate `@record` accessor resolution against backing class at classify time

A downstream consumer (Sikt opptak-subgraph) hit a generator regression that surfaced only at downstream `javac`: a `@record`-mapped GraphQL type with `sakId: ID` against a Java POJO that exposes `getSak(): SakRecord` (no `getSakId()`) compiled the generator clean, then failed `javac` on the generated `BeregnPoengAutomatiskResultatType.java` with `cannot find symbol: method getSakId()`. The mismatch is invisible to graphitron's pipeline because `FetcherEmitter.propertyOrRecordValue` at `graphitron/src/main/java/no/sikt/graphitron/rewrite/generators/FetcherEmitter.java:204-209` synthesizes the getter name purely from the SDL column-name string (`"get" + capitalize(toCamelCase(columnName))`) without ever consulting the backing class's method set. The same purely-string-derived heuristic governs the `JavaRecordType` arm at `:199-202` (bare-name accessor for Java records), so the gap covers both `@record`-on-POJO and `@record`-on-Java-record output types.

The principle this violates is *Classifier guarantees shape emitter assumptions* (`docs/rewrite-design-principles.adoc:107-122`). The classifier (`TypeBuilder.buildResultType`, `graphitron/src/main/java/no/sikt/graphitron/rewrite/TypeBuilder.java:511-544`) currently produces a `PojoResultType(name, location, fqClassName)` (resp. `JavaRecordType(...)`) carrying only the class's binary name. Whether that class actually exposes a method matching the SDL fields is left for the emitter to discover via string concatenation, with no fallback when the heuristic is wrong. The reflection-boundary roster (`docs/rewrite-design-principles.adoc:29`) places `Class.forName + reflective lookup` at the classifier layer; the existing payload-errors path at `FieldBuilder.java:1542-1589` is the local precedent for that pattern, but it fires only when the SDL declares an errors-shaped field. Plain `@record` types never trigger it.

The fix is to lift accessor resolution into the classifier and have it fail closed: at classify time, for every SDL output field of a `@record`-backed type, resolve the call shape against the backing class's `Class<?>` reflectively, attach a sealed `AccessorResolution` (`Resolved(...)` carrying the resolved `Method` plus arm identity, or `Rejected(reason)` carrying an actionable diagnostic) to the field's classification, and propagate `Rejected` through `UnclassifiedField` so `GraphitronSchemaValidator` reports a `ValidationError` naming the SDL field, the backing class, and the candidates that were tried. The emitter consumes the pre-resolved `Method` handle and never recomputes the heuristic. This matches *validator mirrors classifier invariants* (`docs/rewrite-design-principles.adoc:103-105`).

Resolution rules follow graphql-java's `PropertyDataFetcher` lookup order, so consumers migrating from `PropertyDataFetcher` to graphitron's `@record`-emitted fetcher get matching name resolution:

- The base name comes from `@field(name:)` if present (the existing override sites are `FieldBuilder.java:2632-2634` for the scalar-on-record-parent arm and `:2642-2644` for the object-child arm), otherwise the SDL field name. The validator hooks the same pipeline.
- Candidate accessors, in resolution order: `get<CamelName>()` (POJO bean getter, any return type); `is<CamelName>()` (only when the SDL field's resolved Java type is `boolean`/`Boolean`); bare `<camelName>()` (Java record component / fluent accessor); public field access `<camelName>` (graphql-java's last-resort field-read path).
- If the SDL field has arguments, the accessor must declare matching parameters per graphql-java conventions: either a single `DataFetchingEnvironment` parameter for full-env injection, or per-arg parameters whose Java types match each SDL arg's resolved Java type, in declared order. A zero-arg method on a SDL field with arguments is a `Rejected`, not a silent argument-drop.
- The resolved method's return type must be assignable to the SDL field's resolved Java type (e.g. `String` for `ID`, `BigDecimal` for `BigDecimal`). A method that exists by name but returns the wrong type is `Rejected` with the type mismatch as the diagnostic, not a silent acceptance that defers the failure to `javac`. This is the arm that catches the user's `getSakId(): SakRecord` case if a developer rewrites the getter to match by name without changing the return shape.

The diagnostic message follows the project's standard validator surface: name the SDL field by parent + name (e.g. `BeregnPoengAutomatiskResultat.sakId`), the backing class FQN, the accessor candidates tried with reasons each was rejected (no method by that name; method exists but parameter shape doesn't match; method exists but return type incompatible), and the override hint (`@field(name: "sak")` to bind to `getSak()`). Same tone as `ServiceCatalog`'s `Rejection.structural` messages and the `ConditionResolver` field/arg arms at `ConditionResolver.java:96-137`.

A first-pass spec carried (a) a `NotApplicable` arm on `AccessorResolution` so the value type was total over every classified field, (b) a nested `AccessorShape` discriminator inside `Resolved` with a shared `method()` accessor, and (c) a `Map<String, AccessorResolution>` slot on `PojoResultType` / `JavaRecordType` keyed on SDL field name. Architect review (in-session, before hand-off) flagged all three as friction with the principles: `NotApplicable` is the empty-arm smell (`docs/rewrite-design-principles.adoc:25` — "each new sub-taxonomy comes with a one-line note on what distinct information it carries"), nested sealing with shared accessors duplicates the BatchKey "narrow component types over broad interfaces" tension (`:39-43`), and a side-table keyed by SDL name is a lighter version of the same string-derivation smell the spec is fixing. This draft adopts the revisions: `AccessorResolution = Resolved | Rejected` with `Resolved` directly sealed over `GetterPrefixed(Method)` / `BareName(Method)` / `FieldRead(Field)` (each arm carries exactly its data), no `NotApplicable`, and the `AccessorResolution` rides on the per-field arm alongside `ColumnRef` / `MethodRef` rather than on a side-table on the parent type. Fields where reflective resolution doesn't apply are classified into different field arms entirely; the structural arm identity is the "this resolution doesn't apply" signal.

## Implementation

### Phase A — Sealed `AccessorResolution` model + `ClassAccessorResolver` + per-field carrier

**Model (`graphitron/src/main/java/no/sikt/graphitron/rewrite/model/AccessorResolution.java`, new file).**

- Two-layer sealing where the inner layer carries per-arm data:
  - `sealed interface AccessorResolution permits Resolved, Rejected`.
  - `sealed interface Resolved extends AccessorResolution permits GetterPrefixed, BareName, FieldRead`. Each arm carries exactly the data it uses: `record GetterPrefixed(Method method)` for `get*`/`is*` POJO bean getters (the prefix is recoverable from `method.getName()`); `record BareName(Method method)` for Java record components and zero-arg fluent accessors; `record FieldRead(Field field)` for public-field access. No shared `method()` accessor with a null arm; no nested `AccessorShape` discriminator. Per *narrow component types over broad interfaces* (`docs/rewrite-design-principles.adoc:39-43`) and the BatchKey two-axis precedent (`graphitron/src/main/java/no/sikt/graphitron/rewrite/model/BatchKey.java:13`), the variant identity is the typing.
  - `record Rejected(String reason) implements AccessorResolution`.
- A future arm (e.g. a `LombokAccessor` extension point with custom name patterns) extends `Resolved`'s permit list and becomes a compile error at the emitter's exhaustive switch, not a silent fall-through.
- No `NotApplicable` arm. Fields where reflective resolution doesn't apply (jOOQ-record-backed via `FetcherEmitter.propertyOrRecordValue:190-198`, `@service`-bound via `serviceCallTarget`, enum-mapped, or `null fqClassName` `PropertyDataFetcher` fallback) are *classified into different field arms entirely* — they never produce a classified field that would need an `AccessorResolution` slot. The "this resolution doesn't apply" signal is structural (which arm classifies the field), not an arm of `AccessorResolution`.

**Resolver (`graphitron/src/main/java/no/sikt/graphitron/rewrite/ClassAccessorResolver.java`, new file).**

- Public entry: `static AccessorResolution resolve(Class<?> backingClass, String accessorBaseName, java.lang.reflect.Type expectedReturnJavaType, ParamShape expectedArgs, CandidateOrder order)` where `CandidateOrder` is a small enum (`POJO_FIRST` for `PojoResultType`-classified parents, `RECORD_FIRST` for `JavaRecordType`-classified parents). The classifier picks the order at the call site by parent variant; the resolver does not introspect `backingClass.isRecord()` itself, so the type variant remains the single source of truth for the candidate-order fork. The helper joins the existing reflection roster (`ServiceCatalog`, `BatchKeyLifterDirectiveResolver`, `FieldBuilder`'s payload-errors path, `FieldBuilder`'s typed-accessor reflection in `deriveBatchKeyFromTypedAccessor` at `:2872-2994`); the principles doc roster at `docs/rewrite-design-principles.adoc:29` updates in this phase's commit to add `ClassAccessorResolver`.
- Algorithm:
  1. Build candidate-name set, ordered: `["get" + capitalize(camelName)]`, plus `["is" + capitalize(camelName)]` if `expectedReturnJavaType` is `boolean`/`Boolean`, plus `[camelName]`. Capitalisation rule matches `FetcherEmitter.java:208`'s existing `Character.toUpperCase(...) + substring(1)`.
  2. For each candidate name, scan `backingClass.getMethods()` for a non-static, public method whose parameter shape matches `expectedArgs` and whose return type is assignable to `expectedReturnJavaType`. First match wins; arm identity comes from the candidate-name index (0/1 → `GetterPrefixed`, 2 → `BareName`).
  3. If no method matches, scan public fields for `camelName`; if found and type-compatible, return `Resolved.FieldRead(field)`. The public-field path applies only when `expectedArgs` is `PerArgument(List.of())` — a public field cannot accept arguments, so an SDL field with arguments that finds no method match falls through to step 4 even if a same-named public field exists.
  4. If still no match, build the diagnostic: list each candidate name tried, and for the closest by-name match (if any) state *why* it was rejected (return type, param shape) so the developer sees "found `getSak(): SakRecord`, but the field requires `String` (SDL type `ID`)" rather than a generic "not found". Return `Rejected(diagnostic)`.
- The helper does the `Class.forName` itself; the classifier passes the FQN. `ClassNotFoundException` becomes `Rejected("class '<fqn>' could not be loaded")`, mirroring `FieldBuilder.java:1547`.
- `ParamShape` is a small sealed carrier (`FullEnvironment`, `PerArgument(List<ArgShape>)`) describing the two graphql-java-injection forms; the classifier derives it from the SDL field's `getArguments()` and the type-mapping pipeline. SDL fields with no arguments produce `PerArgument(List.of())`.

**Per-field carrier (no side-table on the parent type).**

- Per the architect-review reading: the `AccessorResolution` rides on the *field arm* representing each SDL output field whose parent is `@record`-backed, alongside `ColumnRef` / `MethodRef` / etc. on the existing field arms. No `Map<String, AccessorResolution>` slot on `PojoResultType` / `JavaRecordType`; the parent record's width stays as it is. The emitter reads the resolution from the field component it already reaches for, with no name-keyed lookup step. This eliminates the lighter version of the same string-keying smell the spec is fixing (the emitter would otherwise key into a side-map by SDL name) and dissolves the "what does the map contain for fields where resolution doesn't apply" question by construction.
- The field arms that gain the `AccessorResolution accessor` slot are those that classify a `@record`-parent SDL field today: at minimum `PropertyField` (scalar-on-result-mapped-parent, `FieldBuilder.java:2635-2636`), and the object-child arms starting at `:2640-…` whose source is a `PojoResultType` / `JavaRecordType` parent. The exact arm enumeration is a Phase A discovery — walk every site in `FieldBuilder` that today produces a field whose emit path lands in `propertyOrRecordValue` for a non-jOOQ-record parent, and add the slot there. Field arms representing jOOQ-record-backed fields, `@service`-bound fields, enum-mapped fields, and `null fqClassName` `PropertyDataFetcher`-fallback fields do *not* gain the slot — they don't classify against a backing class to begin with.

**Classifier wiring (`graphitron/src/main/java/no/sikt/graphitron/rewrite/TypeBuilder.java` + `FieldBuilder.java`).**

- `buildResultType` at `TypeBuilder.java:511-544` keeps its current shape (no map computed at the type level). The reflective resolution moves to the per-field classification site in `FieldBuilder`: when a scalar/object-child field is classified against a `@record`-backed parent (i.e. the parent's `ResultType` is a `PojoResultType` or `JavaRecordType` with non-null `fqClassName`), `FieldBuilder` calls `ClassAccessorResolver.resolve(parentClass, accessorBaseName, expectedReturnJavaType, expectedArgs, order)` (with `order` derived from the parent variant per Phase A's resolver signature) and attaches the result to the constructed field arm.
- The `@field(name:)`-or-SDL-name rule for `accessorBaseName` matches the existing `columnName` derivation at `FieldBuilder.java:2632-2634` (scalar arm) and `:2642-2644` (object arm) exactly — same source of truth, no second derivation path.
- `JavaRecordType` classification differs only in the `CandidateOrder.RECORD_FIRST` value passed to `ClassAccessorResolver.resolve`: bare-name first, then `get<Camel>` / `is<Camel>` fallback for hand-rolled methods on the record body. Resolver-side candidate ordering is the only fork; arm identity (`BareName` vs `GetterPrefixed`) holds regardless of which type variant.
- Class-load failure (`ClassNotFoundException`) on the parent `fqClassName` produces `Rejected("class '<fqn>' could not be loaded")` on every field of that type. The error surfaces once per field in the validator (matching the project's per-field `ValidationError` granularity), not once per type — the developer sees which fields the load failure affects.

### Phase B — Validator wiring + emitter consumption + `@LoadBearingClassifierCheck` pair

**Validator (`graphitron/src/main/java/no/sikt/graphitron/rewrite/GraphitronSchemaValidator.java`).**

- Add an arm in the validator's field-walking loop that picks up `AccessorResolution.Rejected` from the classified field's accessor slot (the field arms gaining the slot in Phase A) and produces a `ValidationError`. The message is the resolver's `reason()` verbatim plus a one-line trailing override hint: `"  Hint: use @field(name: \"…\") to bind this SDL field to a differently-named accessor."` The trailing hint is unconditional (we don't fuzzy-suggest, but we do tell the developer the override exists); the per-candidate diagnostic from the resolver carries the specifics.
- Per *validator mirrors classifier invariants*, the validator does not re-run reflection — it consumes the pre-attached resolution. A regression that detaches the resolution from the classifier surfaces as missing validation, which the audit-tier coverage below catches.

**`@LoadBearingClassifierCheck` pair.**

- Producer: `ClassAccessorResolver.resolve` carries `@LoadBearingClassifierCheck(key = "class-accessor-resolver-shape-guarantee", description = "AccessorResolution.Resolved guarantees a @record-backed SDL field's accessor exists on the backing class with a return type assignable to the SDL field's Java type and a param shape matching the SDL field's argument list. Validator-tier coverage in RecordFieldValidationTest mirrors this guarantee.")`.
- Consumer: `FetcherEmitter.propertyOrRecordValue` carries `@DependsOnClassifierCheck(key = "class-accessor-resolver-shape-guarantee", reliesOn = "Reads pre-resolved AccessorResolution.Resolved.method() unconditionally; the resolver rejects mismatched shapes at classify time so the emitter doesn't have to fork on candidate-name fallback or guess at runtime.")`.
- `LoadBearingGuaranteeAuditTest` picks up the pair; the producer/consumer link is auditable at audit-tier.

**Emitter (`graphitron/src/main/java/no/sikt/graphitron/rewrite/generators/FetcherEmitter.java`).**

- `propertyOrRecordValue` at `:185-213` switches exhaustively on the pre-resolved `AccessorResolution.Resolved` arms (read directly from the field component, not via a name-keyed map lookup):
  - `GetterPrefixed(method) | BareName(method)` → `($T env) -> (($T) env.getSource()).$L($args)` where `$L` is `method.getName()` and `$args` is the per-arg or full-env injection lambda body resolved from the `ParamShape`. The current shape at `:202` and `:209` collapses into one branch driven by the resolved method name.
  - `FieldRead(field)` → `($T env) -> (($T) env.getSource()).$L` where `$L` is `field.getName()` (field access; new emit form). The compile-tier fixture exercises this arm.
  - Argument-bearing accessors emit per-arg injection: for `PerArgument(List<ArgShape>)` with non-empty args, the lambda body is `<accessor>(($T) env.getArgument($S), …)` per parameter; for `FullEnvironment`, the body passes `env` through.
- The `toCamelCase + "get" + capitalize` heuristic at `:208` is deleted. The `null fqClassName` arm at `:211-212` (`PropertyDataFetcher.fetching($S)`) survives unchanged — that's the explicit "no class to introspect" fallback, out of scope.

### Phase C — Test coverage: pipeline + execution + audit

**Pipeline tier (validator-mirrors-classifier).** `RecordFieldValidationTest` (existing, at `graphitron/src/test/java/no/sikt/graphitron/rewrite/validation/RecordFieldValidationTest.java`) gains a focused test per shape:

- `pojoRecordField_missingGetter_validatorReportsActionableError` — the user's case. Schema with `sakId: ID`, POJO with only `getSak(): SakRecord`. Asserts the `ValidationError` message names the SDL field (`Type.sakId`), the backing class FQN, the candidates tried (`getSakId`, `sakId` bare-name), and the `@field(name:)` override hint.
- `javaRecordField_missingComponent_validatorReportsActionableError` — Java record backing class with no component matching the SDL field name, no `@field(name:)` override.
- `pojoRecordField_returnTypeMismatch_validatorReportsActionableError` — `getSakId(): SakRecord` against `sakId: ID` (resolves to `String`). Asserts the diagnostic distinguishes "found by name, return type incompatible (`SakRecord` not assignable to `String`)" from "not found by name".
- `pojoRecordField_argumentBearingFieldMatchingMethod_resolves` — positive test: SDL field with arguments, POJO method with matching params (`fooBar(String x): String` against `fooBar(x: String): String`), asserts no validation error and the resolved `Method` arm carries the expected param count.
- `pojoRecordField_fieldNameOverride_resolvesAgainstOverride` — `@field(name: "sak")` redirects resolution to `getSak()`. Asserts the override path is honoured and no validation error is raised when the override matches a real method, even though the SDL name (`sakId`) does not.
- `javaRecordField_bareNameAccessor_resolves` — positive test for the `BareName` arm on a Java record's canonical component accessor.
- `pojoRecordField_publicFieldFallback_resolvesAsFieldRead` — positive test for the `FieldRead` arm: schema field has no matching method, but a public field matches by name and type. Asserts the classified field's `accessor` is an `AccessorResolution.Resolved.FieldRead` carrying the resolved `Field` handle.

Validator-tier tests are pipeline tier per the rewrite test taxonomy (`docs/rewrite-design-principles.adoc:103-130`); they instantiate `GraphitronSchemaBuilder` + `GraphitronSchemaValidator` against a real SDL + Java classpath fixture and assert on the produced `ValidationError` list. No code-string assertions on emitted method bodies.

**Execution tier (emit-shape backstop).** A new `@record`-backed type lands in `graphitron-sakila-example` exercising all three resolution arms in one fixture: a `RecordExampleType` SDL type with three fields (one resolved via `getXxx`, one via bare-name accessor on a Java record component, one via `@field(name:)` override). The fixture extends an existing query (no new top-level field; the addition stays minimal). `mvn install -Plocal-db` compiles the generated sources via `javac`; an emitter regression that produces non-compiling code surfaces at the compile boundary. Per `docs/rewrite-design-principles.adoc:136-140` this is the project's standard emit-shape backstop tier. The fixture also serves as a working example developers can read for the conventions.

**Audit tier.** `LoadBearingGuaranteeAuditTest` picks up the new `@LoadBearingClassifierCheck` / `@DependsOnClassifierCheck` pair from Phase B automatically. The audit test enforces that every `@LoadBearingClassifierCheck` has at least one matching `@DependsOnClassifierCheck` with the same key and that the `reliesOn` strings document the producer's `description` shape — a rename or detachment of the classifier guarantee surfaces here without per-test maintenance.

## Out of scope

- **Input-side ctor / setter validation.** `@record`-mapped *input* types (`JavaRecordInputType`, `PojoInputType`) bind via constructor parameters or setters; an analogous validation gap exists there but the resolution mechanic is structurally different (positional ctor binding vs name-based getter lookup), and the existing payload-errors path at `FieldBuilder.java:1542-1589` already covers the mutation-payload-with-errors case. A sibling roadmap item should generalise that path; opening it under a separate ID keeps this spec's scope to read-side accessors. Filed as a follow-up.
- **`PropertyDataFetcher` fallback removal.** The `null fqClassName` arm at `FetcherEmitter.java:211-212` falls back to graphql-java's `PropertyDataFetcher`, which does its own runtime accessor resolution. The user's note ("we shouldn't use the `PropertyDataFetcher` since we can do better, but we should match the rules") frames this item's resolution-rule alignment as the prerequisite, with a follow-up that removes the fallback in favour of `ClassAccessorResolver` even when no `fqClassName` is declared (resolving instead against the SDL field's runtime `env.getSource().getClass()` at first-call). That requires either runtime resolution or upstream classification of the source-side type, neither of which fits this item's "classifier-side eager resolution" framing. Filed as a follow-up.
- **Lombok / explicit-named accessor extensions.** Custom accessor name patterns (Lombok's `@Accessors(fluent = true)`, etc.) are not part of graphql-java's `PropertyDataFetcher` conventions and aren't supported here. The sealed `Resolved` hierarchy admits a future extension arm (e.g. `LombokAccessor`) by extending its permit list, with the emitter's exhaustive switch breaking compile until the new arm is wired through.
- **Levenshtein-style "did you mean" suggestions.** The diagnostic lists candidates tried (with reasons each was rejected) and the override hint, but does not fuzzy-match nearby method names. The project's regular diagnostic style is precise enumeration, not heuristic suggestion.
- **Existing payload-errors reflective path consolidation.** `FieldBuilder.resolveErrorChannel` at `:1542-1589` does its own `Class.forName` for ctor-shape validation. Reusing `ClassAccessorResolver` there is an attractive consolidation but mixes two reflection concerns (ctor shape vs accessor shape); leave the existing path alone in this item, revisit if the input-side follow-up subsumes it.
