# Validate `@record` accessor resolution against backing class at classify time

A downstream consumer (Sikt opptak-subgraph) hit a generator regression that surfaced only at downstream `javac`: a `@record`-mapped GraphQL type with `sakId: ID` against a Java POJO that exposes `getSak(): SakRecord` (no `getSakId()`) compiled the generator clean, then failed `javac` on the generated `BeregnPoengAutomatiskResultatType.java` with `cannot find symbol: method getSakId()`. The mismatch is invisible to graphitron's pipeline because `FetcherEmitter.propertyOrRecordValue` at `graphitron/src/main/java/no/sikt/graphitron/rewrite/generators/FetcherEmitter.java:204-209` synthesizes the getter name purely from the SDL column-name string (`"get" + capitalize(toCamelCase(columnName))`) without ever consulting the backing class's method set. The same purely-string-derived heuristic governs the `JavaRecordType` arm at `:199-202` (bare-name accessor for Java records), so the gap covers both `@record`-on-POJO and `@record`-on-Java-record output types.

The principle this violates is *Classifier guarantees shape emitter assumptions* (`docs/rewrite-design-principles.adoc:107-122`). The classifier (`TypeBuilder.buildResultType`, `graphitron/src/main/java/no/sikt/graphitron/rewrite/TypeBuilder.java:511-544`) currently produces a `PojoResultType(name, location, fqClassName)` (resp. `JavaRecordType(...)`) carrying only the class's binary name. Whether that class actually exposes a method matching the SDL fields is left for the emitter to discover via string concatenation, with no fallback when the heuristic is wrong. The reflection-boundary roster (`docs/rewrite-design-principles.adoc:29`) places `Class.forName + reflective lookup` at the classifier layer; the existing payload-errors path at `FieldBuilder.java:1542-1589` is the local precedent for that pattern, but it fires only when the SDL declares an errors-shaped field. Plain `@record` types never trigger it.

The fix is to lift accessor resolution into the classifier and have it fail closed: at classify time, for every SDL output field of a `@record`-backed type, resolve the call shape against the backing class's `Class<?>` reflectively, attach a sealed `AccessorResolution` (`Resolved(...)` carrying the resolved `Method` plus arm identity, or `Rejected(reason)` carrying an actionable diagnostic) to the field's classification, and propagate `Rejected` through `UnclassifiedField` so `GraphitronSchemaValidator` reports a `ValidationError` naming the SDL field, the backing class, and the candidates that were tried. The emitter consumes the pre-resolved `Method` handle and never recomputes the heuristic. This matches *validator mirrors classifier invariants* (`docs/rewrite-design-principles.adoc:103-105`).

Resolution rules follow graphql-java's `PropertyDataFetcher` lookup order, so consumers migrating from `PropertyDataFetcher` to graphitron's `@record`-emitted fetcher get matching name resolution:

- The base name comes from `@field(name:)` if present (the existing override sites are `FieldBuilder.java:2636-2638` for the scalar-on-record-parent arm and `:2646-2648` for the object-child arm), otherwise the SDL field name. The validator hooks the same pipeline.
- Candidate accessors, in resolution order: `get<CamelName>()` (POJO bean getter, any return type); `is<CamelName>()` (only when the SDL field's resolved Java type is `boolean`/`Boolean`); bare `<camelName>()` (Java record component / fluent accessor); public field access `<camelName>` (graphql-java's last-resort field-read path).
- If the SDL field has arguments, the accessor must declare matching parameters per graphql-java conventions: either a single `DataFetchingEnvironment` parameter for full-env injection, or per-arg parameters whose Java types match each SDL arg's resolved Java type, in declared order. A zero-arg method on a SDL field with arguments is a `Rejected`, not a silent argument-drop.
- The resolved method's return type must be assignable to the SDL field's resolved Java type (e.g. `String` for `ID`, `BigDecimal` for `BigDecimal`). A method that exists by name but returns the wrong type is `Rejected` with the type mismatch as the diagnostic, not a silent acceptance that defers the failure to `javac`. This is the arm that catches the user's `getSakId(): SakRecord` case if a developer rewrites the getter to match by name without changing the return shape.

The diagnostic message follows the project's standard validator surface: name the SDL field by parent + name (e.g. `BeregnPoengAutomatiskResultat.sakId`), the backing class FQN, the accessor candidates tried with reasons each was rejected (no method by that name; method exists but parameter shape doesn't match; method exists but return type incompatible), and the override hint (`@field(name: "sak")` to bind to `getSak()`). Same tone as `ServiceCatalog`'s `Rejection.structural` messages and the `ConditionResolver` field/arg arms at `ConditionResolver.java:96-137`.

A first-pass spec carried (a) a `NotApplicable` arm on `AccessorResolution` so the value type was total over every classified field, (b) a nested `AccessorShape` discriminator inside `Resolved` with a shared `method()` accessor, and (c) a `Map<String, AccessorResolution>` slot on `PojoResultType` / `JavaRecordType` keyed on SDL field name. Architect review (in-session, before hand-off) flagged all three as friction with the principles: `NotApplicable` is the empty-arm smell (`docs/rewrite-design-principles.adoc:51` — "each new sub-taxonomy comes with a one-line note on what distinct information it carries"), nested sealing with shared accessors duplicates the BatchKey "narrow component types over broad interfaces" tension (`:39-43`), and a side-table keyed by SDL name is a lighter version of the same string-derivation smell the spec is fixing. This draft adopts the revisions: `AccessorResolution = Resolved | Rejected` with `Resolved` directly sealed over `GetterPrefixed(Method)` / `BareName(Method)` / `FieldRead(Field)` (each arm carries exactly its data), no `NotApplicable`, and the `AccessorResolution` rides on the per-field arm alongside `ColumnRef` / `MethodRef` rather than on a side-table on the parent type. Fields where reflective resolution doesn't apply are classified into different field arms entirely; the structural arm identity is the "this resolution doesn't apply" signal.

## Implementation

- **Phase A — Sealed `AccessorResolution` model + `ClassAccessorResolver` + per-field carrier:** shipped at `0bcb6ebe`.
- **Phase B — Validator wiring + emitter consumption + `@LoadBearingClassifierCheck` pair:** shipped at `b2d798a9`.
- **Phase C — Test coverage (pipeline + execution + audit):** shipped at `e5f2dd2e`. Pipeline-tier tests landed in a new `RecordFieldAccessorValidationTest.java` (`@PipelineTier`) rather than the existing `@UnitTier` `RecordFieldValidationTest.java` — keeping tier discipline per file. Audit-tier coverage came for free through `LoadBearingGuaranteeAuditTest` auto-picking up the producer/consumer pair, no new test code needed.
- **Phase D — Documentation refresh:** shipped at `e5f2dd2e`. Reflection roster in `docs/rewrite-design-principles.adoc:29` lists `ClassAccessorResolver` as the fifth permitted boundary-crossing file. `docs/code-generation-triggers.adoc` was checked and does not name the accessor heuristic, so no change there.

### Implementation deviations from spec

- **Per-arg injection in the emitter** uses `Method.getParameters()` for argument names, requiring the backing class to be compiled with `-parameters`. The spec contemplated threading SDL argument names through the field arm, but per the spec's own "raise a Spec amendment rather than widen the threading ad hoc" clause, the parameter-name fallback is the lighter route. If `-parameters` is absent on a candidate method, `methodCallExpr` throws at emit time with a clear error, surfacing the case rather than producing silently-broken code.

### Phase E — Rework: tighten accessor slot to `Resolved`; route `Rejected` through `UnclassifiedField`

In-review feedback flagged the Phase B defensive fallback (the `propertyOrRecordValue` tail kept the pre-R88 `toCamelCase + "get" + capitalize` heuristic as a runtime fallback for `null` or `Rejected` accessors) as a regression on the spec's whole point. `GraphQLRewriteGenerator.generate()` calls `validateAndLogErrors` before any emit, so in production the fallback is unreachable; it exists only to keep test fixtures that bypass `validate()` deterministic, which is precisely "fallbacks for scenarios that can't happen". Worse, the emitter's `@DependsOnClassifierCheck.reliesOn` claim ("Reads pre-resolved `AccessorResolution.Resolved.method() / .field()` unconditionally") is materially false while the fallback lives — the audit-tier link passes, but the documented load-bearing guarantee no longer holds.

The deeper bug Phase B exposed is a typing weakness: the accessor slot on `PropertyField` / `RecordField` is `AccessorResolution` (the union, including `Rejected`), so the emitter has to handle non-`Resolved` values at all, so a fallback looks necessary. Phase E removes the type weakness; the fallback then has nowhere to live.

**Classifier (`FieldBuilder`).** When `ClassAccessorResolver.resolve()` returns `Rejected`, do **not** construct a `PropertyField` / `RecordField` with `accessor = Rejected`. Construct an `UnclassifiedField(Rejection.structural(rejected.reason()))` instead, mirroring the structural-rejection routing the spec called for at line 17 ("propagate `Rejected` through `UnclassifiedField`") and matching every other classify-time failure shape. The accessor-rejection reason flows into the same diagnostic surface the existing `UnclassifiedField` arms already use.

**Field arms (`graphitron/src/main/java/no/sikt/graphitron/rewrite/model/ChildField.java`).** Tighten the slot type so the union narrows: when non-null, `PropertyField.accessor()` and `RecordField.accessor()` return `AccessorResolution.Resolved` (never `Rejected`). The slot stays nullable to carry the existing "this parent isn't accessor-resolved" cases — jOOQ-record-backed parents and `null fqClassName` `PropertyDataFetcher` fallback parents both produce a `PropertyField` whose accessor slot is null today, and Phase E preserves that. The contract change is "null OR `Resolved`, never `Rejected`", documented at the slot's javadoc; option (b) of splitting `PropertyField` into `RecordPropertyField` / `JooqPropertyField` arms (pushing the parent-shape distinction into the type system) is more principle-aligned but a scope expansion, deferred. The static type now carries the load-bearing guarantee for the `Resolved`-or-null contract; the `@DependsOnClassifierCheck.reliesOn` claim becomes literally true (non-null implies `Resolved`) and the audit-tier coverage is no longer documenting a runtime assertion.

**Emitter (`FetcherEmitter.propertyOrRecordValue`).** Switch exhaustively on the three `Resolved` arms (`GetterPrefixed`, `BareName`, `FieldRead`); delete the `toCamelCase + "get" + capitalize` heuristic and the null/`Rejected` branches. A future arm added to `Resolved` becomes a compile error here, which is the point.

**Validator (`GraphitronSchemaValidator`).** Drop the `Rejected`-from-accessor-slot pickup added in Phase B's `validatePropertyField` / `validateRecordField`; the existing `UnclassifiedField` validation path picks up the rejection (one pickup site, not two). The `@field(name:)` override hint that Phase B's pickup appended moves to wherever `UnclassifiedField` rejections format their diagnostic, gated on a typed rejection-kind discriminator. Two acceptable shapes: extend the existing `Rejection.AuthorError.UnknownName` `AttemptKind` enum with an `ACCESSOR` value and let the override hint attach when `attemptKind == ACCESSOR`, or add a sibling `AccessorMismatch(reason)` arm under `AuthorError` and switch on it. Either way the discrimination rides on a typed value; **message-prefix or substring discrimination is not acceptable** — that is the kind-in-prose anti-pattern *Builder-step results are sealed, not strings or out-params* (`docs/rewrite-design-principles.adoc:61-73`) is written to forbid.

**Tests bypassing `validate()` (the original justification for the fallback).** Each fixture / test that tripped the fallback path falls into one of three categories; Phase E walks them all:
1. Tests asserting on classifier behaviour without running the validator: assert on `UnclassifiedField` directly (which is what the field becomes under Phase E), not on the emitter.
2. Tests wanting successful emission that were stumbling over a missing fixture method: fix the fixture (add the method or the `@field(name:)` override) so resolution succeeds and a real `Resolved` flows through.
3. Tests of "emitter behaviour on a Rejected field": delete; this is testing a code path that no longer exists, and shouldn't.

**Audit-tier.** The `LoadBearingGuaranteeAuditTest` link still exercises the producer/consumer pair; the consumer's `reliesOn` claim is now type-system-enforceable rather than a runtime assertion. The producer's `@LoadBearingClassifierCheck.description` text on `ClassAccessorResolver.resolve` needs a stale-reference audit in the same commit — the Phase B description was written when `Rejected` could flow through the model to the emitter; under Phase E the consumer reads `Resolved` (or null, never `Rejected`), and any wording that still mentions `Rejected` reaching downstream emitter consumption must be rewritten. Same shape of stale-reference risk as Phase D's `rewrite-design-principles.adoc:29` roster update, under *Documentation names only live tests/code* (`docs/rewrite-design-principles.adoc:134-137`).

**Pipeline-tier coverage from Phase C.** The existing rejection-case tests in `RecordFieldAccessorValidationTest` (`pojoRecordField_missingGetter_*`, `javaRecordField_missingComponent_*`, `pojoRecordField_returnTypeMismatch_*`) currently assert on the validator-emitted message; under Phase E they continue to work end-to-end because the validator pickup via `UnclassifiedField` produces the same diagnostic. Add **one** structural pipeline-tier assertion per test class — at least one rejection-case test should additionally assert that `schema.field("TestType", "<rejected-field>")` returns an `UnclassifiedField` (not a `PropertyField` with a `Rejected` accessor). This makes the type-tightening visible at the tier where the model lives and locks in the routing change explicitly. The positive-case tests (`pojoRecordField_argumentBearingFieldMatchingMethod_resolves`, etc.) already assert on the `Resolved` arm shape and need no change.

**Execution-tier coverage from Phase C.** The `RecordExampleType` fixture is unaffected — all three of its fields resolve to `Resolved`, so the routing change is invisible at the emit / execute boundary.

**Out of scope for Phase E.** The `null fqClassName` `PropertyDataFetcher` fallback at `FetcherEmitter.java:211-212` (the explicit "no class to introspect" arm) stays. That's a different fallback — the parent type doesn't have a backing class at all, not "the backing class rejected resolution"; the existing "Out of scope" item already covers its eventual removal.

## Out of scope

- **Input-side ctor / setter validation.** `@record`-mapped *input* types (`JavaRecordInputType`, `PojoInputType`) bind via constructor parameters or setters; an analogous validation gap exists there but the resolution mechanic is structurally different (positional ctor binding vs name-based getter lookup), and the existing payload-errors path at `FieldBuilder.java:1542-1589` already covers the mutation-payload-with-errors case. A sibling roadmap item should generalise that path; opening it under a separate ID keeps this spec's scope to read-side accessors. Filed as a follow-up.
- **`PropertyDataFetcher` fallback removal.** The `null fqClassName` arm at `FetcherEmitter.java:211-212` falls back to graphql-java's `PropertyDataFetcher`, which does its own runtime accessor resolution. The user's note ("we shouldn't use the `PropertyDataFetcher` since we can do better, but we should match the rules") frames this item's resolution-rule alignment as the prerequisite, with a follow-up that removes the fallback in favour of `ClassAccessorResolver` even when no backing class is declared (resolving instead against the SDL field's runtime `env.getSource().getClass()` at first-call). That requires either runtime resolution or upstream classification of the source-side type, neither of which fits this item's "classifier-side eager resolution" framing. Filed as a follow-up.
- **Lombok / explicit-named accessor extensions.** Custom accessor name patterns (Lombok's `@Accessors(fluent = true)`, etc.) are not part of graphql-java's `PropertyDataFetcher` conventions and aren't supported here. The sealed `Resolved` hierarchy admits a future extension arm (e.g. `LombokAccessor`) by extending its permit list, with the emitter's exhaustive switch breaking compile until the new arm is wired through.
- **Levenshtein-style "did you mean" suggestions.** The diagnostic lists candidates tried (with reasons each was rejected) and the override hint, but does not fuzzy-match nearby method names. The project's regular diagnostic style is precise enumeration, not heuristic suggestion.
- **Existing payload-errors reflective path consolidation.** `FieldBuilder.resolveErrorChannel` at `:1542-1589` does its own `Class.forName` for ctor-shape validation. Reusing `ClassAccessorResolver` there is an attractive consolidation but mixes two reflection concerns (ctor shape vs accessor shape); leave the existing path alone in this item, revisit if the input-side follow-up subsumes it.
