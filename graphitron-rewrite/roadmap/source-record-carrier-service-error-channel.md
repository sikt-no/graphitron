# Error channel and data projection for source-record-carrier @service mutations

A `@service` mutation whose service method returns a single jOOQ table `Record` (not a `@record`-annotated payload object) into a payload type that pairs a `@splitQuery`/`@table` data field with an `errors` union field gets **no error channel and no data-field projection**. This shape is pervasive in `opptak-subgraph`: every saksbehandling mutation migrated from admissio uses it (`tilordneSaksbehandlerV3`, `fjernTilordnetSaksbehandler`, `settMerknadV2`, `endreSaksbehandlingsstatusV3`, `kvalifiserKravelementV3`), with payloads shaped `{ sak: Sak @splitQuery, errors: [SomeError!] }` and a service returning `SakRecord`. It accounts for 16 of the 19 failsafe failures in `opptak-subgraph/target/failsafe-reports` (see the failure map below).

R244 built the typed `Outcome<T>` channel, but only for `@service` mutations whose method returns a `@record` payload (`ReturnTypeRef.ResultReturnType`); `FieldBuilder.resolveServiceOutcomeChannel` (`FieldBuilder.java:~2068-2119`) returns `NoChannel` for anything else. R268 (shipped, `fc365b4`) added the `@table`-bound data-field arm-switch, but only *inside* those Outcome payloads. DML uses the localContext transport. The source-record-carrier `@service` shape classifies as `MutationServiceTableField` over a `TableBoundReturnType` (`FieldBuilder.java:~3196-3198`) and falls through all three: its generated fetcher has no `Mapping[]`, and its payload's data field gets no registered datafetcher. This item brings that shape onto the existing `Outcome` transport so the R244 error mapping and R268 arm-switch resolve both the data field and the errors field, and fixes the success-arm errors value to honour SDL nullability.

**Post-R276 symptom note (2026-06-05).** R276 (carrier unification) has since landed on trunk, so every carrier is now a `ResultType` and the producer fetcher already wraps its result in the `Outcome` runtime type: the generated producer returns `Outcome.Success(record)` / `Outcome.ErrorList(...)` rather than a bare record. What is still missing is the consumer side of the carrier on this shape: the payload's `sak` data field has no arm-switch datafetcher, so the default `PropertyDataFetcher` now receives an `Outcome` as its source and the field's generated read casts it straight to `SakRecord`. The pre-R276 symptom (silent `null` from a missing `getSak()`, redaction to a generic `DataFetchingException`) has therefore hardened into a `ClassCastException` (`Outcome$Success` / `Outcome$ErrorList cannot be cast to SakRecord`) surfaced as a `DataFetchingException` on the `/<mutation>/sak` path. This is the same defect with a louder failure mode, and the implementation below (data-field arm-switch over `success.value()`, errors-arm `null`) is unchanged by it.

## Design: route the source-record carrier through the existing `Outcome` transport

`Outcome<T>` is the right carrier and is already non-null by construction, which is what closes the localContext silent-errors-drop that R244 documented. The move is to make the classifier recognise one more producer shape and reuse everything downstream.

The relationship that makes this clean: in R244's payload-returning case `Outcome.Success.value()` holds the `@record` payload and the payload's fields read off it; here `Outcome.Success.value()` holds the returned jOOQ record (`SakRecord`), and the payload's single `@splitQuery`/`@table` data field (`sak`) is a split-query field whose correlation key is exactly the record's primary-key columns. That is precisely R268's `@table`-bound arm-switch (`RecordTableField` / `RecordLookupTableField`): narrow `env.getSource()` to `Outcome.Success`, read the key off `success.value()`, dispatch the DataLoader, and return `completedFuture(null)` on the `ErrorList` arm. No new transport, no new arm-switch; the only novelty is that `success.value()` is the table record itself rather than a payload wrapping it.

The producer fetcher becomes the R244 shape it should have had: run the `@service`, return `Outcome.Success(record)` on the happy path, walk the per-field `ErrorMappings.<CONST>` and return `Outcome.ErrorList(List.of(cause))` on the first match, fall through to `ErrorRouter.redact` on no match. That single change converts bucket B from redacted to typed.

### Two carrier invariants, mirrored

The carrier is fully nullable on both arms, pinned by two classify-time rejection rules so a violating schema cannot be built:

* **Data/success field must be nullable** (existing `NonNullableSuccessProjectionField`): a non-null data field would raise `NonNullableFieldWasNullError` on the error arm and drop the sibling errors field. Already enforced; applies unchanged to the source-record carrier.
* **Errors field must be nullable** (new, this item): reject a payload whose `errors` field carries a non-null list type (`[X!]!`). This is the mirror of the success-projection rule: it is what makes `null` a legal success-arm value for the errors field, so the success arm can always emit `null`. A non-null errors field is an author mistake, surfaced as a classification error, not a generator limitation.

With both rules in force, the success arm emits `null` for the errors field unconditionally; there is no nullable-vs-non-null branch to maintain at the emit site.

## Implementation

* **`FieldBuilder` channel detection.** Generalise the service-outcome classification so a `@service` field over a `TableBoundReturnType` whose SDL payload carries an `errors` field (with at least one `@error` handler) and exactly one non-errors data field is classified as an `OutcomeType` (`ErrorChannel.Mapped`) rather than a channel-less `MutationServiceTableField`. The `OutcomeType`'s `successProjection` is that single data field; the `Outcome.Success` value type is the returned jOOQ record. This needs no `OutcomeType` / `Outcome.Success` widening: `JooqTableRecordType` is already a `GraphitronType.ResultType` (`GraphitronType.java:93`), the same backing family R244's `@record`-payload case wraps, so the carrier holds it without a type change. Keep the channel-less `MutationServiceTableField` path for service-table fields that have no `errors` field (unchanged behaviour). Coordinate the `successProjection` population with R274 (see Coordination).
* **Producer fetcher emit.** For the new classification, emit the R244 catch-arm shape (the `ErrorMappings.<CONST>` walk producing `Outcome.ErrorList`, `redact` fallback) and the `Outcome.Success(record)` happy path, replacing today's bare `redact`-only catch. Reuse `ChannelCatchArmEmitter` / `ErrorMappings` generation already built by R244.
* **Data-field projection.** Register the data field's datafetcher via R268's `@table`-bound arm-switch (the `RecordTableField` / split-query path), source-bound to `success.value()`, with `completedFuture(null)` on the `ErrorList` arm. This is the missing registration that fixes bucket D; it is the same emit R268 produces for a `@table` data field sitting next to an errors field inside an Outcome payload.
* **Errors success-arm value.** Change `FetcherEmitter`'s `WrapperArm` transport so the success arm resolves the errors field to `null` instead of `List.of()`. Safe by the new errors-nullable rule. This fixes bucket C and the bucket-B/D success assertions.
* **New rejection rule.** Add an `ErrorChannelWalkerError` arm (e.g. `NonNullableErrorsField`) in the output-walking `ErrorChannelWalker` / `GraphitronSchemaValidator`, alongside `MultipleErrorsFields` and `NonNullableSuccessProjectionField`, with a matching LSP `graphitron.error-channel.*` diagnostic code and a `typed-rejection.adoc` entry.

## As built (2026-06-05)

The plan above predates R244/R268/R276 landing on trunk; the as-built diff is smaller than the plan implies, because two of the three pieces were already in place once R276 unified the carriers:

* **Producer + classification: already done, no change.** Post-R276 the source-record carrier's producer already classifies with an `ErrorChannel.Mapped` and emits the full R244 shape (`Outcome.Success(record)` happy path, `ErrorMappings.<CONST>` walk to `Outcome.ErrorList`, `ErrorRouter.redact` fallback). Verified against the generated `MutationFetchers.tilordneSaksbehandlerV3` in `opptak-subgraph`. So the "FieldBuilder channel detection" and "Producer fetcher emit" bullets needed no work; there is no new `OutcomeType`-vs-`MutationServiceTableField` reclassification. The bullets' framing ("channel-less `MutationServiceTableField`", "bare `redact`-only catch") describes the pre-R276 state and is stale.
* **The only real gap was the consumer side of the carrier (buckets B + D).** The data field is a `ChildField.SingleRecordTableField` (`SourceKey.Wrap.TableRecord`, `Reader.ResultRowWalk`), not a `RecordTableField`; its inline split-query fetcher (`FetcherEmitter.buildSingleRecordTableFetcherValueTableRecordWrap`) cast `env.getSource()` straight to the jOOQ record. Post-R276 that source is the non-null `Outcome`, so the cast threw `ClassCastException` (`Outcome$Success`/`Outcome$ErrorList cannot be cast to SakRecord`) on both arms. Fix: the source envelope rides on the field's `SourceKey` as `Reader.ResultRowWalk.SourceEnvelope` (`DIRECT` for the DML carrier, `OUTCOME_SUCCESS` for the `@service` error-channel carrier), set at classification in `FieldBuilder` from `carrierPayloadHasErrorsField(payload)` (the same errors-field signal that gives the producer its channel). The `OUTCOME_SUCCESS` arm narrows `Outcome.Success`, reads the record off `success.value()`, and returns `null` on the `ErrorList` arm. A `SourceKey` compact-constructor invariant pins `OUTCOME_SUCCESS` to `Wrap.TableRecord`, so only the `TableRecord` emit arm carries the narrowing; the `Record` (DML) arm is untouched. This is a deliberate departure from the plan's "reuse R268's `RecordTableField` arm-switch": the source-record carrier's data field is a `SingleRecordTableField`, a different emit path, and the envelope-on-the-field model keeps the emitter from re-deriving the carrier shape from sibling fields (`hasWrapperArmErrors`).
* **Bucket C and the rejection rule: as planned.** `FetcherEmitter`'s `WrapperArm` success arm now resolves `null` instead of `List.of()`. `NonNullableErrorsField` was added as a typed `ErrorChannelWalkerError` arm; rather than a new producer-level check, the pre-existing structural "errors-shaped list field must be nullable" rejection in `FieldBuilder.liftToErrorsField` was upgraded to raise it (so the errors field's own nullability check, where it already lived, now carries the LSP code). `successProjection` was left unpopulated (R274 still owns that consolidation); the data-field nullability check stays inline in `resolveServiceOutcomeChannel`.
* **As-built test coverage.** Execution: a new sakila fixture `serviceFilmByIdWithErrors` (`FilmCarrierWithErrorsService` returning a bare `FilmRecord` into `FilmCarrierWithErrorsPayload { film: Film, errors: [FilmReviewError] }`) with happy-arm (`film` read off `success.value()`, `errors: null`) and error-arm (`film: null`, typed error in `errors`) tests in `GraphQLQueryTest`. The data field is a to-one `Film` (single), not the opptak `@splitQuery` list, but it exercises the same `SingleRecordTableField` `OUTCOME_SUCCESS` arm. Pipeline: `SingleRecordTableFieldServiceProducerPipelineTest.serviceProducer_withErrorsField_recordsOutcomeSuccessEnvelope` asserts the data field is a `SingleRecordTableField` whose `ResultRowWalk` envelope is `OUTCOME_SUCCESS` and the sibling errors field is `WrapperArm`; the existing no-errors test now also asserts the `DIRECT` envelope. Unit: `SourceKeyTest` pins the `OUTCOME_SUCCESS ⟹ Wrap.TableRecord` compact-constructor invariant (accept + reject); the `NonNullableErrorsField` rejection is covered by `GraphitronSchemaBuilderTest`'s `NON_NULL_LIST_OF_ERRORS_REJECTS` and the LSP `RejectionSeverityCoverageTest` sample. The planned "classifies as `OutcomeType`" pipeline assertion does not apply: the data field is a `SingleRecordTableField`, the carrier model is not reclassified.

## Reopened scope (2026-06-10): `@splitQuery`-list and `@nodeId`-from-record source-record carriers

The As-built slice above closed the **to-one, non-`@splitQuery`** source-record carrier (`{ entity: Table, errors }`, projected off `success.value()`). It did not cover the two data-field shapes the `opptak-subgraph` saksbehandling mutations actually use, and both currently produce an **invalid assembled schema** rather than a runtime error — strictly worse than buckets B/D:

- **`@splitQuery` list:** `LeggTilTaggerPayload { saker: [Sak!] @splitQuery, errors: [LeggTilTaggerError] }` (service returns `List<SakRecord>`).
- **`@nodeId` from the record:** the delete mutations `fjernSakTagg`/`fjernSakTagger` — `FjernSakTaggerPayload { tagger: [ID] @nodeId(typeName: "Tagg"), errors }` (service returns the `SoknadsbehandlingTaggRecord`(s)). Note the original `SakTaggPayload { tagg: Tagg, errors }` shape was the *delete-then-re-fetch* defect: a `@table` node field re-`SELECT`s a row the mutation just deleted and resolves to `null`; the schema is asking the server to break its own contract, which is why the consumer reframed the field to echo the id from the service result instead.

**Symptom and root cause.** The payload SDL Object is never classified or registered (`GraphitronSchema.type("…Payload") == null`), yet the mutation field still classifies — `MutationServiceRecordField` with a `ScalarReturnType` over the unregistered payload — and emits `typeRef("…Payload")`. graphql-java assembly then fails with `type …Payload not found in schema`. Two distinct drop paths feed this one symptom:
- `@splitQuery` on the carrier's data field is in `BuildContext.FORBIDDEN_CARRIER_DATA_FIELD_DIRECTIVES`, so `scanStructuralDmlPayload` returns `NotApplicable` and the carrier is never recognized.
- An `[ID] @nodeId` data field is recognized by `scanStructuralDmlPayload` only as `DmlElementKind.IdElement`, admitted solely for `@mutation(typeName: DELETE)`; on an `@service` mutation it grounds no producer binding (`groundServicePayloadBinding` requires a `@table` data field), so the carrier is dropped.
In both cases the scan's `NotApplicable`/`Reject` is **swallowed on the `@service` path** (it is surfaced as an author error only on the DML `@mutation` path), so there is no validation diagnostic — the first sign is the broken assembled schema.

**Two requirements this slice must satisfy (per maintainer, 2026-06-10):**
1. **Never emit an invalid schema.** A dangling `typeRef` to a dropped payload type is a critical bug. A source-record `@service` carrier shape the classifier cannot support must be a loud author error at classification time (`UnclassifiedField` with a clear reason + an LSP `graphitron.*` diagnostic), never a silently-dropped type.
2. **Support `@nodeId` on the result of a service mutation when the data field is a jOOQ table record — without a re-fetch.** The service already returns the record(s) on `success.value()`; `@nodeId` must encode the PK columns straight off that in-memory record into the node id, the same `success.value()` seam R275 already uses to read the split-query key. No DB round-trip, deletion-safe by construction — this is the fix for the `fjernSakTagg`/`fjernSakTagger` delete-then-re-fetch failures. The `@splitQuery`-list shape (`saker: [Sak!] @splitQuery`) is the parallel case where the entity *survives* the mutation; it should project via the existing `OUTCOME_SUCCESS` split-query path, just over a list.

**Red tests already in place (drive this slice).** `graphitron/src/test/java/no/sikt/graphitron/rewrite/GraphitronSchemaBuilderTest.java`, enum `RootFieldCase`:
- `SERVICE_MUTATION_SPLITQUERY_CARRIER_DROPS_PAYLOAD_TYPE` (**failing**): asserts the `@splitQuery`-list source-record carrier keeps its payload type (`schema.type("FilmsPayload")` non-null, field is `MutationServiceRecordField`). Currently red on `isNotNull` (type dropped).
- `SERVICE_MUTATION_ID_CARRIER_SILENTLY_DROPS_PAYLOAD` (**failing**): asserts the `[ID!] @nodeId` source-record carrier is rejected as an `UnclassifiedField` (loud author error) rather than accepted into a dangling `typeRef`. Currently red (it is a `MutationServiceRecordField` over the dropped payload).
- Green baselines for contrast: `SERVICE_MUTATION_SINGLE_RECORD_CARRIER_OVER_RESOLVABLE_TABLE` and `SERVICE_MUTATION_LIST_RECORD_CARRIER_OVER_RESOLVABLE_TABLE` (same carrier, no forbidden directive — already classify correctly).

**Caveat to resolve before coding.** The two red tests are currently **errors-less** carriers, which drop via the `scanStructuralDmlPayload` `FORBIDDEN`/ID path. The opptak payloads are **errors-bearing**, which route through R275's error-channel recognition (`carrierPayloadHasErrorsField`) — a different path. So the first step re-casts them errors-bearing once verified, to ensure the fix lands in R275's `@service` path and not the DML scan path.

**What we intend to do now.**
1. Verify the errors-bearing `@splitQuery`-list and `@nodeId` carrier behavior in the harness; pin which path drops them.
2. Re-cast the two red tests as errors-bearing to match the opptak shapes and R275's path.
3. Extend the classifier so a source-record `@service` carrier with (a) a `@splitQuery` `@table`-list data field projects over the `OUTCOME_SUCCESS` list path, and (b) an `@nodeId` ID data field encodes from `success.value()`'s record PK with no re-fetch; and so any unsupported source-record carrier shape becomes a classification-time author error, never a dropped type.

### As built, reopened-scope slice 1 (2026-06-10): `@splitQuery` carrier admitted, ID carrier rejected loudly

Steps 1-2 and the (a) + author-error halves of step 3 are done; (b), `@nodeId`-from-record support, is the remaining slice (see below).

* **Verification (step 1).** Errors-bearing variants drop through the same two paths as the errors-less red tests; the errors field is skipped by the structural scan (`detectErrorsFieldShape`) and changes nothing about the drop. One interplay pinned along the way: an `@error` type missing `path: [String!]!` demotes to `UnclassifiedType`, which silently turns the sibling errors field into a second *data* field and the scan's `Admit` into a `Reject` ("2 data-channel-shaped fields"); test SDL must declare the full `path` + `message` contract.
* **`@splitQuery` carrier (step 3a): tolerated as redundant, not routed away.** New `BuildContext.scanStructuralServiceCarrierPayload`, the `@service`-carrier variant of the structural scan whose forbidden-directive set excludes `@splitQuery`; `TypeBuilder.promoteSingleRecordPayloads` now gates DML candidates on the strict scan and `ServiceEmitted` candidates on the tolerant one. Rationale: the carrier data field's emit (`FetcherEmitter`'s `Wrap.TableRecord` arm) already runs a PK-keyed follow-up SELECT off the producer's records, so `@splitQuery` adds nothing; the established `warnIfSplitQueryOnRecordParent` advisory ("redundant on a @record-parent field... will be ignored") now also fires from the serviceEmitted classification arm. Once promoted, everything downstream existed: the data field classifies `SingleRecordTableField` with MANY cardinality and the `OUTCOME_SUCCESS` envelope, and the emitter's `outcomeWrapped` narrowing composes orthogonally with its MANY arm. No emitter change was needed.
* **Orphan-carrier author error (step 3's "never a dropped type"), scoped to carrier shapes.** The mutation `@service` Scalar arm now rejects (`UnclassifiedField`, `FieldBuilder.orphanServiceCarrierReason`) when the return names an SDL Object type that (i) has no registry entry and (ii) the service-carrier scan `Admit`s — a recognized carrier that failed to bind/promote. The ID-element arm gets a tailored reason (DELETE-only permit, R275 follow-up pointer). Deliberately *not* covered: scan-`Reject` / `NotApplicable` orphan payloads keep the pre-R275 laxity, because seven long-standing unit fixtures (`MUTATION_SERVICE_RECORD_FIELD`, the R53/R150/R155 argMapping and input-bean rows) intentionally pair `String`-returning stubs with unbacked `FilmDetails @record` payloads to test arg-mapping mechanics in isolation; those scan as `Reject` (scalar data field). Closing that residual invalid-schema hole for non-carrier orphans needs a model-level soundness pass (and fixture rework) — a separate design conversation, also relevant to the unguarded *query*-side `@service` Scalar arm.
* **Tests.** The two red `GraphitronSchemaBuilderTest` rows re-cast errors-bearing (opptak shapes) and now green: the `@splitQuery` row additionally asserts the data field's `MANY` + `OUTCOME_SUCCESS` projection; the ID row asserts the rejection reason names the payload, ID-element, and DELETE. Pipeline: `SingleRecordTableFieldServiceProducerPipelineTest.serviceProducer_splitQueryListCarrier_withErrors_admitsAndWarnsRedundantDirective` (admit + envelope + `WrapperArm` errors + redundancy advisory) and `serviceProducer_idElementCarrier_rejectsLoudly`. Execution: new sakila fixture `serviceFilmsByIdsWithErrors` (`FilmCarrierWithErrorsService.filmsByIds` returning `List<FilmRecord>` into `FilmsCarrierWithErrorsPayload { films: [Film!] @splitQuery, errors }`) with success-arm (input-order projection, `errors: null`) and error-arm (`films: null`, typed error) tests in `GraphQLQueryTest` — the first execution coverage of the MANY + `OUTCOME_SUCCESS` combination.

### Remaining: `@nodeId`-from-record support (requirement 2)

The `fjernSakTagg`/`fjernSakTagger` shape (`{ tagger: [ID] @nodeId(typeName: "Tagg"), errors }` over a service returning the deleted record(s)) is now a *loud author error* instead of an invalid schema, but not yet supported. The implementation seam is known: encode the PK columns straight off `success.value()`'s in-memory record(s), the same seam the split-query key read uses — no DB round-trip, deletion-safe by construction. The `SingleRecordIdFieldFromReturning` / `NodeIdEncodeKeys` machinery from the DELETE carrier is the model to follow, but sourced from the `Outcome` envelope rather than DML RETURNING. When it lands, flip `SERVICE_MUTATION_ID_CARRIER_SILENTLY_DROPS_PAYLOAD` and `serviceProducer_idElementCarrier_rejectsLoudly` from rejection to classification, and add the execution fixture (a delete-shaped sakila service echoing ids).

## Carrier model and `NoBacking` removal moved to R276

The carrier unification (DML and `@service` carriers bind to their jOOQ record) and the removal of `PojoResultType.NoBacking` were originally folded in here. They moved to R276 (record-binding-reflection-only) on 2026-06-03, because they are part of making reflection-only binding *sound and complete*, which is R276's reopened scope, and R276 is the item on trunk that broke and must be un-broken. Once R276 lands, every carrier is a `ResultType` (`JooqTableRecordType`) and the source-record carrier this item targets binds the same way. This item then consumes that and owns only the error-channel and data-projection wiring below.
