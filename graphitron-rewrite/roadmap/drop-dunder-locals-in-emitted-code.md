# Retire dunder-prefixed locals in emitted generator code

Many emitters spell their generated locals with a `__` (dunder) prefix: `__fetched`, `__byPk`, `__r`, `__src`, `__key`, `__match`, `__ordered` (`FetcherEmitter`), `__elt` / `__k` (`GeneratorUtils.buildAccessorKeySingle` / `buildAccessorKeyMany`), `__t` / `__m` (named in a stale `ChannelCatchArmEmitter` javadoc), and more across `TypeFetcherGenerator`, `MultiTablePolymorphicEmitter`, `SplitRowsMethodEmitter`, and the `util` / `schema` generators (~hundreds of occurrences in ~10 files). This reads against the generated-code-readability direction the project has otherwise committed to (R260, In Review, drops the underscore style from `ArgCallEmitter`'s NodeId decode; R268, Spec, plans the `__elt` / `__k` rename as incidental cleanup; the design-principles doc calls for readable locals with explicit types and ordinary control flow).

The `__` prefix was apparently adopted as a collision-avoidance device (the `ChannelCatchArmEmitter` javadoc still claims it "binds `__t` and `__m` to avoid colliding with author-visible names"), but that justification does not hold and the device is just a lazy default. We generate every line of these method bodies, including their signatures, so every identifier in scope is one the generator chose. Author input (schema type / field / argument names) becomes *types, method names, and field references* (`$T`, `Tables.X.Y`), never a free-floating local that could shadow a generated one. So a collision is always knowable at generation time, and in the rare scope where an author-derived *parameter* name could sit beside a generated local, the generator emits that signature too and can disambiguate deterministically. The `__` prefix buys no safety that plain readable names plus generation-time awareness do not already provide. Two pieces of in-tree evidence: `ChannelCatchArmEmitter.mappedCatchArm` already emits readable names (`mapping`, `cause`, `e`) with no dunders and no collision machinery, so the class javadoc that names `__t` / `__m` is stale and describes code that no longer exists (fix it as part of this item); and `FetcherEmitter` uses `r` (readable, in a lambda) and `__r` (dunder, in a for-loop) for the same kind of value a few lines apart in one method, so the prefix is not even applied consistently.

The convention this item lands is: readable names with explicit types and ordinary control flow, matching the design-principles doc. The sweep must be *complete and self-defending*, the reason this is a Spec and not an ad-hoc PR: a naive find-and-replace is actively dangerous here, because two of the things that look like lazy dunders are not, and a future reader (or a future agent) re-running the cleanup would either "fix" them (breaking generation) or leave them looking exactly as suspect as the real offenders. The spec therefore pins the scope boundary in a *test*, not in prose that can go stale (the same `ChannelCatchArmEmitter` javadoc that started this is itself a stale comment, see below). The standing rule goes into the design-principles doc; the boundary and the no-regression invariant go into a meta-test.

## Two things that look like dunders but are not in scope (the trap)

This is the heart of the spec. A leading `__` appears in three distinct namespaces, and only one is the lazy default:

1. **Java local / lambda-parameter names (IN SCOPE).** `__r`, `__src`, `__fetched`, `__byPk`, `__ordered`, `__match`, `__key`, `__out`, `__ids` (`FetcherEmitter`); `__elt`, `__k` (`GeneratorUtils`); `__insertKey`, `__bulkKey`, `__setKey`, `__bulkSetKey`, `__violations`, `__lookupKey`, `__validator`, `__v`, `__arg` (`TypeFetcherGenerator`, several being base-name prefixes that emit `__insertKey0`, `__insertKey1`, …); `__e`, `__c` (`InputRecordGenerator` lambda params); `__violations` (`ChannelEarlyReturnEmitter`). These are Java identifiers the generator picks for its own emitted locals. No author identifier can shadow them (see lede). Rename to readable names.

2. **Synthetic SQL column aliases (OUT OF SCOPE, by a real rationale).** `__typename` (`TYPENAME_COLUMN`), `__sort__` (`SORT_COLUMN`), `__idx__` (`IDX_COLUMN`), `__rn__`, `__pk0__`..`__pkN__` (`PK_COLUMN_PREFIX`/`SUFFIX`) ; in `MultiTablePolymorphicEmitter`, `SplitRowsMethodEmitter`, `GraphitronSchemaClassGenerator`, the `util` generators, and `QueryField`. These are *SQL identifiers* projected into a result set (`DSL.inline(...).as("__sort__")`, `m.get("__typename")`), declared as named constants, and they live in the same column namespace as real table columns, which **are** author-controlled (the consumer's DB schema). `__sort__` exists precisely so it cannot collide with a real column named `sort`. This is the genuine collision-avoidance case the Java locals only pretended to be; the `__name__` (double-underscore-wrapped) convention is deliberate and correct. They are **not** renamed. The spec's job for these is to make their intent legible: confirm each is a named constant with a javadoc that states the collision rationale, so they no longer read as "more of the same laziness".

3. **External tokens we do not own (OUT OF SCOPE).** `__NODE_TYPE_ID` / `__NODE_KEY_COLUMNS` are read by reflection off jOOQ-generated consumer classes (we consume them, we do not emit them); `federation__FieldSet`, `link__Import`, etc. are Apollo-Federation SDL scalar names fixed by the federation spec. Renaming either would break interop. Untouched.

The discriminator that separates (1) from (2)/(3) cleanly, and the one the meta-test keys on, is **Java identifier vs string literal in the emitted output**: a lazy dunder surfaces in generated code as a bare identifier (`__elt.into(...)`), a synthetic column as a string literal (`.as("__sort__")`, `get("__typename")`) or a reflective field name. `__typename` breaks the "wrapped in `__`" heuristic (it is leading-only), so the heuristic is wrong; identifier-vs-literal is the rule.

## Inventory (grounding; regenerate before implementing)

In-scope Java-local sites, by emitter (counts approximate, from the Spec-time sweep): `FetcherEmitter` (~9 names), `TypeFetcherGenerator` (~9), `GeneratorUtils` (2), `InputRecordGenerator` (2), `ChannelEarlyReturnEmitter` (1). The `ChannelCatchArmEmitter` `__t` / `__m` appear *only in the class javadoc*, not in emitted code (`mappedCatchArm` already emits `mapping` / `cause`) ; the implementer must re-run the sweep at the head of implementation, because R260 (In Review) and R268 (Spec) may have already converted some sites by then.

## Implementation seams

1. **Per-emitter rename of the in-scope locals** to readable names (`__elt` → `element`/`elt`, `__byPk` → `byPk`, `__fetched` → `fetched`, base-prefixes `__insertKey` → `insertKey`, etc.). Behaviour-preserving; the generated programs differ only in local identifiers.
2. **Delete the stale `ChannelCatchArmEmitter` javadoc sentence** naming `__t` / `__m`; replace with a one-line note that the cause-walk binds `cause` and the mapping `mapping` (matching the code).
3. **Document the synthetic-column convention** at each constant (`SORT_COLUMN`, `IDX_COLUMN`, `TYPENAME_COLUMN`, `PK_COLUMN_PREFIX`/`SUFFIX`) with the collision rationale, so the carve-out is self-explaining at the site. **`__rn__` is not yet a named constant** ; it is a bare repeated string literal in `MultiTablePolymorphicEmitter` (and the row-number alias in `SplitRowsMethodEmitter`). Promote it to a named constant (`RN_COLUMN` or similar) *before* documenting it, so seam-3's "at each constant" is true rather than aspirational. (`QueryField` and the `util` generators *reference* these synthetic columns but do not own the constants; the constants live on `MultiTablePolymorphicEmitter` and `SplitRowsMethodEmitter`.) This per-constant rationale is **deliberately descriptive prose, not test-pinned**: it is a conscious exception to seam-4's standing rule, because "this alias avoids colliding with a real DB column" is genuine prose, not a mechanizable invariant. The meta-test (the no-regression guard below) pins the *absence of dunder Java identifiers*, not the alias rationale; naming this exception keeps the principle honest rather than pretending the prose is enforced.
4. **Add the standing rule to `rewrite-design-principles.adoc`**: emitted Java locals/params use readable names; the `__`-prefix is reserved for synthetic SQL column aliases (with the DB-column-collision rationale) and never for Java identifiers; external tokens (`__NODE_*`, `federation__*`, `link__*`) are not ours.

## The no-regression guard (the user's core ask: do not get tricked again)

Add a meta-test that *enforces the boundary mechanically*, so neither a reintroduced lazy dunder nor a future stale comment can mislead. Model it on `TierAnnotationEnforcementTest` (walk + collect-violations shape). The test asserts that no Java *identifier* (local, parameter, field) in emitted output begins with `__`. Three points the implementer must get right, all surfaced by principles review:

- **Strip set: comments as well as literals.** Before the identifier scan, strip not just string and char literals but *also line comments, block comments, and javadoc*. The synthetic-column aliases survive only because they reach generated code as string literals (`.as("__sort__")`, `get("__typename")`) or reflective string arguments (`__NODE_*`); but emitters also add inline `// ...` comments into generated bodies (e.g. `FetcherEmitter`'s R141 PK-keyed-map note), and such a comment naming `__idx__` would false-positive a strings-only scan. The robust form is to parse each generated file and walk only identifier nodes (`SimpleName`), rather than regex over raw text; if a regex is used, the strip set must include comments explicitly. A reintroduced `__foo` *local* is a bare identifier and fails the test with file and line.
- **Do not let it pass vacuously.** A scan over `graphitron-sakila-example`'s `target/generated-sources/**.java` passes trivially if generation did not run or the catalog jar was clobbered (the documented `-Plocal-db` footgun empties the catalog and cascade-fails pipeline tests). Guard against the no-op: either assert a non-trivial file floor (scanned ≥ N generated files), or make the **primary** variant a `@PipelineTier` scan that generates in-process over a representative schema and walks the resulting `TypeSpec` identifiers, which cannot be satisfied by an empty tree. Recommendation: pipeline-tier in-process scan as the load-bearing guard, with the generated-sources walk as a redundant compile-tier backstop only if it carries the file-floor assertion.
- **This is a blanket structural lint, not a per-body string assertion.** It asserts a single property (no `__`-led identifier) across all output, never the content of a specific fetcher body, so it does not run afoul of the banned-body-assertion rule. It pins the invariant in a live test rather than a comment ; directly the design-principles "documentation names only live tests/code" rule, the exact failure mode that produced the stale `__t`/`__m` javadoc.

## Tests

- **The meta-test above** is the new standing guard and the deliverable's centrepiece.
- **Compilation (`@CompilationTier`)**: `mvn install -Plocal-db` green over `graphitron-sakila-example`; a behaviour-preserving local rename is caught here if any reference is missed.
- **Execution (`@ExecutionTier`)**: the existing suite round-trips unchanged; no new execution fixture, the rename changes no behaviour.
- **No per-emitter generated-body string assertions** (banned). Correctness is "still compiles, still round-trips"; the *naming convention* is enforced by the blanket meta-test, not by grepping individual fetcher bodies.

## Design fork for In Progress (flag to principles review)

The lede claims no in-scope local can be shadowed by an author-derived identifier. The implementation must *confirm* this rather than assume it: audit whether any emitter inlines a generated local into a method body that also carries a parameter named from a GraphQL argument (`@service`/resolver signatures are the only plausible source). If none do (expected, since the dunder locals live in DataFetcher/batch-loader/helper methods with generator-fixed signatures), the rename is unconditionally safe and **no disambiguation machinery is built** ; pick readable names and stop. If some do, the convention's escape hatch is a deterministic suffix computed against the known-in-scope names (we emit the signature, so the set is known), never a blanket `__` prefix. Resolve this audit at the top of In Progress and record the finding; do not pre-build collision machinery the audit may show is dead weight.

## Out of scope

- **Synthetic SQL column aliases** (`__typename`, `__sort__`, `__idx__`, `__rn__`, `__pkN__`): deliberate, real collision surface; documented, not renamed.
- **External tokens** (`__NODE_TYPE_ID` / `__NODE_KEY_COLUMNS`, `federation__*`, `link__*`): not emitted by us / fixed by external specs.
- **R260 and R268's point renames**: already in flight on their own emitters; this item converts the remainder and adds the guard. If they land first, the sweep is smaller; the guard subsumes them either way.
