# Typed jOOQ class references for multi-schema correctness (TableRef, ForeignKeyRef)

## Success criteria

End-state contract a Spec → Ready reviewer should hold the implementation against:

1. **No emit-side string concatenation against `jooqPackage`** for table, table-record, `Tables`, or `Keys` references. The two infrastructure sites (`JooqCatalog` constructor's `<jooqPackage>.DefaultCatalog` load and `CatalogBuilder`'s filesystem-path conversion) are the only survivors.
2. **`TableRef` carries `tableClass`, `recordClass`, `constantsClass` as `ClassName`s** populated at parse time from the resolved jOOQ `Table<?>`. The `String javaClassName` field is gone.
3. **`ForeignKeyRef` replaces `String fkJavaConstant` on `JoinStep.FkJoin`**, with the FK-holder schema's `Keys` class as a typed `ClassName`.
4. **Catalog-miss is structural, not sentinel-based.** `BuildContext.resolveTable` returns `Optional<TableRef>`, `synthesizeFkJoin` returns `Optional<FkJoin>`, and consumers route absence through `UnclassifiedType` / `UnclassifiedField`.
5. **Resolution-time disambiguation is strict.** Unqualified `@table(name: "x")` resolves iff exactly one schema contains `x`; collisions reject with the structured ambiguity message. Qualified `@table(name: "schema.x")` resolves to the named schema. The eight directive-parsing sites and the catalog's internal `tableSqlName`-chained APIs all participate.
6. **Single-schema regression equivalence is verified.** Compilation tier recompiles `graphitron-sakila-example` and asserts byte-identical sources for committed example code.
7. **The multi-schema fixture exercises every emit path *and* every resolution path R78 changes** (see Phase 1 enumeration). Compilation tier red before R78, green after.

## Bug surfaced

Generated code does not compile against multi-schema jOOQ codegen. Imports are emitted as `<jooqPackage>.tables.X` and `<jooqPackage>.tables.records.XRecord`, with `Tables.X` and `Keys.<FK>` referenced from `<jooqPackage>.Tables` / `<jooqPackage>.Keys`. In a multi-schema layout jOOQ generates one package per schema, so the correct fully-qualified names carry a schema segment (e.g. `<jooqPackage>.<schemaName>.tables.X`). Single-schema setups using the PostgreSQL `public` default, including all in-tree fixtures, happen to flatten and mask the bug; consumer projects with multiple schemas cannot compile the generated output.

A second, latent bug shares the same root: `JooqCatalog.findTable(sqlName)` resolves an `@table(name: "x")` directive value by `catalog.schemaStream().flatMap(...).findFirst()` and silently picks whichever schema iterates first when two schemas share a table name. Single-schema setups never hit this; multi-schema setups with collisions have been picking the wrong table at random, with no error and no warning. R78 fixes both axes (FQN emission *and* resolution-time disambiguation) because the two share a fixture, a catalog rewrite, and a phasing.

## Root cause

The rewrite threads a single `String jooqPackage` through every emitter and concatenates `".tables"`, `".tables.records"`, `"Tables"`, `"Keys"` literals against it at every reference site. The schema-per-table fact known to the catalog is dropped on the floor; emitters reconstruct a guessed FQN that only matches single-schema layouts.

## Proposed direction

Replace string concatenation against `jooqPackage` with javapoet `ClassName` values populated once at parse time from `Table<?>` reflection. Each catalog reflection result already knows its own schema; the model captures it once at parse time, emitters consume the typed value with no derivation.

- `TableRef` carries — alongside the existing `tableName: String` (SQL identity, kept as the catalog lookup key) and `primaryKeyColumns` — three javapoet `ClassName` fields populated by reflection: `tableClass` (the `<schemaPackage>.tables.<X>` class, e.g. `Film`), `recordClass` (`<schemaPackage>.tables.records.<X>Record`), and `constantsClass` (the schema's `Tables` constants class). Plus the `Tables`-side field name as `String` (e.g. `"FILM"`). The simple-class `String javaClassName` field goes away — it was only ever used to rebuild a `ClassName` at every read site. The third field is named `constantsClass` rather than `tablesClass` deliberately: `tableClass` and `tablesClass` would differ by a single `s` and code review of 50+ migrated sites would not catch a swap. `constantsClass` describes what the `Tables` class actually is (a holder for table constants) and reads distinctly next to its siblings.
- New `ForeignKeyRef` for FK references: the `Keys` class as `ClassName` plus the FK constant name as `String`. Replaces the bare `String fkJavaConstant` on `JoinStep.FkJoin` and the `keysClass` lookup at every emit site. The `Keys` class belongs to the FK-holder table's schema (the side carrying the constraint, which jOOQ exposes as `ForeignKey.getTable().getSchema()`), so each `ForeignKeyRef` carries its own host class and cross-schema FKs join correctly without any per-emitter schema arithmetic. Note: "FK-holder" here is the SQL-level constraint-owner, distinct from `JoinStep.FkJoin.originTable` which is the *traversal-origin* (the side the join enters from at runtime, which can be either side of the FK depending on direction). The two are equal for child-to-parent traversal and unequal for parent-to-child; the `Keys` class is determined by the SQL fact (FK-holder), not the traversal direction. Named `ForeignKeyRef` (not `KeyRef`) because in jOOQ "Key" covers PKs and FKs both, and the host class is literally `Keys` containing both kinds; PKs already live on `TableRef.primaryKeyColumns` and have no `Ref` cousin, so the asymmetry is real and the name should reflect the FK-only scope.
- `JooqCatalog.TableEntry` becomes the single typed source. Its API: `tableClass(): ClassName`, `recordClass(): ClassName` (from `Table.getRecordType()`), `constantsClass(): ClassName`, `pkColumnRefs(): List<ColumnRef>`, and `toTableRef(sqlName): TableRef` as the one factory. Both `BuildContext.resolveTable` and `ServiceCatalog.buildTableRef` collapse to `tableEntry.toTableRef(sqlName)` — the duplicated PK-column materialisation in those two methods is consolidated as a side-effect of this change. Note the collapse is asymmetric: `ServiceCatalog.resolveTable` already returns `Optional<TableRef>`; `BuildContext.resolveTable` is the side that returns the empty-shape sentinel and migrates to match. The `sqlName` parameter on `toTableRef` is the directive-supplied form (case-preserved for error messages), not `entry.table().getName()` — the catalog lookup key is preserved from the user-visible name even when jOOQ canonicalises differently.
- The reflection plumbing this design needs already lives in `JooqCatalog` as the private `tablesClass(Schema)` (today's line 485) and `keysClass(Schema)` (line 247) helpers, both of which do `Class.forName(schema.getClass().getPackageName() + ".Tables" | ".Keys")` — the schema-correct derivation. The migration is exposing what's already there, not inventing reflection. `TableEntry.constantsClass()` calls into `tablesClass(table.getSchema())`; `findForeignKeyByName(...)` calls into `keysClass(fk.getTable().getSchema())`. The `String javaFieldName` (today populated from `table.getClass().getFields()` iteration at line 465) likewise stays computed where it already is. Net: the load-bearing reflection is in place; this change is mostly a model-shape and call-site rewrite, not a reflection-engineering effort.

  Existence proof: enum-column FQNs already follow this pattern. `JooqCatalog.findColumn` (line 452) populates `ColumnEntry.columnClass()` from `col.getType().getName()`, which jOOQ reflection produces schema-qualified for both single- and multi-schema codegen. `EnumMappingResolver.validateEnumFilter` (line 136) consumes that string via `Class.forName(...)`. Enums are unaffected by the multi-schema bug because the catalog already uses reflection rather than `<jooqPackage>` concatenation for them. R78 generalises the same approach to tables, records, constants, and keys.
- **Schema-package edge case.** `tablesClass(Schema)` returns `Optional` because `Class.forName(... + ".Tables")` may throw `ClassNotFoundException` — i.e. the catalog jar contains the schema but no generated `Tables` constants class. This is a degenerate consumer-side codegen state (jOOQ producing partial output). Today the empty-string fallback hides it. After R78 it must surface: `TableEntry.toTableRef(sqlName)` returns `Optional.empty()` when `tablesClass` is empty, and the caller routes to `UnclassifiedType`. Same shape for `keysClass` empty when constructing `ForeignKeyRef` (the FK's containing field becomes `UnclassifiedField`). Both cases warrant a one-line warning at catalog-load time so consumers can diagnose their codegen.
- `JooqCatalog.findForeignKeyByName(sqlConstraintName): Optional<ForeignKeyRef>` replaces today's `fkJavaConstantName(): Optional<String>`. The new shape returns the typed ref directly so the empty-string `.orElse("")` fallback at `BuildContext.synthesizeFkJoin` becomes impossible by construction; on miss, the containing field is rejected upstream (see "Catalog-miss handling"). The method is `findForeignKeyByName` rather than `findKey` for the same disambiguation reason that motivated naming the ref `ForeignKeyRef`: in jOOQ "Key" covers PKs and FKs, so `findKey` would suggest a PK lookup. The companion `JooqCatalog.findForeignKey(sourceTable, targetTable)` (existing) returns the raw `ForeignKey<?,?>` for column-set lookup; the new `findForeignKeyByName` lifts a SQL constraint name to the typed ref. The two names sit cleanly side-by-side.
- `ClassName` directly on the model records, not behind a domain wrapper. Follows the precedent of `BatchKey`, `MethodRef`, `HelperRef`, `LifterRef`, `AccessorRef`, `ResultAssembly`, `PayloadAssembly`, `RowsMethodShape`, `ErrorChannel`, `DefaultedSlot`, `ChildField`. The `Tables` / `Keys` simple-name literals are jOOQ codegen conventions kept inside `JooqCatalog`, not stored on every ref.
- No separate `TableRecordRef`. The record class is the same table viewed from the projection axis; `Table.getRecordType()` produces it from the same `Table<?>` the `tableClass` came from. Per the design-principles rule "each new sub-taxonomy proposal comes with a one-line note on what distinct information it carries that a sibling cannot — otherwise it's probably a field on an existing record," `recordClass: ClassName` is a field on `TableRef`. Every consumer that needs it already has the `TableRef` in hand.
- **Schema-qualified `@table(name:)` directive value** for multi-schema disambiguation. The directive parser accepts both `@table(name: "film")` (unqualified) and `@table(name: "public.film")` (qualified). Unqualified values continue to resolve as today *iff* exactly one schema in the catalog contains a table by that name; when two or more do, classification fails with `UnclassifiedType` carrying a structured error: `"@table(name: 'film') is ambiguous: defined in schemas [public, archive]; qualify as 'public.film' or 'archive.film'"`. Qualified values resolve directly to the named schema's table; unknown schema or unknown table within the schema also routes to `UnclassifiedType`. The split happens on the first `.` (PostgreSQL identifiers can't contain unquoted dots; quoted-identifier support `"my.schema"."weird.table"` is out of scope and would require a follow-up). The directive's SDL signature does not change — only the parser semantics for the existing `name:` argument. Identifier matching is case-insensitive on both halves (consistent with `JooqCatalog.findColumn` at today's line 472).
- **Catalog API takes `TableRef` everywhere internally.** The R78 ambiguity fix in `findTable` is necessary but not sufficient: every catalog method that today takes `String tableSqlName` and chains through `findTable(tableSqlName)` (`findColumn`, `findForeignKey`, `findIndexColumns`, `findPkColumns`, `qualifierForFk`, `findForeignKeysBetweenTables`, etc.) inherits the same ambiguity in multi-schema setups. The fix is to add a `TableRef`-taking overload for each: callers that already hold a resolved `TableRef` (the common case — see the ~17 `tableName()` consumer sites in `BatchKeyLifterDirectiveResolver`, `ServiceCatalog`, `NodeIdLeafResolver`, `GraphitronSchemaValidator`) switch to the typed overload, which routes the lookup through the resolved `Schema` directly and never re-parses. The `String` overloads remain for the directive-parsing call sites that don't yet have a `TableRef`. Net: the `tableName()` accessor on `TableRef` keeps its un-qualified `"film"` shape (used as a display name for error messages and by the `String` overloads at parser sites); the `Schema`-bound resolution lives entirely inside `JooqCatalog` behind the typed overloads. No new `schemaName` field on `TableRef` — the resolved-schema identity sits where the resolution lives.

## Catalog-miss handling

Today `BuildContext.resolveTable` returns `new TableRef(sqlName, "", "", List.of())` and `synthesizeFkJoin` returns `fkJavaConstant = ""` when the catalog has no entry. Empty-string sentinels survive `String javaClassName` but `ClassName` has no good empty value. The fix is structural, not a new sentinel: catalog-miss classifies the containing type as `UnclassifiedType` (and the field as `UnclassifiedField`) before any `TableRef` / `ForeignKeyRef` is constructed. Emit-site code never sees a partially-resolved ref.

Concretely, `BuildContext.resolveTable` becomes `Optional<TableRef>`, and the call sites that today consume the empty-shape fallback are reachable only on the resolved arm. Same shape for `synthesizeFkJoin`'s FK-constant fallback. (The `Resolved.{Ok | Rejected}` sealed shape exists in `BatchKeyLifterDirectiveResolver` and a parallel `ArgConditionResult.{Ok | Rejected}` in `ConditionResolver`, but those are local-to-resolver contracts that carry rich rejection metadata; `resolveTable` returns either a found `TableEntry` or "not in catalog," with no nuance to carry. `Optional<TableRef>` is the right shape — don't import a heavyweight envelope where a `null/present` axis suffices.) This expands R78's surface area but is the only design that survives Java's type system: the alternatives (`@Nullable ClassName` propagated through 100+ call sites, or `Optional<TableRef>` propagated through the model itself) lose more than they save.

Optional propagation touches a small enumerable set inside `BuildContext`. The four `synthesizeFkJoin` call sites — implicit-FK inference (`parsePath`, today line 500), the `{table:}` and `{key:}` arms of `parsePathElement` (today lines ~626 and ~648), and the NodeId synthesis shim (today line 1086) — all pre-resolve the FK via `catalog.findForeignKey(...)` or `findForeignKeysBetweenTables(...)` and so already operate on the resolved arm. The shim site additionally calls `resolveTable(targetTableOpt.get())` on a name pulled from a present-`Optional`; that call lifts to `Optional<TableRef>` and propagates upward to the existing `Unresolved` return when absent, with no new control flow. `synthesizeFkJoin` itself becomes `Optional<FkJoin>`: when `originTable` or `targetTable` cannot resolve (catalog-miss on a FK whose endpoint is not in the catalog jar), the helper returns empty and the surrounding builder routes to `UnclassifiedField` along the same path that handles `findForeignKeyByName` misses.

A fifth `resolveTable` call site sits at `buildInputNodeIdReference` (today line 938), reached only after the surrounding `@nodeId(typeName:)` branch has confirmed that `refTypeName` corresponds to a `NodeType`. NodeType existence implies a resolved `TableRef` by construction (catalog-miss → `UnclassifiedType`, never `NodeType`), so this call is total. The migration leaves it as a direct `tableEntry.toTableRef(...)` rather than threading `Optional` through; the safety argument is the type system already enforced upstream.

## Resolution-time disambiguation

Today's `JooqCatalog.findTable(sqlName)` does `catalog.schemaStream().flatMap(s -> s.getTables().stream()).filter(t -> sqlName.equalsIgnoreCase(t.getName())).findFirst()` and picks whichever schema iterates first. The iteration order is jOOQ's, not anything stable a consumer can rely on. R78 replaces this with a strict resolver:

1. **Parse the directive value.** A new `JooqCatalog.parseQualifiedTableName(String)` helper splits on the first `.`. Returns `(Optional<String> schema, String table)` — schema is empty for unqualified values. Both halves must be non-empty when a `.` is present (`"film."` or `".film"` are malformed; classifier rejects with a clear message). Used at the three `BuildContext` sites that read `argString(obj, DIR_TABLE, ARG_NAME)` (today's lines 407, 906, 1262).

2. **Resolve against the catalog.** Two `findTable` shapes:
   - `findTable(qualifiedName: String): Optional<TableEntry>` — single-arg form, accepts both `"film"` and `"public.film"`. For unqualified values, scans all schemas and applies the ambiguity policy below. For qualified values, scopes the lookup to the named schema and returns empty when either the schema or the table-within-schema is missing.
   - `findTable(schemaSqlName: String, tableSqlName: String): Optional<TableEntry>` — two-arg form for callers that have already split. The single-arg form delegates here once parsing is done.

3. **Ambiguity policy.** When an unqualified name matches tables in multiple schemas, the catalog returns empty *and* records a structured ambiguity rejection — the classifier reads it and routes to `UnclassifiedType` with the `[schema-list]; qualify as 'schema.name'` message. The rejection is structured (not a free-form string) so future tooling (LSP quick-fix, SDL formatter) can suggest the fix automatically. Single-schema setups never hit this path; multi-schema setups with non-colliding names never hit it either. Only collisions force qualification.

This three-step shape is the same pattern the catalog uses for FK resolution (parse → look up → reject-with-context); putting `@table(name:)` resolution on the same rails is the design coherence argument.

The resolution rules apply uniformly to every `@table(name:)` consumer site: type classification, `@nodeId(typeName:)` target tables, `@reference(table:)`, the type-uniqueness map. None of these get a special exception. Single-schema and unique-name multi-schema setups remain untouched; only collision sites are forced to qualify.

## Single-schema regression equivalence

Single-schema consumers (PostgreSQL `public`, the in-tree fixtures' baseline) must continue to compile unchanged after R78. They do, because the reflection path is schema-correct in both modes:

- Single-schema codegen produces `<root>.tables.X` for tables and `<root>.Tables` for the constants class. Reading `table.getClass().getPackageName()` returns `<root>.tables`, and `tablesClass(schema)` returns `Class.forName("<root>.Tables")`. After `ClassName.get(...)` the emitted FQN is `<root>.tables.X` and `<root>.Tables` — identical to today's string-concatenation output.
- Multi-schema codegen produces `<root>.<schemaName>.tables.X` and `<root>.<schemaName>.Tables`. The same reflection paths return the schema-segmented FQNs. Today's string-concatenation emits `<root>.tables.X` (wrong); after R78 it emits the schema-correct form.

The change is strictly net-positive: single-schema output is byte-equivalent before and after; multi-schema output goes from broken to correct. No consumer of in-tree fixtures (`graphitron-sakila-example` etc.) needs source updates. The Phase 3 compilation tier verifies this by recompiling sakila against the new generator output and asserting zero source diffs in committed example code.

## `jooqPackage` survivors

After the migration, `jooqPackage` is gone from every emitter that takes a `TableRef` / `ForeignKeyRef`. Two production sites legitimately keep it as the schema-package root for codegen helpers that scan or produce files in that namespace:

- `JooqCatalog` constructor — loads `<jooqPackage>.DefaultCatalog` reflectively. Unchanged.
- `CatalogBuilder` — converts to a filesystem path (`jooqPackage.replace('.', '/')`) for catalog-jar build wiring. Unchanged.

`NodeIdEncoderClassGenerator` is *not* a survivor. The non-empty arm builds a single `ClassName.get(jooqPackage, "Tables")` and reuses it across every `NodeType`'s decode method (today's lines 156-163, 191-229), which is multi-schema-broken whenever node types live in different schemas. Each `NodeType` already carries a `TableRef`; the migration reads `nt.table().constantsClass()` per node type instead of synthesising one from `jooqPackage`. The `jooqPackage` parameter drops from the signature entirely; the `tablesClass` derivation moves into the per-NodeType loop, where it executes zero times when `nodeTypes` is empty (the per-type for-loop is the only `tablesClass` consumer). Today's `jooqPackage == null` arm at line 156 (which suppresses decode-method emission in no-catalog mode) likewise disappears: catalog-miss → `UnclassifiedType`, so a `NodeType` cannot exist without a resolved `TableRef`, and the no-catalog mode either produces zero `NodeType` instances (the loop body simply doesn't run) or produces real ones with usable `constantsClass()`. The null-arm signal collapses into the empty-list signal already present.

`QueryNodeFetcherClassGenerator.generate` accepts `String jooqPackage` but never reads it (today's signature is dead-parameter). The migration drops the parameter as a trivial cleanup.

`EntityFetcherDispatchClassGenerator` takes `jooqPackage` today and threads it through `SelectMethodBody`, which still concatenates `jooqPackage + ".tables"` against `entity.table().javaClassName()` (today line 81). The migration switches `SelectMethodBody` to read `entity.table().tableClass()` directly; the `jooqPackage` parameter then drops out of `EntityFetcherDispatchClassGenerator`'s public surface in the same change. The inner `HandleMethodEmitter` carries a `jooqPackage` field (today's line 247) that is held but never read — dead weight that drops alongside the parameter. No emitter survives needing `jooqPackage` other than the two infrastructure sites listed above.

The success criterion is "no emitter constructs `<jooqPackage>.tables.X`, `<jooqPackage>.tables.records.X`, `<jooqPackage>.Tables`, or `<jooqPackage>.Keys` by string concatenation." Bare `jooqPackage` for the two infrastructure purposes above (filesystem path; catalog loading) is fine. A `jooqPackage` audit (~250 references in `graphitron/src/main`) finds no other patterns: `<jooqPackage>.enums.X` strings exist in docstrings as illustrative examples, but the actual enum-class FQNs flow through reflection (`col.getType().getName()`) and are already schema-correct. The four concatenation patterns above are exhaustive.

## Phasing

Three independently buildable phases. The fixture lands first so the compilation tier turns red before any migration code is written; each subsequent phase shrinks the failure surface against a known-failing baseline. This matches "validator mirrors classifier invariants" from the design principles: every classifier decision that implies a generator branch fails at validate time if the branch is unimplemented, and the same shape applies here for the emitter branch.

1. **Multi-schema fixture + regression test.** Two-schema jOOQ-codegen output added to `graphitron-fixtures-codegen` so pipeline and compilation tiers can reuse it. Compilation tier asserts the generated output compiles against the multi-schema layout. After this phase the compilation-tier check on the multi-schema fixture is red, with a failure message that names the missing schema segment in the FQN. Phases 2-3 each turn a slice green.

   Trunk CI must stay green between Phase 1 and Phase 3 (per the trunk-based dev rule). Two alternatives, the implementation picks one and calls it out in the Phase 1 commit message:
   - **Bundle Phase 1 + Phase 2** as a single first commit so the fixture only lands once at least the dominant `TableRef` migration is in place. Preferred — the rewrite codebase has no existing `@Disabled` / `@DisabledUntil` precedent, so the marker approach would invent a pattern.
   - **Disable-marker the fixture's compilation assertion** behind a JUnit 5 `@Disabled("R78 phase 3 — see roadmap")` (or equivalent) so the failure is documented but does not red-line trunk; the marker is removed in the Phase 3 commit. Acceptable but introduces a transient marker the codebase doesn't otherwise use.

   The fixture must exercise — at minimum — every emit path *and* every resolution path R78 changes, otherwise it can pass without exercising the failure modes:
   - **Cross-schema table reference** (table-class FQN): Schema B referencing a table in Schema A as a `@table` return.
   - **Cross-schema record reference** (record-class FQN): a `@service` field returning a `Record` from a different schema's table.
   - **Cross-schema FK reference** (`Keys` class): a `@reference(path: [{key: ...}])` traversing schemas.
   - **`Tables.<FIELD>` constant reference** (`constantsClass`): exercised implicitly by every `@table`, but the fixture should include a `@nodeId` decode arm so the per-NodeType `Tables` reference inside `NodeIdEncoderClassGenerator` is exercised.
   - **Schema-qualified `@table(name:)`**: at least one type uses `@table(name: "schema.table")` to exercise the qualified resolution path.
   - **Unqualified-and-unique `@table(name:)`**: at least one type uses `@table(name: "x")` where `x` exists in only one schema, exercising the unqualified-but-unambiguous fast path.
   - **Shared SQL table name across the two schemas + qualification**: at least one table name (e.g. `event`) is defined in both schemas, with one type qualifying as `"public.event"` and another as `"archive.event"`. A pipeline-tier test additionally asserts that an unqualified `@table(name: "event")` against this fixture is rejected with the structured ambiguity message (this is the resolution-time correctness regression test).
2. **Catalog-side typed `TableEntry` API + TableRef migration + qualified-name resolver.** Replace `javaClassName: String` with `tableClass: ClassName`, `recordClass: ClassName`, `constantsClass: ClassName` on `TableRef`. `JooqCatalog.TableEntry` exposes the typed values; `BuildContext.resolveTable` and `ServiceCatalog.buildTableRef` collapse to a single factory (`BuildContext` aligns to `ServiceCatalog`'s already-correct `Optional<TableRef>` shape — the collapse is one-sided alignment, not a merge of two equally-broken halves). Catalog-miss returns `Optional<TableRef>` and consumers handle absence via `UnclassifiedType` / `UnclassifiedField` rather than empty sentinels. Drops `+ ".tables"` and `ClassName.get(jooqPackage, "Tables")` from every emit site. The dominant consumer is `TypeFetcherGenerator` (~18 `ResolvedTableNames.{of,ofTable}` sites, plus internal helpers that thread the resolved-names record); secondary sites include `InlineTableFieldEmitter`, `InlineLookupTableFieldEmitter`, `MultiTablePolymorphicEmitter`, `SplitRowsMethodEmitter`, `TypeClassGenerator`, `TypeConditionsGenerator`, `QueryConditionsGenerator`, `FetcherRegistrationsEmitter`, `SelectMethodBody`, `FetcherEmitter`, `NodeIdEncoderClassGenerator`'s per-NodeType decode arm, `TableMethodDirectiveResolver`, `ExternalFieldDirectiveResolver`. ~50 main sites; the existing `TestFixtures` helper (already centralising `new TableRef(...)` for test code) is extended to expose `tableClass`/`recordClass`/`constantsClass` so the 107 test sites flip uniformly without each test re-touching.

   `GeneratorUtils.ResolvedTableNames` simplifies as a side-effect: of its three fields, two (`tablesClass`, `jooqTableClass`) become direct getters on `TableRef` (`constantsClass()`, `tableClass()`), leaving only `typeClass` (a function of `outputPackage` + `returnTypeName`, not catalog-derived). Either degenerate the record to a single `typeClass` field plus a static `tableLocalName(TableRef)` helper, or drop the record outright and inline both call shapes. The decision belongs in the implementation; the spec records that the record's reason-for-existence shrinks to one field. Same logic strips `String jooqPackage` from the private GeneratorUtils helpers (`buildRowKey`, `buildAccessorKeySingle`, `buildAccessorKeyMany`, `buildKeyExtraction`, `buildKeyExtractionWithNullCheck`) — they all use it only to derive `tablesClass` and become parameter-pure once the field is on `TableRef`.

   This phase also ships the qualified-name resolver: `JooqCatalog.parseQualifiedTableName(String)` plus the two-arg `findTable(schemaSqlName, tableSqlName)` and the rewritten single-arg `findTable(qualifiedName)`. The three `BuildContext` directive-value sites (today's lines 407, 906, 1262, all reading `argString(obj, DIR_TABLE, ARG_NAME)`) switch to the parser. Folding it into Phase 2 rather than its own phase costs ~5 extra sites against the same `JooqCatalog` rewrite — phasing it separately would mean two passes over `JooqCatalog`'s public API, and the Phase 1 fixture already includes the ambiguity test.
3. **`ForeignKeyRef` on `FkJoin` + record-class typed refs.** Replaces `String fkJavaConstant` with `ForeignKeyRef` on `JoinStep.FkJoin`; `JooqCatalog.findForeignKeyByName(sqlConstraintName): Optional<ForeignKeyRef>` replaces `fkJavaConstantName(): Optional<String>`. Drops `keysClass = ClassName.get(jooqPackage, "Keys")` from `InlineTableFieldEmitter`, `InlineLookupTableFieldEmitter`, `SplitRowsMethodEmitter`. Same commit drops `+ ".tables.records"` concatenation: `TypeFetcherGenerator` (lines 1050, 1118), `ServiceDirectiveResolver` (line 267), and the strict-return computation in `computeExpectedServiceReturnType` switch to `tableRef.recordClass()`. Note the boundary with Phase 2: the `recordClass: ClassName` field is *added* to `TableRef` in Phase 2 (alongside `tableClass` and `constantsClass`) but its consumers are *flipped* in Phase 3 — Phase 2 leaves the record-class concatenation sites untouched because they pair naturally with the FK-ref migration here, and splitting them across phases would mean two passes over `TypeFetcherGenerator`. ~8–10 main sites combined; cross-schema FK references now compile correctly because each `ForeignKeyRef` carries its own host `Keys` class, and the last `<jooqPackage>.tables.records` site goes away. Compilation tier turns fully green.

   These were two phases in earlier drafts. They're merged here because each is small (3–5 sites), neither blocks the other, and a phase boundary is a commit-and-CI cost not justified by the size of either slice on its own.

Phase 1 establishes the forcing function; Phase 2 is the load-bearing change; Phase 3 is mechanical and ships in one commit.

## Touchpoints

Approximate scope from a `jooqPackage` audit (251 references in `graphitron/src/main`):

- `model/`: `TableRef` (rename + new typed fields), `JoinStep.FkJoin` (`ForeignKeyRef` replaces `String fkJavaConstant`).
- Catalog boundary: `JooqCatalog.TableEntry` (new typed API + `toTableRef` factory), `JooqCatalog.findForeignKeyByName` (replaces `fkJavaConstantName`; sits next to existing `findForeignKey(source, target)`), `JooqCatalog.parseQualifiedTableName` (new) plus the rewritten single-arg `findTable(qualifiedName)` and new two-arg `findTable(schemaSqlName, tableSqlName)`, `ServiceCatalog.buildTableRef`, `BuildContext.resolveTable` and FK-resolution helpers (`synthesizeFkJoin`).
- Directive parsing (eight sites total): three in `BuildContext` (today's lines 407, 906, 1262), four in `TypeBuilder` (lines 358, 551, 655, 772 — `@table` on object types, interfaces, input types, and `@nodeId` target lookups), and one in `NodeIdLeafResolver` (line 225). All read `argString(obj, DIR_TABLE, ARG_NAME)` and switch to the qualified-name parser. No directive-SDL changes; only the parsed value's interpretation.
- Catalog internal callers (the ~17 `targetTable.tableName()` / `parentTable.tableName()` / `containingTable.tableName()` sites across `BatchKeyLifterDirectiveResolver`, `ServiceCatalog`, `NodeIdLeafResolver`, `GraphitronSchemaValidator`) switch to TableRef-taking overloads of `findColumn`, `findForeignKey*`, `findPkColumns`, etc. Those that build error messages keep `tableName()` as the display string.
- Emitters with `<jooqPackage>.tables`, `<jooqPackage>.tables.records`, `<jooqPackage>.Tables`, `<jooqPackage>.Keys` concatenations: `GeneratorUtils` (the `ResolvedTableNames` factory and ~5 private helpers), `FetcherEmitter`, `InlineTableFieldEmitter`, `InlineLookupTableFieldEmitter`, `SplitRowsMethodEmitter`, `MultiTablePolymorphicEmitter`, `TypeClassGenerator`, `TypeFetcherGenerator` (the dominant Phase 2 consumer with ~18 `ResolvedTableNames` sites, plus two `.tables.records` sites picked up in Phase 3), `TypeConditionsGenerator`, `QueryConditionsGenerator`, `FetcherRegistrationsEmitter`, `SelectMethodBody`, `NodeIdEncoderClassGenerator`, plus the `*DirectiveResolver` family. `QueryNodeFetcherClassGenerator.generate` carries a dead `jooqPackage` parameter that drops with the migration.
- After the migration, `jooqPackage` no longer threads through any emitter that takes a `TableRef` / `ForeignKeyRef`. It survives only at the two sites listed under "`jooqPackage` survivors" above.

## Test posture

A multi-schema test fixture is missing; sakila uses only `public` and the `nodeidfixture` / `idreffixture` schemas live in their own package roots, so neither exercises the failure. The Phase 1 fixture (see "Phasing" for the enumerated code paths it must exercise) lives in `graphitron-fixtures-codegen` so pipeline and compilation tiers can reuse it; the compilation tier should fail without R78 and pass with it. Phase 3 additionally recompiles `graphitron-sakila-example` against the new generator output and asserts byte-identical sources for the committed example code, locking in the single-schema regression equivalence claim.

For the 107 `new TableRef(...)` test sites, migration extends the existing `TestFixtures` helper (which already centralises ~30 of those constructions today) so each `TableRef` factory grows the new typed fields once. Helpers absorb future `TableRef` evolution without re-touching every test; a backward-compat overload would bias every future change toward preserving the legacy shape.

## User-facing impact and adjacent work

- **Documentation site update.** The dotted `@table(name: "schema.table")` syntax is user-visible and needs a how-to entry under `/docs/` and `/graphitron-rewrite/docs/`. R68's how-to series is already touching the `@table` directive (e.g. `map-types-to-tables.adoc`); R78's docs delta should land in the same family rather than as a standalone page. The R78 implementation commit lands the docs update in the same PR — out-of-band documentation lags would surface the structured ambiguity error to consumers without explaining the fix.
- **Migration messaging for multi-schema consumers with collisions.** Today these consumers compile and ship silently-wrong-schema results — the worst kind of latent bug. After R78 the same SDL fails with `"@table(name: 'film') is ambiguous; qualify as ..."`. Technically a breaking change in the sense that builds that compiled before now fail, but the previous behaviour was a correctness bug. Note in the eventual release notes / changelog so consumers know to qualify.
- **LSP directive-validation pass.** `graphitron-lsp` parses SDL independently for completion / diagnostics; the implementer should verify it accepts the dotted `name:` value and (optionally) suggests qualifications when the catalog has a collision. Out of scope as a hard R78 requirement, but a one-line check during implementation prevents a regression.
- **Multi-schema execution-tier coverage.** R78's compilation tier proves the generated code compiles. An execution-tier check (Testcontainers PostgreSQL with two schemas, exercising cross-schema FK joins at runtime) would verify jOOQ's runtime behaviour around the schema-qualified aliases. Out of scope for R78; tracked as a separate Backlog stub captured at hand-off (one paragraph: extend `graphitron-sakila-service` with a multi-schema variant).

## Rejected alternatives

- `String schemaPackage` field on `TableRef` with a `tablePackage(fallbackJooqPackage)` helper. Rejected because it still reconstructed FQNs at every consumer; the design here remembers typed references instead. Captured so a future Spec author does not retry it.
- A domain-level `QualifiedClassRef` / `QualifiedFieldRef` wrapper around `ClassName`. Rejected because every recent model record (`BatchKey`, `MethodRef`, `HelperRef`, `LifterRef`, `AccessorRef`, `ResultAssembly`, `PayloadAssembly`, `RowsMethodShape`, `ErrorChannel`, `DefaultedSlot`, `ChildField`) carries javapoet `ClassName` / `TypeName` directly. Inventing a wrapper is generality without a forcing function.
- Separate `TableRecordRef` carrying just the record class. Rejected because the record class is the same table viewed from the projection axis (`Table.getRecordType()` on the same `Table<?>`), and every consumer that needs the record class already has the `TableRef` in hand. Per the design-principles rule that new sub-taxonomies must carry distinct information a sibling cannot, the record class is a field on `TableRef`, not a separate variant.
- Sentinel empty `TableRef` (`@Nullable ClassName`, an `EMPTY` sentinel `ClassName`, or `Optional<TableRef>` propagated through the model). Rejected for the structural fix in "Catalog-miss handling": catalog-miss classifies as `UnclassifiedType` before any ref is built, so emit sites never see a partial ref.
- Naming the FK ref `KeyRef` (parallel to `TableRef`) and the constants-class field `tablesClass` (parallel to `tableClass`). Rejected for review-clarity reasons: in jOOQ "Key" covers PKs and FKs both (the host class `Keys` literally holds both kinds), and `tableClass`/`tablesClass` differ by a single `s` so a swap would not catch reviewer eye across 50+ migrated sites. Use `ForeignKeyRef` and `constantsClass` to keep the names self-disambiguating.
- Resolution-time disambiguation: "first schema wins with a warning." Rejected because warnings are routinely ignored; consumers wouldn't notice they're querying the wrong table until a customer reports missing or wrong data. The R78 design (fail-loudly + qualified syntax) makes the failure mode unmissable while giving consumers an explicit fix.
- Resolution-time disambiguation: a separate `@table(schema:)` directive argument instead of dotted syntax. Rejected because it adds a new directive surface (SDL migration; per-call-site explicit naming) for what is structurally one piece of data — the qualified table identity. Dotted syntax matches SQL's own `schema.table` notation, doesn't require an SDL change, and parses at one site (`parseQualifiedTableName`).
- Resolution-time disambiguation: deferring to a follow-up roadmap item. Rejected because R78 already touches every relevant site (`JooqCatalog.findTable`, the `BuildContext` directive readers, the multi-schema fixture). Splitting it would force the same fixture to land twice, the same `findTable` to be rewritten twice, and would leave the silent-wrong-schema bug live across two release cycles. The incremental cost inside R78 is ~5 sites against a phase that already touches ~50; outside, it's a full second pass.
- Resolution-time disambiguation: support quoted-identifier syntax (`"my.schema"."weird.table"`) for SQL identifiers containing literal dots. Out of scope for R78. PostgreSQL identifiers without quoting can't contain dots, the in-tree fixtures don't need it, and deferring the lexer-level work keeps R78 focused. A follow-up Backlog stub captures it (one paragraph: lex the directive value with a tiny escape-aware scanner, otherwise unchanged).
