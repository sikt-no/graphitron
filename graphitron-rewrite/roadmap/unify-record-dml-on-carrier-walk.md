# Retire DmlReturnExpression.Payload: unify @record-returning DML on the carrier-walk path

Two parallel designs exist today for `@record`-returning DML mutations. R75 + R141 introduced the carrier-walk path (`MutationField.MutationDmlRecordField` / `MutationBulkDmlRecordField`); the older reflective-payload path (`DmlReturnExpression.Payload` consumed by the four `DmlTableField` permits) stayed in place, and the two compete for the same SDL shapes at classify time. The duplication is structurally unprincipled: Path 2 reflectively constructs the user's payload class with the row dropped into a constructor slot and defaults elsewhere, then hands the typed instance to graphql-java as a source value; the carrier walk hands graphql-java a PK record and lets per-field DataFetchers project. Both wire through graphql-java's field resolution, but Path 2 leans on `PropertyDataFetcher` reading from a typed Java source (the eagerly-assembled DTO), while Path 1 contributes explicit per-field DataFetchers. graphql-java's per-field wiring is the right abstraction here; reflective DTO assembly is a workaround that pre-materializes a courier whose only function is to satisfy `PropertyDataFetcher`. R161 retires Path 2 entirely.

After R161, `Mutation*TableField` permits are guaranteed to never carry a `@record` return — the `DmlReturnExpression.Payload` arm that admitted them is gone, and the type system enforces narrowness structurally rather than via classifier-acceptance shape.

## Concrete state today

- `DmlReturnExpression` is a sealed type with five arms; the `Payload` arm wraps a `PayloadAssembly` (`payloadClassName`, `rowSlotType`, `rowSlot ∈ {CtorParameterIndex, SetterMethod}`, `defaultedSlots`) — the validated wiring recipe between the DML's typed jOOQ row record and the user's payload-class construction.
- `FieldBuilder.resolveDmlPayloadAssembly` (declared at `graphitron-rewrite/graphitron/src/main/java/no/sikt/graphitron/rewrite/FieldBuilder.java:2028`, body extends to `:2098`; locate by symbol) does reflection + jOOQ catalog lookup + SDL field walk to produce the assembly. Rejection cases include class-not-loadable, no row-typed parameter, multiple row-typed parameters, unresolvable construction shape, bean-arm with no row setter.
- `TypeFetcherGenerator.emitPayload` (declared at `graphitron-rewrite/graphitron/src/main/java/no/sikt/graphitron/rewrite/generators/TypeFetcherGenerator.java:3088`; locate by symbol) emits `var payload = new FilmPayload(row, null, ...)` literal DTO construction, with sibling functions `emitPayloadCtor` (`:3130`) and `emitPayloadSetters` (`:3118`) handling the two construction shapes.
- `FieldBuilder.classifyMutationField` routes to `MutationDmlRecordField` / `MutationBulkDmlRecordField` when `BuildContext.tryResolveSingleRecordCarrier(returnTypeName, kind) == Ok`; otherwise it falls through to `buildDmlField` which calls `buildDmlReturnExpression`. The `ResultReturnType` branch constructs `new DmlReturnExpression.Payload(assembly)`, and the resulting `MutationInsertTableField` (or sibling) carries a `@record` return inside its `returnExpression`.
- `BuildContext.tryResolveSingleRecordCarrier` (declared at `graphitron-rewrite/graphitron/src/main/java/no/sikt/graphitron/rewrite/BuildContext.java:530`; the candidate predicate sits in the method body at `:534-535`) admits `PlainObjectType` or `PojoResultType.NoBacking`; every other `ResultType` arm (`PojoResultType.Backed`, `JavaRecordType`, `JooqRecordType`, `JooqTableRecordType`) resolves to `NotCandidate`. So every `@record(record: {className:})` wrapper routes to Path 2 today, regardless of which sub-arm the className's class resolves to in `TypeBuilder.buildResultType` (graphitron-rewrite/graphitron/src/main/java/no/sikt/graphitron/rewrite/TypeBuilder.java:712-726): Java record → `JavaRecordType`, jOOQ `TableRecord` → `JooqTableRecordType`, jOOQ `Record` → `JooqRecordType`, plain POJO → `Backed`.
- `MutationInputResolver.validateReturnType` (`graphitron-rewrite/graphitron/src/main/java/no/sikt/graphitron/rewrite/MutationInputResolver.java:169-260`) admits `ResultReturnType` whenever `fqClassName != null` (Path 2) or `fqClassName == null` and `tryResolveSingleRecordCarrier == Ok` (Path 1). The two probes are disjoint today: the carrier walk only treats `PlainObjectType` and `PojoResultType.NoBacking` as candidates, so every wrapper with `fqClassName != null` (`Backed`, `JavaRecordType`, `JooqRecordType`, `JooqTableRecordType`) returns `NotCandidate` and routes through Path 2's reflection. The canonical pipeline-tier fixtures `DML_RECORD_PAYLOAD_RETURN_HAPPY` / `DML_RECORD_PAYLOAD_ROW_ONLY_HAPPY` exercise the `JavaRecordType` sub-arm (`DeleteFilmPayload` / `DeleteFilmRowOnlyPayload` are `public record` classes); there is no fixture exercising `JooqRecordType` or `JooqTableRecordType` as a DML wrapper, but the validator accepts them on the same `fqClassName != null` arm.

## Spec direction

- **Delete `DmlReturnExpression.Payload`.** The sealed type collapses to four arms (`EncodedSingle`, `EncodedList`, `ProjectedSingle`, `ProjectedList`). `buildDmlReturnExpression`'s signature loses its `Optional<PayloadAssembly>` parameter; the `ResultReturnType` branch in `buildDmlReturnExpression` becomes unreachable from `buildDmlField` and is deleted. The type-level Javadoc on `graphitron-rewrite/graphitron/src/main/java/no/sikt/graphitron/rewrite/model/DmlReturnExpression.java:10-17` (claiming totality over Invariant #14's admitted set) is rephrased without the `ResultReturnType` row, and the "5 arms → 4" lift surfaces in that Javadoc as well as in the trigger-table doc bullet below — both edits land in the same commit.

- **Delete the `PayloadAssembly` subsystem in the model package.** `PayloadAssembly`, `RowSlot` (entire sealed type including `CtorParameterIndex` and `SetterMethod` permits) all go from the model package. `PayloadConstructionShape` (`AllFieldsCtor` / `MutableBean`) stays in the model package: still consumed by `resolveErrorChannel` and `resolveServiceResultAssembly` for the catch-arm / service-result payload shapes. `NonBoundSetter` likewise stays: still consumed by `ErrorsSlot.SetterMethod` and `ResultSlot.SetterMethod`. The `FieldBuilder`-internal `DmlPayloadAssemblyResult.{Assembly, Reject, NoAssembly}` sealed and its `NO_ASSEMBLY` constant retire with their sole caller `resolveDmlPayloadAssembly`.

- **Delete the DML-specific reflection layer in `FieldBuilder`.** `resolveDmlPayloadAssembly` (the entry point, including its `Class.forName(result.fqClassName(), ...)` payload-class load and its `tableResolved.entry().table().getRecordType()` row-record probe) and `buildDmlPayloadAssemblyBeanArm` (the bean-arm builder; declared at `FieldBuilder.java:2100`; locate by symbol). The `FieldBuilder`-internal `DmlPayloadAssemblyResult` sealed and `NO_ASSEMBLY` retire with the entry point per the previous bullet. The shared helpers `resolvePayloadConstructionShape` (`FieldBuilder.java:465`; locate by symbol) and `collectDefaultedSlots` (`:2536`; locate by symbol) stay: both have surviving callers in `resolveErrorChannel` (`:1892` / `:1940`) and `resolveServiceResultAssembly` (`:2239` / `:2276`) for the catch-arm and service-result payload shapes. `collectNonBoundSetters` likewise stays for the same two callers (`:1963` / `:2313`).

- **Delete the emit layer in `TypeFetcherGenerator`.** `emitPayload`, `emitPayloadCtor`, `emitPayloadSetters`. The `MutationField` switch arms keyed on the deleted `DmlReturnExpression.Payload` go with them. *Load-bearing annotation hygiene.* `emitPayloadSetters` carries a `@DependsOnClassifierCheck(key = "payload-construction.setter-name-matches-sdl-field")`; that consumer retires with the method. The producer (on `FieldBuilder.resolvePayloadConstructionShape`) and the remaining consumers (the catch-arm `payloadFactoryLambda` and the service-result success-arm sites, all `ErrorsSlot`/`ResultSlot`-keyed, not `RowSlot`-keyed) stay. `LoadBearingGuaranteeAuditTest` sees one consumer disappear; no orphans, no producer-without-consumer warnings. The companion `payload-construction.shape-resolved` key has no DML-side consumer to retire. In the same commit, the producer descriptions on both `@LoadBearingClassifierCheck` annotations at `FieldBuilder.java:451-464` lose their "DML-row emitPayload" enumeration — both descriptions list three consumer sites (catch-arm payload-factory, service-result `buildSuccessPayload`, DML-row `emitPayload`); the DML-row mention is dropped so the description text matches the post-R161 consumer set.

- **Widen `BuildContext.tryResolveSingleRecordCarrier`.** The candidate predicate inside the method body (today at `BuildContext.java:534-535`; locate by symbol) expands to admit any `GraphitronType.ResultType` alongside `PlainObjectType`. One-line change. The carrier walk then handles every wrapper state uniformly: `PojoResultType.NoBacking` (already admitted), `PojoResultType.Backed`, `JavaRecordType`, `JooqRecordType`, `JooqTableRecordType`. The SDL author's `@record(record: {className:})` directive on a wrapper becomes informational metadata, not routing input. Codegen no longer reads `fqClassName` to drive routing for this case. The canonical pipeline-tier fixtures (`DML_RECORD_PAYLOAD_RETURN_HAPPY` / `DML_RECORD_PAYLOAD_ROW_ONLY_HAPPY`, both `JavaRecordType` carriers via `DeleteFilmPayload` / `DeleteFilmRowOnlyPayload`) admit through the same widened walk that admits the sakila `FilmPayload` `NoBacking` carrier; no fixture changes are required to route them. If graphitron ever needs to know the wrapper's Java class for some future purpose, reflection on the actual classpath is the source of truth ; the directive can drift, reflection cannot. (Narrower alternatives considered: admit only `Backed` and reject the three jOOQ-record / Java-record arms in `validateReturnType` as deferred. Rejected: every `ResultType` wrapper exposes the same SDL field shape the carrier walk dispatches on, and the className informs nothing the post-R161 codegen consumes, so there is no principled reason for the carrier walk's candidate predicate to fork on which sub-arm the className resolves to.)

  *Consumer-audit confirmation.* The structural argument above (the field-set walk in `classifyCarrierField` doesn't fork on the carrier type's `ResultType` sub-arm) is also empirically true on trunk: zero post-R161 consumers of `types.get(carrierName)` switch on `instanceof ResultType.*` sub-arms; the only carrier-type reads are the field-set walk and informational-metadata reads of `ResultType.fqClassName()` in service-side paths that operate uniformly across the arms. There is a residual structural asymmetry (`TypeBuilder.promoteSingleRecordCarriers` at `TypeBuilder.java:192` promotes `PlainObjectType → NoBacking`; the four className-carrying arms stay in their classified arm) that any future site forking on `instanceof PojoResultType.NoBacking` vs the other arms would observe. R161's commit pairs the widening with a grep audit of `types.get` callsites (or equivalent classifier-side lookups) to confirm none switch on this asymmetry; if a future patch introduces such a fork, that is the moment for **Generation-thinking**'s "lift the fork into the model" treatment, not R161.

- **Update `validateReturnType.ResultReturnType`'s admission.** Both `fqClassName != null` and `fqClassName == null` arms admit via `tryResolveSingleRecordCarrier`; the existing `fqClassName == null + Rejected` rejection arm extends to cover `fqClassName != null + Rejected` with the same diagnostic shape. Concretely: the `r.fqClassName() == null` guard at `MutationInputResolver.java:209` is deleted; the probe runs unconditionally (one probe over the SDL shape, not two probes composing). The two-arms-with-the-same-diagnostic shape would have left the validator with twin code paths that read identically; merging them is what makes the principled framing ("one probe keyed off the SDL shape") visible in the code.

  *Validator-rejection-message delta.* Path 2's reflection-based validate-time rejections retire with `resolveDmlPayloadAssembly`, but they split into two buckets:
   - **Becomes-unreachable.** `payload class 'X' could not be loaded`: post-R161 no emitter site references the wrapper's developer-supplied class, so a misconfigured `className` doesn't even reach javac (the className becomes informational SDL metadata that nothing reads). This diagnostic just goes away; if the className silently drifts, no failure surfaces until a runtime reflection consumer (none today) reaches for it.
   - **Re-expressed or deferred.** `payload class 'X' has no parameter typed as <RowRecord>`, `payload class 'X' has multiple parameters typed as <RowRecord>`, bean-arm `no setter typed as <RowRecord>`: these fed `emitPayload*` shape assumptions that themselves retire, so the checks are no longer load-bearing in the residual codebase. When the underlying defect also shows up at the SDL layer (e.g., a wrapper with no `DataChannel` field), the existing `"' requires a new CarrierFieldRole permit"` rejection in `BuildContext.classifyCarrierField` surfaces it via a different diagnostic. Otherwise no check fires; nothing downstream depends on the shape.

  Acceptable trade per the principle "if graphitron ever needs to know the wrapper's Java class for some future purpose, reflection on the actual classpath is the source of truth ; the directive can drift, reflection cannot". Per the converse direction of "Classifier guarantees shape emitter assumptions", a check is load-bearing only when an emitter relies on the narrowed shape; once the emitter retires, the check is hygiene and may retire with it.

- **Document the RETURNING(PK) design decision near `buildMutationDmlRecordFetcher`** (declared at `TypeFetcherGenerator.java:3656`; locate by symbol). The carrier-walk emit uses `.returningResult(pkCols).fetchOne()` deliberately: minimize work in the write transaction, do all projection work in a read-only follow-up SELECT outside the transaction. Today the rationale is implicit; a class-level doc comment on the method captures it so future refactors don't "optimize" the follow-up SELECT away by switching to `.returning(*)` and projecting the captured row directly. The same principle applies to `buildMutationBulkDmlRecordFetcher` (`:3923`) and earns the same documentation treatment.

- **Document the `*TableField` ⇒ no-`@record`-return rule in `code-generation-triggers.adoc`.** The structural change is a sealed-hierarchy collapse on `DmlReturnExpression` (5 arms → 4) ; not a record-component-type narrowing in the sense of `rewrite-design-principles.adoc` § "Narrow component types over broad interfaces" (which is about declaring a component as `TableBoundReturnType` rather than `ReturnTypeRef`; here `DmlTableField.returnExpression` stays typed as `DmlReturnExpression`). The doc work that earns its place in R161 is the user-facing trigger table at `graphitron-rewrite/docs/code-generation-triggers.adoc:185-198` § "Mutation Fields". Concretely: (1) the four existing `MutationInsertTableField` / `MutationUpdateTableField` / `MutationDeleteTableField` / `MutationUpsertTableField` rows have their "Schema Pattern" cell qualified with the return-shape filter — e.g. `@mutation(typeName: INSERT)`, returning `ID` or a `@table` type → `MutationInsertTableField`, replacing the unqualified `@mutation(typeName: INSERT)` text; this lifts the rule into the routing column where it belongs rather than tucking it into the "Generates" column or a separate caveat row; (2) two new rows are appended for `MutationDmlRecordField` (`@mutation(typeName: INSERT|UPDATE|DELETE|UPSERT)`, returning a single-record `@record(record:{className:...})`-or-bare-SDL carrier with one `DataChannel` and optional `ErrorChannelRole` → `MutationDmlRecordField`) and `MutationBulkDmlRecordField` (same but with bulk `@table` input and a list-shaped data field on the carrier), so the redirect lands on a visible row in the same table rather than a forward reference. No edit to `rewrite-design-principles.adoc` ; the existing worked examples in § "Narrow component types" stay; this collapse is structural editing rather than a new pattern to teach. Additional structural splits on `DmlReturnExpression` itself (e.g., factoring into `EncodedReturn` / `ProjectedReturn` sub-sealed roots) are deferred to R162's verb-on-permit-identity consolidation.

- **No rename, no permit split.** `MutationDmlRecordField` and `MutationBulkDmlRecordField` keep their names. The verb-on-permit-identity consolidation (`MutationInsertResultField` / `MutationUpdateResultField` / etc., with the four `DmlTableField` permits and the service-backed pair folding under the same scheme) is tracked separately at **R162** (`mutation-result-field-sealed-on-kind`). R161 is bare retirement.

## Status of the R12 dependency

The carrier walk rejects any wrapper field that doesn't classify into a `CarrierFieldRole` permit. R161 needs the `errors:` carrier field to classify into `CarrierFieldRole.ErrorChannelRole` so wrappers shaped `{ film: Film, errors: [DeleteFilmError] }` route end-to-end after the `BuildContext` widening (test fixture `DML_RECORD_PAYLOAD_RETURN_HAPPY` at `graphitron-rewrite/graphitron/src/test/java/no/sikt/graphitron/rewrite/GraphitronSchemaBuilderTest.java:5982` is the canonical case).

The producer-side R12 wiring R161 leans on has already landed on trunk (even though R12's overall status remains "In Progress" pending execute-tier coverage). Specifically:

- `CarrierFieldRole.ErrorChannelRole` producer in `BuildContext.classifyCarrierField` (locate by symbol; runs ahead of the `DataChannel` resolution, emits `ErrorChannel.LocalContext`).
- `ChildField.Transport` sealed type (`PayloadAccessor` / `LocalContext`) on `ChildField.ErrorsField`, with `FetcherEmitter.dataFetcherValue`'s `ErrorsField` arm switching on `field.transport()` exhaustively.
- `TypeFetcherGenerator.catchArm`'s sealed switch on `ErrorChannel` with `PayloadClass` ⇒ `ErrorRouter.dispatch` and `LocalContext` ⇒ `dispatchToLocalContextCatchArm` arms; the latter emits `DataFetcherResult.newResult().data(null).localContext(List.of(t)).build()`.
- `MappingsConstantNameDedup`'s grouping switch is sealed-aware over `ErrorChannel`, and its `withResolvedChannel` rewrite switch carries arms for `MutationDmlRecordField` and `MutationBulkDmlRecordField`.
- `LoadBearingClassifierCheck(key = "error-channel.local-context-transport")` producer / matching `DependsOnClassifierCheck` consumers wired and audited by `LoadBearingGuaranteeAuditTest`.

What's still pending on R12 (per its In Progress plan) does not block R161: it's primarily new execute-tier coverage (`ResultAssembly.Assembly` end-to-end for service-method-returns-domain-object, and `ValidationHandler` end-to-end gated on R94's input-record emission). R161 can begin once Ready; sequencing remains R12 → R161 for orderly handoff, but the structural blocker is gone.

## Relationship to R12 / R158 / R159

R12 (`error-handling-parity`) is the structural dependency per the section above. R12's classifier-producer and consumer-side wiring R161 leans on have already landed on trunk; R161 is unblocked. R12 remains "In Progress" pending execute-tier coverage that R161 does not touch.

R158 (`SourceKey.Reader` sub-taxonomy for `@service`-backed producers) and R159 (the semantic admission question for payload carriers) sit on the consumer-side data-field axis (`ChildField.SingleRecordTableField`'s producer). R161 sits on the producer-side classifier axis. R158 inherits the cleaner invariant downstream — once R161 lands, `MutationInsertTableField` etc. never carry `@record`, so any future reasoning about the `MutationField` ↔ `ChildField.SingleRecordTableField` producer-consumer pair lives entirely inside the carrier-walk permits. But R158 doesn't *require* R161: `TableTargetField`'s sealed-interface contract already enforces `TableBoundReturnType returnType` on `SingleRecordTableField`, which is the narrower invariant R158 actually leans on.

R12, R158, R159, R161, R162 can land in any order modulo the structural dependencies: R161 → R12 (R12's carrier-walk producer + LocalContext wiring before R161, now satisfied on trunk), R162 → R161.
