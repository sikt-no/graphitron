## Remaining work

One item remains after items 1 and 3 shipped.

### 2. Composite-PK parent execution test (junction-as-parent fixture)

The B4c-2 round-2 unit tests pin the RowN widening path against a synthetic `Project` parent with composite PK `(org_id, project_id)`. The execution tier does not exercise it: sakila's composite-PK tables (`film_actor`, `film_category`) are pure junctions with no children FK-referencing the junction.

**Implementation:** add a real composite-PK parent to the fixture DB. Synthetic `project (org_id, project_id, …)` plus a child table that FK-references the composite parent (e.g. `task (org_id, project_id, task_id, …)`). Append the new tables to `graphitron-fixtures/src/main/resources/init.sql` (the fixtures' single canonical DDL script; no Flyway in this module) and let the next build regenerate jOOQ. Declare a polymorphic interface return type for the child connection field, e.g. `Project.workItems: [WorkItem!]! @asConnection(defaultFirstValue: 5)` where `interface WorkItem` is implemented by a couple of tables with single-PK participants; the parent is the composite-PK side, participants stay single-PK to scope this item.

**Tests:** new execution tests in `GraphQLQueryTest` mirroring the `addressOccupantsConnection_*` shape (parent-scoped rows, parent-scoped totalCount, after-cursor pagination, inline-fragment per-type resolution) against the composite-PK parent fixture. Extend the `dataLoaderBatchesAcrossParents` SQL_LOG ratchet so multi-parent batching is verified for the composite-key path too (one `as "parentInput"` statement regardless of parent count).

**Out of scope here:** this item only adds the execution-tier ratchet for composite-PK parents. Composite-PK *participants* in connection mode is item 1, separately. If items 1 and 2 land together, a single fixture can exercise both paths.
