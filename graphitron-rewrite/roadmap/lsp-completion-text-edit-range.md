# Set explicit TextEdit range on every completion item so clients do not concatenate the prefix with the candidate

Every completion provider under `graphitron-lsp/.../completions/` builds its `CompletionItem` as `new CompletionItem(label); item.setKind(...);` and returns; no `textEdit`, no `insertText`, no `filterText`. Per LSP, when an item omits `textEdit`/`insertText` the client decides which span to replace from its own word-boundary rules, and those rules vary by client. The user reported the concrete failure mode in eglot: with `com.example.FilmServ|` at the cursor inside `@service(service: {className: "com.example.FilmServ"})`, eglot's `bounds-of-thing-at-point` with `'symbol` is governed by `graphql-mode`'s syntax table, which does not include `.` as a symbol constituent, so the "word" at point is `FilmServ`. The server returns the candidate label `com.example.FilmService`. Eglot replaces just `FilmServ` with the full label, yielding `com.example.com.example.FilmService`. VS Code happens to mask this for dotted FQNs because its default word definition is broader, but the LSP spec puts the responsibility on the server and relying on client heuristics is a portability footgun. The fix is to attach an explicit `TextEdit` to every completion item whose range covers the value (or partial value) at the cursor.

**Scope.** All eight `CompletionItem` construction sites across the providers in `graphitron-lsp/src/main/java/no/sikt/graphitron/lsp/completions/`: the six string-literal value providers (`ClassName`, `Method`, `Table`, `Field`, `Reference`, `ScalarType`), `NodeTypeCompletions`, and `ArgNameCompletions` (both the user-directive and bundled-directive paths). The bug is acute for value providers with dots in the candidate (`ClassName`/`ScalarType` FQNs, `Method` `Class#method`, dotted `table.column` in `Field`/`Reference`/`Table`); `NodeType` and `ArgName` write single tokens where eglot's default heuristic happens to land correctly, but the fix should be uniform so the provider contract is the same across the dispatch.

**Design.** The cursor's replaceable range is a sibling output to the cursor's coordinate; both come from the same descent through the directive's argument tree (`enclosing arg` → `object_field` chain → leaf value). That descent already lives in `LspVocabulary.coordinateAt`, so the range computation lives next to it as `LspVocabulary.replaceRangeAt(directive, pos, source) -> Optional<Range>`. Siting it in `Nodes` would split the parse boundary (`Nodes` is pure tree-sitter mechanics today with no opinion on GraphQL value-node kinds); siting it in a per-provider helper would duplicate the walk in eight places and let it drift from `coordinateAt`. For a `string_value` leaf the inner range is `[startByte + 1, endByte - 1]` (content between the quotes); for an `enum_value` or bare `name` token the range is the node's full span; for a `block_string` the helper returns `Optional.empty()` and the provider falls back to today's no-`textEdit` behaviour (block-string slots are not bound to value-completion coordinates today, so this is informational rather than a degradation); for a cursor on whitespace inside the directive's parens (arg-name slot with no partial identifier yet) the helper returns a zero-width range at `pos`. UTF-8↔UTF-16 conversion goes through the existing `Positions.toLspPosition`.

The range is computed once at the dispatch site in `GraphitronTextDocumentService.coordinateBasedCompletions` (right next to where it would already compute the coordinate if it were hoisted there too, but that hoist is out of scope), then threaded into every provider's `generate(..., Optional<Range> replaceRange, ...)` signature. Each provider's `toCompletionItem(...)` helper applies it via `setTextEdit(Either.forLeft(new TextEdit(range, label)))` when the range is present; absent range means today's no-`textEdit` item. lsp4j 0.24's `CompletionItem.setTextEdit` takes `Either<TextEdit, InsertReplaceEdit>`; the plain `TextEdit` form is enough because insert position and replace position coincide.

**Test coverage.** One parameterized test in a new `CompletionTextEditTest` (or appended to `TextDocumentServiceTest`) over the eight providers, asserting the shared invariant: "for a candidate selected with a non-trivial prefix already typed at the cursor, the returned item's `TextEdit.getRange()` covers the whole value at the cursor, not just the suffix." Each parameter row carries a small SDL fragment, a cursor position, the directive name, and the expected range. A second row per provider with an empty value (`"|"` or whitespace) pins the zero-width-range case. A `block_string`-cursor row pins the helper's `Optional.empty()` degradation: the returned item has no `textEdit`. The existing per-provider `*CompletionsTest` files stay focused on label/kind/keyset assertions; they're not the regression-pin tier for the wire-format invariant. Tests construct `Range` against the test source string; no client involvement needed.

**Out of scope.** This item does not change which coordinates are completable, what the candidate set is, or the providers' dispatch shape. Block-string slot support, `filterText` for partial-match scoring, and `insertText` snippet syntax are separate concerns and stay on the table for later items.
