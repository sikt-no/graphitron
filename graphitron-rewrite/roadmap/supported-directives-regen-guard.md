# Regenerate and guard the generated supported-directives migration doc against directive-set drift

`docs/manual/_generated/supported-directives.adoc` is a generated fragment (included by
`docs/manual/how-to/migrating-from-legacy.adoc`) listing every directive the rewrite declares, the
legacy-only directives, and the argument-shape changes between the two. It is produced by
`graphitron-roadmap-tool directive-support --mode=migration`
(`DirectiveSupportReport.renderMigration`). The on-disk file has silently drifted from what the
generator would emit today: it is missing `@routine` (added to `directives.graphqls` by R300) and
`@scalarType`, and `@sourceRow` is mis-ordered (it sits second, though the generator sorts the
`Supported directives` list alphabetically). So the migration guide under-reports the supported
directive set, and a reader migrating off legacy is told `@routine` / `@scalarType` do not exist in
the rewrite.

The root cause is process, not content: the file's own header says "Regenerate via the verify-mode
CI guard," but **no such guard exists** — there is no workflow step, pom execution, or script in
the repo that regenerates this file or diffs it against the committed copy (confirmed by grepping
`.github/workflows`, the docs/rewrite poms, and the repo for `directive-support` /
`supported-directives`). It is regenerated by hand with ad-hoc CLI args, so it drifts every time a
directive is added or removed and nothing fails. The `directive-support` CLI
(`Main.java`) takes three positional args (legacy `directives.graphqls`, rewrite
`directives.graphqls`, colon-separated fixture dirs) plus `--mode=migration --output=<path>`; none
of these are pinned anywhere, which is also why a careless regeneration risks churning the
`Legacy-only directives` and `Argument-shape changes` sections (those depend on the legacy file and
fixture-dir args).

## Scope

1. **Pin the canonical invocation.** Record the exact `directive-support --mode=migration` argument
   set (which legacy `directives.graphqls`, which fixture dirs, the output path) in one place so
   regeneration is deterministic and reproducible, rather than reconstructed by hand each time. The
   `leaf-coverage` regenerate step in `.github/workflows/rewrite-build.yml` and the roadmap-README
   regeneration (`roadmap-tool` default `exec:java` run) are the two in-repo precedents for "a
   generated doc artifact with a pinned regeneration command."
2. **Add the verify-mode CI guard the header already promises.** Regenerate into a temp location and
   diff against the committed `supported-directives.adoc`, failing the build on any difference. This
   mirrors the existing README sync check (CI verifies the rolled-up README stays in sync with
   front-matter). Pick the model that fits a *committed* generated file: a verify-diff guard (like
   the README), not an upload-artifact flow (like leaf-coverage, whose output is not committed).
3. **Regenerate the file once** so it reflects the current directive set (adds `@routine` and
   `@scalarType`, fixes the `@sourceRow` ordering), landing the content fix the guard will then hold
   in place.

## Design forks for the reviewer

- **Where the guard lives.** Candidate A: extend the `roadmap-tool` default `exec:java` run (which
  already regenerates the README, runs `check-adoc-tables`, and renders the roadmap staging) to also
  regenerate `supported-directives.adoc`, then let the existing sync check cover it. Candidate B: a
  dedicated `rewrite-build.yml` step in the docs path. Recommendation: A, because it co-locates this
  fragment's regeneration with the other generated docs the tool already owns and reuses one sync
  mechanism; B duplicates the diff-guard logic. The reviewer should confirm the roadmap-tool default
  run is the right home, or steer to B if the docs-build job is the better seam.
- **Prose-vs-filter mismatch (decide explicitly).** The generated `Supported directives` paragraph
  claims each listed directive is "exercised by at least one execution-tier or pipeline-tier test
  fixture," but `renderMigration` filters the list purely on "declared in the rewrite
  `directives.graphqls`" (`rewriteByName.containsKey`); fixture usage (`inFixtures`) is computed but
  does not gate the list. Either gate the list on `inFixtures` (and surface unexercised directives
  separately) or correct the prose to match the actual "declared by the rewrite" criterion.
  Recommendation: correct the prose, since the list's job is the legacy→rewrite parity surface, not
  a fixture-coverage report (leaf-coverage already owns coverage). The reviewer should ratify which
  way the mismatch resolves before implementation, since it changes what the guard pins.

## Out of scope

- The directive *reference* index wiring for `@routine` (already fixed on trunk at `0ed125e`); this
  item is only the `_generated/` migration fragment and its regeneration guard.
- Any change to the migration guide's hand-authored prose beyond the one generated paragraph named
  in the fork above.
- `supported-schema-shapes.adoc` (the sibling generated fragment). It is generated by
  `leaf-coverage --mode=migration` and shares the identical "Regenerate via the verify-mode CI
  guard" header. Deferring its guard to a follow-up is fine, but the deferral must rest on the true
  state of the world (see reviewer note below), not on a claim that CI already keeps it fresh.

## Spec review (R346, Spec -> Spec): revisions requested

Reviewer session `session_01WvsS7jsHHZvRvH2XbTdFLk` (independent of the Backlog -> Spec committer).
The plan's shape is sound, well-scoped, and the two design forks are honest. One material factual
error and two smaller refinements before this is Ready.

1. **The motivating "asymmetry" / "closest precedent" is false (blocking).** The original
   *Out of scope* bullet claimed `supported-schema-shapes.adoc` "*is* regenerated by the
   `Regenerate leaf-coverage report` step in `rewrite-build.yml` ... so it is kept fresh by CI; the
   gap is specific to `directive-support`." It is not. That CI step
   (`.github/workflows/rewrite-build.yml:79-85`) runs `leaf-coverage graphitron-rewrite` with **no**
   `--mode=migration`, so it regenerates the internal `inference-axis-coverage.adoc` report
   (`LeafCoverageReport.run`, default branch at line 134), *not* the migration fragment. Nothing in
   `.github/workflows`, the docs/rewrite poms, or any script runs `leaf-coverage --mode=migration`
   (the only callsite is the `include::` in `migrating-from-legacy.adoc:18`). So
   `supported-schema-shapes.adoc` has the *identical* unguarded gap as `supported-directives.adoc`
   and carries the same false "verify-mode CI guard" header. This matters because the claim is cited
   as "the clearest evidence of the missing guard and the closest precedent for wiring one" ; an
   implementer following that precedent will find no guard to copy. Bears on the stale-reference rule
   ("Documentation names only live tests/code": a named CI step credited with behaviour it does not
   have). *Revision:* correct the framing, and make a deliberate call ; since both migration
   fragments share one mechanism (`roadmap-tool`'s `--mode=migration` + a verify diff), guarding both
   in one pass is the lower-churn option. If `supported-schema-shapes.adoc` stays out of scope, say
   why (e.g. its drift is independently visible / a separate follow-up item) rather than asserting CI
   covers it.

2. **Candidate A oversells reuse (refinement).** The fork says Candidate A "reuses one sync
   mechanism; B duplicates the diff-guard logic." But the existing CI-wired verify
   (`verify-roadmap-readme` phase execution in `roadmap-tool/pom.xml`) runs the `verify` subcommand,
   which diffs only the rolled-up README against front-matter ; it does not diff these `_generated`
   fragments. Whichever home wins, a new diff-guard for the fragment(s) must be written; A reuses the
   *hosting seam* (the verify phase already runs on `mvn verify`), not a ready-made diff. Note also
   that `directive-support` has **no** `--verify` flag today (only `--mode=migration` / `--output=`),
   whereas `leaf-coverage` already has one (unwired). Scope #2 should name which of these the
   implementer extends. *Revision:* restate A's advantage as "reuses the verify-phase seam," and have
   Scope #2 pick: add `--verify` to `directive-support` (mirroring `leaf-coverage`), or do the
   regenerate-to-temp + diff in the pom/CI step.

3. **Prose-vs-filter fork: recommendation is sound; confirm the verify implication (minor).** The
   spec correctly identifies that `renderMigration` filters the list on `rewriteByName.containsKey`
   (line 533) while the prose claims fixture exercise; `inFixtures` is computed but only gates the
   *Argument-shape changes* rows. The recommendation (correct the prose) is the right call and aligns
   with leaf-coverage owning coverage. No change needed beyond making sure the regenerate-once step
   (Scope #3) lands the prose fix in the same pass the guard then pins, so the guard isn't pinning a
   sentence the team has already agreed is wrong.

Everything else verified clean: `@routine`/`@scalarType` are declared in the rewrite
`directives.graphqls` (lines 421, 27) but absent from the committed fragment; `@sourceRow` sits at
position 2 though the generator sorts alphabetically (`.sorted()`, line 534), so it re-orders on
regen; the header's "verify-mode CI guard" promise is unbacked; `@routine` came from R300; the
reference-index fix is at `0ed125e`; the `directive-support` CLI arg shape matches.

Sign off: **no** ; request revisions (status stays Spec). Address finding 1 (and ideally 2),
then hand to a fresh reviewer session (not this one, since this note makes me the last committer).
