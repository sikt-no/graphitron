* Ingressen
#+BEGIN_QUOTE
Vi på Sikt skriver skjemaet vårt først for deretter å koble dette opp mot
våre datakilder. Dette er ganske enkelt og smertefritt, men hva gjør du
når det dukker opp verbose mønster som GraphQL SDL ikke lar oss
generalisere?
#+END_QUOTE

#+BEGIN_NOTES
Lukk loopen på slutten med samme bilde: vi løste det med makroer som
løfter frem semantikken, ikke bare ekspanderer tekst.
#+END_NOTES

* SDL er deklarativt. Det er styrken.

Men SDL har ingen abstraksjons­mekanisme:

- ingen funksjoner
- ingen generics over feltdefinisjoner
- ingen include

Når et mønster gjentar seg, ender du med å copy-paste det.

* Eksempel: Relay-paginering, håndskrevet

#+ATTR_REVEAL: :code_attribs data-line-numbers=1-5|7-11|13-15|17-23|25-30
#+BEGIN_SRC graphql
type Query {
    films(first: Int, after: String,
          last: Int, before: String): FilmsConnection!
    students(first: Int, after: String,
             last: Int, before: String): StudentsConnection!
}

type FilmsConnection {
    edges: [FilmsEdge!]!
    nodes: [Film!]!
    pageInfo: PageInfo!
}

type FilmsEdge {
    cursor: String!
    node: Film!
}

type StudentsConnection {
    edges: [StudentsEdge!]!
    nodes: [Student!]!
    pageInfo: PageInfo!
}

type StudentsEdge {
    cursor: String!
    node: Student!
}

type PageInfo {
    hasNextPage: Boolean!
    hasPreviousPage: Boolean!
    startCursor: String
    endCursor: String
}
#+END_SRC

#+BEGIN_NOTES
Pek og si: tre typer per paginerte felt, ti dersom du har ti felt. Spec
står i Relay-dokumentet. Vi har ingen mekanisme for å si "dette er en
connection" på en kort måte.
#+END_NOTES

* Sett scenen

- Sikt skriver skjemaet først; det er kontrakten med konsumenten.
- Graphitron kobler skjemaet til jOOQ og genererer resolvere.
- Frem til verbose mønstre dukket opp var dette enkelt og smertefritt.
- Så kom Relay, federation, feature-flagg.

* Første forsøk: ny syntaks

#+ATTR_REVEAL: :code_attribs data-line-numbers=1-3|2
#+BEGIN_SRC graphql
type Query {
    films: [Film!]! FilmConnection
}
#+END_SRC

=FilmConnection= fantes ikke i utviklerens skjema. Pipeline produserte
den.

Resultatet: skjemaet utvikleren jobbet med var ikke gyldig GraphQL.

#+BEGIN_NOTES
LSP-en forsto det ikke. Federation-verktøy avviste det. Tooling-økosystemet
rundt GraphQL forventer at filen du redigerer er en lovlig SDL-fil.
Vi hadde laget vårt eget språk, ikke en utvidelse av et eksisterende.
Det var første lærdom: skjemaet utvikleren skriver må være lovlig SDL.
#+END_NOTES

* Andre forsøk: tekstmakroer

#+ATTR_REVEAL: :code_attribs data-line-numbers=1-3|2
#+BEGIN_SRC graphql
type Query {
    films: [Film!]! @asConnection
}
#+END_SRC

Lovlig SDL. En transformasjons­fase ekspanderer direktivet til Connection-
og Edge-typer før generatoren får skjemaet.

Pre-prosessor. Direktivet er borte før generatoren ser noe.

#+BEGIN_NOTES
Dette er legacy-tilnærmingen. Stort steg fra forrige forsøk: skjemaet
validerer, tooling fungerer. Men det er fortsatt bare det mekaniske.
Generatoren vet ikke at dette er en connection, bare at det finnes en
type =FilmsConnection= med =edges= og =nodes=.
#+END_NOTES

* Hvor knirker det?

To hull i den mekaniske tilnærmingen:

#+ATTR_REVEAL: :frag (appear)
- *Håndskrevne connections.* Vil skjemaforfatteren ha et tilpasset
  =edge=-felt, må hele Relay-shapen håndskrives. To parallelle veier i
  koden.
- *Direktiver gror i generatorene.* Hver generator må selv vite hvilke
  direktiver den bryr seg om. N steder å oppdatere når makroen vokser.

* Det semantiske løftet

=Connection= og =Edge= er ikke shapes.

De er /begreper med betydning/.

#+BEGIN_NOTES
Tesen for resten av talen. Vi har ikke bare flyttet ekspansjonen ett
abstraksjonslag opp; vi har gitt navn på domenebegreper en plass i
modellen. Det er det som gjør at neste makro (=@asFacet=) faller naturlig
på plass.
#+END_NOTES

* Hva /betyr/ disse typene?

#+ATTR_REVEAL: :frag (appear)
- *Connection.* Paginert traversering av en samling.
- *Edge.* En posisjon i traverseringen — cursor pluss node.
- *PageInfo.* Kontroller for traverseringen.

Ikke vilkårlige felt-kombinasjoner. Navn på begreper i domenet vårt.

* Modellen navngir begrepene

#+ATTR_REVEAL: :code_attribs data-line-numbers=1|2-9|11-12
#+BEGIN_SRC java
sealed interface GraphitronType {
    record ConnectionType(
        String name,
        SourceLocation location,
        String elementTypeName,
        String edgeTypeName,
        boolean itemNullable,
        boolean shareable,
        GraphQLObjectType schemaType
    ) implements GraphitronType {}

    record EdgeType(...)         implements GraphitronType {}
    record TableBackedType(...)  implements GraphitronType {}
    // ...
}
#+END_SRC

En variant i modellen er et /navn/ på et begrep. Compileren tvinger
hver generator til å håndtere alle begrepene.

#+BEGIN_NOTES
Sitat fra rewrite-design-principles.md:9 å nevne her hvis det er tid:
"Generators receive a model in terms of 'what to emit', not 'what to
interpret'." Modellen bærer mening; generatoren oversetter mening til kode.
#+END_NOTES

* Klassifikatoren gjenkjenner /begrepet/

#+BEGIN_SRC text
Classifier produces this variant for two inputs that are
semantically the same:

  Directive-driven: a field carrying @asConnection on a bare list.
                    The classifier synthesises the Connection type.

  Structural:       a hand-written object type whose field shape
                    matches the Relay connection pattern.
                    The classifier recognises the pattern and
                    routes the declared type here.
#+END_SRC

/GraphitronType.java:350/

Direktiv-drevet eller strukturelt — det er samme /begrep/, samme variant,
samme generator.

#+BEGIN_NOTES
Dette er hjertet av talen. Les sitatet sakte. Direktivet sier "dette er
en connection". Strukturen /viser/ "dette er en connection". Begge gir
samme begrep i modellen, fordi det /er/ samme begrep.
#+END_NOTES

* Begrepet er bundet til feltet

#+BEGIN_SRC graphql
type Query {
    films:      [Film!]! @asConnection
    favoritter: [Film!]! @asConnection
}
#+END_SRC

Produserer:

#+BEGIN_SRC graphql
QueryFilmsConnection
QueryFavoritterConnection
#+END_SRC

Ikke én delt =FilmsConnection=. To begreper, fordi to feltkontekster.

Det koster SDL-verbosity. Det er bevisst.

* Hvorfor ikke gjenbruke?

En Connection er ikke "en paginert liste av Film".

Det er /denne/ traverseringen, fra /dette/ feltet:

#+ATTR_REVEAL: :frag (appear)
- cursor er bundet til sorteringen
- =totalCount= er bundet til filtrene
- fasettene (kommer) er bundet til filterinputen

#+ATTR_REVEAL: :frag appear
Gjenbruk ville kollapset to begreper til én struktur. Det er det
/strukturelle/ synet, ikke det /semantiske/.

#+BEGIN_NOTES
Dette er det strukturell type-gjenkjenning ikke gir oss. Klassifikatoren
ruter direktiv-drevne og strukturelt-detekterte connections til samme
ConnectionType-variant — men varianten har feltkonteksten med seg fordi
hver felt-Connection er sin egen. Det hindrer "structural detection" fra
å smitte over til "structural sharing".
#+END_NOTES

* Et nytt begrep i samme familie: Facet

Connection er /navigasjon/ i en filtrert samling.

Facet er /feedback/ om den samme samlingen — hvor mange treff finnes
under hver mulige filter-verdi?

To navn på beslektede domenebegreper.

#+BEGIN_NOTES
Vi har ikke bygget =@asFacet=-makroen enda. Status er Spec, fem faser
planlagt, Phase 1 SQL-spike er kjørt og bekreftet. Poenget her er at
Facet ikke er "nok et tilfelle av samme arkitektur", men "nok et begrep
i samme domene". Det er to forskjellige måter å si "ja, mer av dette".
#+END_NOTES

* Problemet: fasettering

Brukeren har filtrert på =rating: [PG]= i et søk.

UI-en vil vise:

#+BEGIN_QUOTE
"Hvis jeg fjerner denne, hvor mange treff får jeg på hver av de andre
rating-verdiene? Og hvor mange får jeg per kategori, gitt at PG fortsatt
er valgt?"
#+END_QUOTE

Det er omtrent det styggeste mønsteret du kan håndskrive i SDL og SQL.

* =@asFacet=: hva forfatteren skriver

#+ATTR_REVEAL: :code_attribs data-line-numbers=1-3|5-9|7|8
#+BEGIN_SRC graphql
type Query {
    filmer(filter: FilmFilter): [Film!]! @asConnection
}

input FilmFilter {
    rating:   [MpaaRating!] @field(name: "RATING")        @asFacet
    category: [String!]     @field(name: "CATEGORY_NAME") @asFacet
    title:    String        @field(name: "TITLE")
}
#+END_SRC

Én annotasjon per fasett-felt. Det er hele inngrepet.

#+BEGIN_NOTES
Pek på at =title= ikke har =@asFacet=. Ikke alle filterfelt skal være
fasetter; det er forfatterens valg.
#+END_NOTES

* =@asFacet=: hva konsumenten ser

#+ATTR_REVEAL: :code_attribs data-line-numbers=1-7|7|9-12|14-15
#+BEGIN_SRC graphql
type QueryFilmerConnection {
    edges: [QueryFilmerConnectionEdge!]!
    nodes: [Film!]!
    pageInfo: PageInfo!
    totalCount: Int
    facets: QueryFilmerConnectionFacets
}

type QueryFilmerConnectionFacets {
    rating:   [MpaaRatingFacetValue!]!
    category: [StringFacetValue!]!
}

type MpaaRatingFacetValue { value: MpaaRating! count: Int! }
type StringFacetValue     { value: String!     count: Int! }
#+END_SRC

* Tre nøkkelpoenger

* (i) Det er en kontrakt, ikke en ekspansjon

=@asFacet= etablerer et bånd mellom filter-input og facet-output.

Modellen vet at fasetten =rating= kommer /fra/ filterfeltet =rating=.

* Filter-minus-self

Når filteret er =rating: [PG]=:

- =facets.rating= viser /alle/ rating-verdier (slik at brukeren kan pivotere).
- =facets.category= viser kun tellinger der =rating = PG= er aktivt.

Dette er semantikk du *ikke* kan uttrykke som tekst-ekspansjon.

#+BEGIN_NOTES
Tekstmakroer ekspanderer; de etablerer ikke kontrakter. Filter-minus-self
er en kontrakt mellom input og output som runtime må respektere. Det
kommer av at klassifikatoren har båndet i modellen.
#+END_NOTES

* (ii) Round-trip-symmetri er typesikret

#+ATTR_REVEAL: :code_attribs data-line-numbers=1|3|5
#+BEGIN_SRC graphql
type MpaaRatingFacetValue { value: MpaaRating! count: Int! }

input FilmFilter { rating: [MpaaRating!] @asFacet ... }

filter: { rating: [facetValue.value] }   # virker, ingen coercion
#+END_SRC

Verdien har /samme skalar/ som filterfeltet den kom fra. Klienten kan
fôre =facetValue.value= rett tilbake i neste filter.

GraphQL-typesjekken passer på.

#+BEGIN_NOTES
Original Jira-ticket foreslo å kode alle fasett-verdier som =String=. Vi
avviser det. Det er nettopp den semantiske kontrakten makroen er der for
å bevare; en stringly-typed API ville bryte den.
#+END_NOTES

* (iii) Additiv arkitektur

Hvorfor er dette enkelt nå?

* Fordi sporet er lagt

#+ATTR_REVEAL: :frag (appear)
- Pipelinen som ekspanderer Connection finnes. Fasett legger seg på som
  en ny gren.
- Modellen som bærer Connection-begrepet finnes. Den får ett nytt felt
  for fasetter.
- Valideringen finnes. Ulovlige kombinasjoner avvises ved build-tid.
- Skalar-fasettypene gjenbrukes på tvers av hele skjemaet — én
  =StringFacetValue=, ikke én per Connection.

* Hele "intelligensen" i en record

#+BEGIN_SRC java
public record FacetSpec(
    String inputFieldName,    // "rating"
    String columnName,        // "RATING"
    String valueTypeName,     // "MpaaRating"
    String facetValueTypeName // "MpaaRatingFacetValue"
) {}
#+END_SRC

Hele kontrakten lever i den lille recorden.

Ingen tekst-rewrite, ingen ny direktiv-parsing. Klassifikatoren har
allerede gjort jobben.

#+BEGIN_NOTES
Sammenlign med tekstmakro-tilnærmingen: vi måtte ekspandert ny SDL,
gjettet hva filter-bindingen var, mistet typeinformasjonen om
=MpaaRating= underveis, og generatoren ville aldri visst at det var en
filter-pluss-fasett-konstruksjon. Den semantiske kontrakten finnes ikke
i SDL-streng-formen.
#+END_NOTES

* Win for produsenten

#+ATTR_REVEAL: :frag (appear)
- *Korte skjemaer.* Én annotasjon der det før var 25 linjer.
- *Verktøyet vet hva en Connection /betyr/.* Paginering, cursor-encoding
  og totalCount implementeres én gang, korrekt, av rammeverket.
- *Build-tid fanger feil.* Ulovlige kombinasjoner avvises før kompilering
  er ferdig — ingen overraskelser i prod.
- *Domenet kan vokse.* Connection. Edge. Facet. Neste begrep har en
  plass å sette seg.

* Win for konsumenten

#+ATTR_REVEAL: :frag (appear)
- *Spec-konformt Relay.* Klienter virker ut av boksen og kan ikke gli ut
  av spec.
- *Typer i begge retninger.* En fasettverdi kan fôres rett tilbake i
  filteret — samme skalar, ingen coercion.
- *Filter-minus-self er kontrakt, ikke håp.* Endrer du selektoren på en
  fasett, får du nye tellinger; alle andre er stabile.
- *Ingen lekket implementasjon.* Du ser et vanlig GraphQL-skjema, aldri
  en makro.

* Avslutning

SDL-makroer er ikke pre-prosessor-tekst.

De er /navn/ på domenebegreper rammeverket vårt forstår.

#+ATTR_REVEAL: :frag (appear)
- *Connection.* Navigasjon.
- *Edge.* Posisjon.
- *PageInfo.* Kontroller.
- *Facet.* Feedback.

#+ATTR_REVEAL: :frag appear
Hvert begrep implementeres /én gang/, korrekt, av rammeverket.

#+ATTR_REVEAL: :frag appear
Vi snakker om domenet vårt med presise navn — ikke om SDL-strenger.

#+ATTR_REVEAL: :frag appear
*Det er den ekte usability-vinningen.*
