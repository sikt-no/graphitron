# Regenerate and guard the generated supported-directives migration doc against directive-set drift

`docs/manual/_generated/supported-directives.adoc` is a generated fragment (included by
`docs/manual/how-to/migrating-from-legacy.adoc`) listing every directive the rewrite declares, the
legacy-only directives, and the argument-shape changes between the two. It is produced by
`graphitron-roadmap-tool directive-support --mode=migration`
(`DirectiveSupportReport.renderMigration`). The on-disk file has silently drifted from what the
generator would emit today: it is missing `@routine` (added to `directives.graphqls` by R300) and
`@scalarType`, and `@sourceRow` is mis-ordered (it sits second, though the generator sorts the
`Supported directives` list alphabetically). So the migration guide under-reports the supported
directive set, and a reader migrating off legacy is told `@routine` / `@scalarType` do not exist in
the rewrite.

The root cause is process, not content: the file's own header says "Regenerate via the verify-mode
CI guard," but **no such guard exists** — there is no workflow step, pom execution, or script in
the repo that regenerates this file or diffs it against the committed copy (confirmed by grepping
`.github/workflows`, the docs/rewrite poms, and the repo for `directive-support` /
`supported-directives`). It is regenerated by hand with ad-hoc CLI args, so it drifts every time a
directive is added or removed and nothing fails. The `directive-support` CLI
(`Main.java:78`, `DirectiveSupportReport.run`) takes three positional args (legacy
`directives.graphqls`, rewrite `directives.graphqls`, colon-separated fixture dirs) plus
`--mode=migration --output=<path>`; none of these are pinned anywhere, which is why a careless
regeneration risks churning the `Legacy-only directives` and `Argument-shape changes` sections
(those depend on the legacy file).

Two facts about the current generator shape constrain the fix:

- **The legacy file lives outside the rewrite tree.** The legacy directive set the report diffs
  against is `graphitron-common/src/main/resources/directives.graphqls` (25 directives, the only file
  carrying `@experimental_procedureCall`, the lone `Legacy-only` entry today). `graphitron-common` is
  a legacy module, not in the rewrite reactor. A guard that reads it during the rewrite's `verify`
  phase would break the "Rewrite builds independently of legacy Graphitron modules" principle for any
  consumer who has only the rewrite tree on disk. (Note `scripts/verify-standalone-build.sh` would
  *not* catch this: it isolates only the Maven local repo, not the filesystem, so the legacy source
  file is still present when that check runs. The regression would surface only for a real
  rewrite-only consumer.) The fix below snapshots the legacy set into the rewrite tree to break the
  coupling.
- **In `--mode=migration` the fixture-dir positional is vestigial.** `renderMigration`
  (`DirectiveSupportReport.java:510`) gates the `Supported directives` list purely on
  `rewriteByName.containsKey` (line 533) and renders the `Argument-shape changes` rows from
  `name` + `argDiff` only (lines 568-573); the computed `fixtureUses` set feeds only `Row.inFixtures`,
  which the migration renderer never emits. The CLI still *requires* the positional (`run` rejects
  fewer than three args), but it does not influence the migration fragment. This bears on both the
  pin (one less moving input) and the prose-vs-filter decision below.

## Scope

1. **Snapshot the legacy directive set into the rewrite tree, and pin the invocation.** Copy
   `graphitron-common/src/main/resources/directives.graphqls` to a committed resource inside the
   rewrite tree (`graphitron-rewrite/roadmap-tool/src/main/resources/legacy-directives.graphqls`),
   so regeneration and the verify guard read only rewrite-tree files. This both honours the
   standalone-build principle (no read outside the rewrite tree on the `verify` path) and makes the
   invocation self-contained; the legacy directive set is frozen (legacy is in maintenance), so the
   snapshot is low-maintenance. Pin the full command in one place: the phase-bound `exec-maven-plugin`
   execution added to `roadmap-tool/pom.xml` (see decision 1 below), with `commandlineArgs` naming the
   snapshot, the rewrite `directives.graphqls`
   (`graphitron/src/main/resources/no/sikt/graphitron/rewrite/schema/directives.graphqls`), `--mode=migration`,
   and `--output=../../docs/manual/_generated/supported-directives.adoc`. The `roadmap-README`
   regeneration and the `check-adoc-tables` execution already in that pom are the in-repo precedent for
   a pinned, phase-bound regeneration command.
2. **Add the verify-mode guard the header promises, mirroring `leaf-coverage`.** Give
   `directive-support` a `--verify` flag with the same contract `leaf-coverage` already exposes
   (`LeafCoverageReport.run`, lines 94/143-149): regenerate in-memory and compare against the committed
   `--output` file, exit non-zero with a "regenerate with: <command>" diagnostic on any difference.
   Wire it as a phase-bound `verify` execution in `roadmap-tool/pom.xml` alongside `verify-roadmap-readme`
   and `check-adoc-tables`, so it runs on every `mvn verify` (PRs included; see decision 1 for why the
   verify phase, not a trunk-only CI step, is the correct seam). The two report tools (`directive-support`,
   `leaf-coverage`) then expose one consistent verify surface.
3. **Regenerate the file once, and correct the generated prose.** Run the pinned command to land the
   current directive set (adds `@routine` and `@scalarType`, re-sorts `@sourceRow` into alphabetical
   position) and, in the same pass, fix the `renderMigration` prose so the guard pins a true sentence
   (see decision 2). The regenerate-once output and the source change to `renderMigration` land
   together so the committed fragment matches what the guarded command emits.

## Resolved design decisions

1. **Where the guard lives: the `roadmap-tool` `verify` phase, not a trunk-only CI step.** The guard
   must catch drift on the PR that adds or removes a directive, not after it merges. `@routine` drifted
   precisely because nothing failed when R300 added it. The existing `verify-roadmap-readme` and
   `check-adoc-tables` executions are phase-bound to `verify` and run on every `mvn verify` (the
   `build` job runs `mvn ... verify -Plocal-db`, which covers PRs); a trunk-only step like the current
   `Regenerate leaf-coverage report` would miss PR drift entirely. So the new guard is a phase-bound
   `verify` execution in `roadmap-tool/pom.xml`. Note this reuses the *hosting seam*, not a ready-made
   diff: `verify-roadmap-readme` runs the `verify` subcommand, which diffs only the rolled-up README
   against front-matter, so a new diff path (the `--verify` flag from Scope #2) is genuinely new code.
   The snapshot from Scope #1 is what makes this seam viable: without it the verify phase would read a
   legacy-module file and break standalone-build for rewrite-only consumers.
2. **Prose-vs-filter mismatch: correct the prose.** The generated `Supported directives` paragraph
   claims each listed directive is "exercised by at least one execution-tier or pipeline-tier test
   fixture," but `renderMigration` gates the list purely on `rewriteByName.containsKey` (line 533).
   Gating the list on `inFixtures` instead would require new plumbing (migration mode never renders
   `inFixtures` today; see the intro), and it would conflate this fragment's job, the legacy→rewrite
   parity surface, with coverage reporting, which `leaf-coverage` already owns. So correct the prose to
   state the actual criterion (declared by the rewrite generator), and drop the unbacked "exercised by
   a fixture" claim. This is the per-leaf application of "Documentation names only live tests/code":
   the generated sentence asserted an invariant (every listed directive is fixture-exercised) that
   nothing pins; the fix is to stop asserting it rather than to wire a check the fragment does not need.
   Landing this in the same regenerate-once pass (Scope #3) keeps the guard from pinning a sentence the
   team has already agreed is wrong.
3. **Fixture-dir positional: leave the CLI shape alone.** Since the positional is required by `run`
   but vestigial in migration mode (see intro), the pinned invocation passes the existing fixtures
   root for a valid, stable value; relaxing the CLI to make it optional is deliberately *not* done
   here, to keep the change to `directive-support` limited to the additive `--verify` flag and avoid
   churning the internal-report path that still consumes fixtures.

## Tests

This is a build-infrastructure change, not a generator-pipeline change, so the design-principles test
tiers (unit / pipeline / compilation / execution) do not map directly; the verify guard itself is the
primary regression net. Coverage:

- **The guard is its own regression test.** Once wired, the phase-bound `verify` execution fails
  `mvn verify` on any drift between the committed fragment and the regenerated content. A directive
  added to the rewrite `directives.graphqls` without regenerating the fragment fails the build, which
  is exactly the failure mode that let `@routine` drift. No separate end-to-end test is needed beyond
  confirming the guard fails on a deliberate edit during implementation.
- **Unit test for `--verify`** in `DirectiveSupportReportTest` (the existing home for
  `directive-support` unit coverage): one case asserting `run(... --verify)` returns 0 when the
  on-disk output matches the regenerated content, one asserting non-zero on a mismatch. Mirror the
  `--verify` shape `LeafCoverageReportTest` establishes for the sibling tool.
- **Prose-correction assertion.** Extend `DirectiveSupportReportTest` to assert `renderMigration`
  output no longer contains the "exercised by ... test fixture" claim and does contain the corrected
  criterion, so the prose decision (decision 2) is pinned and cannot silently regress.
- No execution-tier or `graphitron-sakila-example` compilation coverage applies; nothing in the
  generated Java output changes.

## Out of scope

- The directive *reference* index wiring for `@routine` (already fixed on trunk at `0ed125e`); this
  item is only the `_generated/` migration fragment and its regeneration guard.
- Any change to the migration guide's hand-authored prose beyond the one generated paragraph named in
  decision 2.
- Relaxing the `directive-support` CLI to make the fixture-dir positional optional in migration mode
  (decision 3); the change to the tool is limited to the additive `--verify` flag.
- `supported-schema-shapes.adoc` (the sibling generated fragment). It is generated by
  `leaf-coverage --mode=migration` and has the **identical** unguarded gap and false "verify-mode CI
  guard" header; nothing runs `leaf-coverage --mode=migration` in CI, poms, or scripts (the
  `Regenerate leaf-coverage report` step runs `leaf-coverage graphitron-rewrite` with no
  `--mode=migration`, producing the internal `inference-axis-coverage.adoc`, not the migration
  fragment). It is deferred to a follow-up (filed as a Backlog item) rather than folded in here,
  because its guard has a *different shape*: regenerating it depends on build-time
  `leaf-coverage.jsonl` trace files, so its verify step cannot live in the simple SDL-only `verify`
  execution this item adds and must run after the test phase. The deferral is on that genuine
  shape-difference, not on a (false) claim that CI already keeps it fresh.
