# Remove convention-based scalar resolution; require @scalarType for non-spec scalars

Graphitron carries a second, implicit path for binding non-spec scalars: a hardcoded "convention table" in `ScalarTypeResolver` (`ScalarTypeResolver.java:433-481`) mapping bare SDL scalar names (`BigDecimal`, `UUID`, `DateTime`, ...) to `graphql.scalars.ExtendedScalars` constants. When a scalar declaration carries no `@scalarType` directive, the classifier falls back to this table (`TypeBuilder.java:1244-1251`), resolves the constant against the consumer's codegen classpath, and recovers the Java type from its `Coercing`. This item removes that fallback: `@scalarType(scalar: "fully.qualified.Class.FIELD")` becomes the single, explicit way to bind any non-spec, non-federation scalar. The two implicit paths that are *not* convention (the five GraphQL spec built-ins and the federation-namespace scalars) are untouched.

## Why remove it

The convention layer is not worth its carrying cost. Graphitron deliberately does **not** ship graphql-java-extended-scalars to consumers (it is `test` scope only; `graphitron/pom.xml:66-69` states "the runtime artifact is the consumer's choice and graphitron stays out of it"), so the fallback only fires for a consumer who has *already explicitly* added the library to their own classpath, exactly the consumer for whom writing one `@scalarType` line is trivial. In exchange for that marginal, opt-in-conditional convenience we pay: (a) recurring curation, every extended-scalars bump adds constants and trips the drift test, forcing a per-scalar map-or-exclude decision (the bump to extended-scalars 24.0 surfaced six: `YearMonth`, `Year`, `AccurateDuration`, `NominalDuration`, `SecondsSinceEpoch`, `HexColorCode`); and (b) a genuine footgun, because resolution keys off classpath presence, a consumer who pulls extended-scalars in *transitively* would have a bare `scalar BigDecimal` silently start resolving with no directive and no intent, so the same SDL classifies differently depending on the dependency graph. Requiring `@scalarType` everywhere collapses scalar binding to one explicit path, deletes the classpath-dependent surprise, and removes the drift test as a gate on future graphql-java / extended-scalars upgrades (it tripped during the graphql-java version-upgrade dry run and motivated this item, and blocks R465). This is a pre-1.0 breaking change: the one bare `scalar BigDecimal` in the sakila-example schema must gain an explicit `@scalarType`, as must any consumer relying on directive-less extended-scalar names; the fix is mechanical and the actionable error message already points at it.

## Implementation

Generator side is pure deletion: the classification ladder loses its convention rung and `@scalarType` becomes the single explicit binding path. LSP side *replaces* the convention-table completion with a better one: instead of a hardcoded 30-entry list keyed off graphql-java-extended-scalars, `@scalarType(scalar: |)` now completes from the `public static final GraphQLScalarType` constants actually present on the consumer's codegen classpath, found by the classpath scanner that already walks it. That finds the consumer's *own* scalar constants (`com.example.Scalars.MONEY`), not just extended-scalars, and drops the completion's coupling to that library entirely, the same "stay out of the consumer's runtime-artifact choice" stance that motivates removing the resolution fallback. Removing the resolver's table while upgrading the completion lands atomically here so editor support never regresses.

* **`graphitron/src/main/java/no/sikt/graphitron/rewrite/ScalarTypeResolver.java`.** Delete `CONVENTION_TABLE` (the static map + its initializer block, `~433-481`), the `conventionTable()` accessor (`~500-502`), and `resolveByConvention(String, ClassLoader)` (`~523-533`). Everything else stays: `resolveBuiltIn`, the federation-namespace path, `resolveFromConstantFqn` (both overloads), `resolveFromDirectiveValue`, `parseDirectiveValue`, `recoverInputType`, `coercionOutputType`, `isClassifiedScalarJavaType`. Scrub *every* surviving `convention` / `Phase 3` mention from the javadoc, not a fixed count: grep the file. The live ones after deletion are the class-level "Phase 3 wires it to the extended-scalars convention layer", `resolveBuiltIn`'s "consumer-declared / convention paths", `resolveFromConstantFqn`'s "Phase 3's convention layer" references, and `isClassifiedScalarJavaType`'s "via `@scalarType` or the extended-scalars convention layer" (`~617`). `coercionOutputType` / `isClassifiedScalarJavaType` read resolved Java types out of `classifiedTypes` at runtime, never the table, so their bodies are untouched and only the prose changes.
* **`graphitron/src/main/java/no/sikt/graphitron/rewrite/model/ScalarResolution.java`.** Javadoc-only. `ScalarResolution.Synthesised`'s doc (`~37-42`) uses the convention-layer `GraphQLBigDecimal` alias as one of its two worked examples of the alias case. After removal only aliasing `@scalarType` and the federation namespace reach `Synthesised`; drop the convention example so the doc describes only live producers.
* **`graphitron-lsp/src/main/java/no/sikt/graphitron/lsp/parsing/Behavior.java`.** Javadoc-only. `Behavior.ScalarTypeBinding`'s doc (`~44-51`) says it "completes against the extended-scalars convention table (preferring the entry that matches...)". Reword to describe the new source: classpath-scanned `GraphQLScalarType` constants, preferring one whose field name matches the enclosing SDL scalar name.
* **`graphitron/src/main/java/no/sikt/graphitron/rewrite/TypeBuilder.java`.** Delete the convention fallback block (`~1244-1251`), so a directive-less non-spec/non-federation scalar falls straight through to the existing hard-error `UnclassifiedType`. Simplify the hard-error message (`~1256-1261`) to drop the "or add graphql-java-extended-scalars to the project classpath if the SDL name matches a convention-table entry" clause; the message now names `@scalarType(scalar: "fully.qualified.Class.FIELD")` as the single fix.
* **`graphitron/src/main/java/no/sikt/graphitron/rewrite/catalog/ClasspathScanner.java`.** Add a `readScalarConstants(ClassModel)` pass alongside the existing `readMethods` / `readRecordComponents`, reading `public static` fields whose JVM field descriptor is exactly `Lgraphql/schema/GraphQLScalarType;`. This is the same exact-descriptor-match idiom the scanner already uses for the jOOQ `Condition` classification (`JOOQ_CONDITION_DESCRIPTOR`, `~50-51,166`): a parse-only scan resolves no type hierarchy, and the field is declared as `GraphQLScalarType` directly, so exact match is both sufficient and all the scanner can do. Fields are read off `cm.fields()`, checking `AccessFlag.PUBLIC` + `AccessFlag.STATIC`. Thread the captured field names through `readIfCandidate` (`~96-125`) into the `ExternalReference` it builds (`~124`). Note the scan sees the field *type*, not its runtime value or `Coercing` generics: it offers a candidate FQN; the reflective resolver and diagnostics remain the source of truth and still reject a null-at-codegen or `Coercing<Object,Object>` constant at build time. This is the same contract method completion already lives under (it suggests methods that may not typecheck), not a new compromise.
* **`graphitron/src/main/java/no/sikt/graphitron/rewrite/catalog/CompletionData.java`.** Add a `List<ScalarConstant> scalarConstants` slot to the `ExternalReference` record (`~157-168`), defensively copied in its compact constructor like `methods` / `recordComponents`, and a `public record ScalarConstant(String fieldName)` next to `RecordComponent` (`~176`) with a javadoc pointing at `ClasspathScanner` as its source. The owning class FQN is `ExternalReference.className()`, so the completion composes `className + "." + fieldName`; no need to repeat the FQN on the component, matching the `RecordComponent` / `Method` shape. Existing `ExternalReference` construction sites (only `ClasspathScanner:124` in main; test builders) pass the new list; add a back-compat constructor defaulting it to `List.of()` if that keeps test callers compiling, mirroring the `Method` back-compat constructor (`~204-209`).
* **`graphitron-lsp/src/main/java/no/sikt/graphitron/lsp/completions/ScalarTypeCompletions.java`.** Rewrite the body (keep the file, keep the `Completions.java` dispatch arm at `~92-94` unchanged). Replace the `ScalarTypeResolver.conventionTable()` read (`~54-60`) with a walk over `data.externalReferences()`, emitting a `className + "." + fieldName` completion item for each `ScalarConstant`. Preserve the name-preference behaviour: when the enclosing `scalar X` name matches a scanned constant's field name (case-insensitive, so `scalar UUID` prefers `...ExtendedScalars.UUID`), offer that one first. Rewrite the class javadoc: the old "we do not enumerate static `GraphQLScalarType` fields off the classpath ... `CompletionData` carries methods but not field metadata" note (`~24-32`) is now exactly what we *do*, via the new `ScalarConstant` slot. **Keep** the `Behavior.ScalarTypeBinding` variant and its diagnostics untouched: an authored `@scalarType(scalar: "...")` value still needs its malformed-reference / unknown-class diagnostics, which live in `Diagnostics`, not here.
* **`graphitron-sakila-example/src/main/resources/graphql/schema.graphqls`.** The one bare `scalar BigDecimal` (`:12`, referenced by `input RentalRateRange` at `:1822`) becomes `scalar BigDecimal @scalarType(scalar: "graphql.scalars.ExtendedScalars.GraphQLBigDecimal")`. Update the `:10-11` comment that says "resolved via the extended-scalars convention layer". The existing `scalar LocalDate @scalarType(...)` alias (`:8`) already uses the directive and is unaffected; it is now the *only* remaining scalar-binding style in the fixture, which is the point (the pipeline verifies the `@scalarType` path end-to-end, and the convention path no longer has a fixture because it no longer exists).

## Tests

* **`graphitron/src/test/java/no/sikt/graphitron/rewrite/ScalarTypeResolverTest.java`.** Delete the "Phase 3: convention layer" block (`conventionTable_recognisesBigDecimal`, `conventionTable_rejectsUnknownName`, `resolveByConvention_bigDecimal_*`, `resolveByConvention_aliasedName_*`, `resolveByConvention_uuid_*`, `resolveByConvention_nameNotInTable_*`) and the "convention-table drift signal" block (`conventionTable_coversEveryExtendedScalarsField`, `conventionTable_everyAuthoredFqnResolvesAgainstExtendedScalars`). Keep every built-in and `@scalarType`-directive test. Removing the drift test is a stated goal, not collateral: it is the gate that trips on extended-scalars bumps.
* **Add one classification test** asserting that a directive-less non-spec, non-federation scalar now classifies as `UnclassifiedType` with the `@scalarType`-pointing structural rejection (previously it could resolve via convention when extended-scalars was on the classpath). This pins the new single-path contract so a future re-add of a fallback fails loudly. Place it next to the existing `TypeBuilder`/classification tests that exercise the ladder.
* **Scanner test.** Add a `ClasspathScanner` (or `CatalogBuilder` snapshot) test asserting a class exposing a `public static final GraphQLScalarType` field surfaces as a `ScalarConstant` on its `ExternalReference`, and that a non-`GraphQLScalarType` static field does not. `graphql.scalars.ExtendedScalars` is already on the test classpath and exposes many such fields, but the scanner walks a *classes directory*, so prefer a small hand-authored fixture class compiled into the test's scan root over pointing at the library jar (matches how the existing method / record-component scanner tests are structured).
* **LSP completion tests.** Rewrite `graphitron-lsp/.../ScalarTypeCompletionsTest.java` rather than deleting it: the three convention-table assertions (`scalarMatchingConventionEntryPrefersExtendedScalarsFqn`, `scalarMatchingPrefixedFormPrefersSameConstant`, `unknownScalarNameStillSuggestsConventionTable`) become tests that build a `CompletionData` carrying an `ExternalReference` with a couple of `ScalarConstant`s and assert (a) the `className.fieldName` items are offered, (b) the field-name match for the enclosing `scalar X` is offered first, (c) `cursorOnDifferentDirectiveReturnsEmpty` still holds. Note the source of truth moves from the static `conventionTable()` (which is why the old tests worked off `CompletionData.empty()`) to `data`, so the rewritten tests must populate `data`. Update the `ScalarTypeCompletions` arm of `CompletionTextEditTest.java` (`~157-160`) the same way: feed a populated `CompletionData` instead of `CompletionData.empty()` and assert the text-edit range against a scanned constant FQN.
* **Pipeline (`graphitron-sakila-example`).** The compile+execute pipeline over the edited schema is the real regression guard for the sakila `BigDecimal` retarget: it must still compile the generated `RentalRateRange` record with a `java.math.BigDecimal` component and execute the range filter. No new pipeline test needed; the existing fixture coverage exercises it once the schema edit lands.

## User documentation (first-client check)

The convention shortcut is a documented, user-visible surface, so the docs are the first client of its removal. If the "explicit `@scalarType` only" story does not read simply, the design is wrong.

* **`docs/manual/how-to/custom-scalars.adoc`.** Delete the "Skip the directive when extended-scalars covers the shape" section (`~31-52`) outright. Rewrite the intro paragraph (`~7`) to drop "the convention-table shortcut that lets you omit the directive entirely". In "Resolution order and failure modes" (`~86-96`), collapse the three-layer list to two implicit paths + the directive: (1) spec built-ins, (2) federation-namespaced scalars (auto-wired via `@link`), (3) `@scalarType(scalar:)` for everything else; drop the convention layer as layer 3. Rewrite "Editor support" (`~104-106`): the LSP still completes `@scalarType(scalar: |)`, but now from `GraphQLScalarType` constants scanned off the codegen classpath (the consumer's own and any library's), preferring one whose field name matches the SDL scalar name; diagnostics on the authored value stay. Fix the constraints line (`~111`) that says "or the convention-table fallback".
* **`docs/manual/reference/directives/scalarType.adoc`.** Drop resolution-order layer 3 (`~42`). Rewrite "Editor support" (`~57-59`) the same way as the how-to: completion is now classpath-scanned constants, not convention-table FQNs. The canonical example (`~24-34`) already shows the explicit `@scalarType` form and needs no change.
* **`docs/architecture/reference/code-generation-triggers.adoc`.** Delete the convention-table classification row (`~182`) and fold its "no directive and no convention-table entry → `UnclassifiedType`" successor row (`~184`) into a single "non-spec/non-federation `scalar` with no `@scalarType` → `UnclassifiedType`" row.

This is a contributor + user doc sweep, not a code-generation trigger change; the `docs/` user-facing-doc check at `In Review → Done` applies (no roadmap-internal markers leaking into `docs/manual/`).

## Deferred alternative (not chosen)

Keep the zero-config convenience but drop the hardcoded allow-list, resolving *any* `ExtendedScalars` constant reflectively (indexing by field name and `getName()` to preserve the `BigDecimal`/`GraphQLBigDecimal` dual spelling) so new library scalars flow through automatically without curation. This removes cost (a) but not footgun (b): resolution still keys off classpath presence, so a transitive extended-scalars pull still changes classification silently. Revisit only if consumers actually ask for directive-free scalar binding, considered unlikely.
