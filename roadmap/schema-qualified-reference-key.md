# Schema-qualify @reference(key:) to escape cross-schema FK-name collisions

A `@reference(path: [{key: "<fk>"}])` foreign-key name is resolved by `JooqCatalog.findForeignKey(String name, String sourceSqlName)` (`JooqCatalog.java`) across every schema in one flat namespace: first the SQL constraint name (`fk.getName().equalsIgnoreCase(name)`), then, only on a miss, the jOOQ `TABLE__CONSTRAINT` Java-constant form via each schema's generated `Keys` class. The method takes a nullable `sourceSqlName` and, when that resolves to a single catalog table, scopes candidates to the FKs touching it (identity-based). When the same constraint name exists in two or more schemas and source scoping cannot break the tie (the source is `null`, non-table-backed, or itself resolves ambiguously), the lookup returns `ForeignKeyLookup.Ambiguous` and the author sees the "ambiguous" rejection from `BuildContext.ambiguousForeignKeyRejection`.

Unlike `@table(name:)`, which accepts a schema-qualified `schema.table` form (`JooqCatalog.parseQualifiedTableName`, the lever the large audited schema-qualified-`@table` bug class settled on), the `key:` grammar has **no** schema-qualified form. So a cross-schema constraint-name collision is not merely ambiguous, it is unfixable by the author: there is no lever to disambiguate. This bites on real multi-schema databases (e.g. Samordna opptak) whose generated schemas carry colliding constraint names; the reactor's own multi-schema fixture already reproduces it (`multischema_a.note` and `multischema_b.note` both hold an FK literally named `note_event_fk`).

The fix is to let a `key:` value carry a leading `schema.` qualifier (`key: "multischema_a.note_event_fk"`) that scopes the FK candidate set to the FK-holder table's schema, symmetric to the `@table` precedent, leaving unqualified behaviour byte-for-byte unchanged.

## Design

**Why a schema-name filter is not the retired bare-name anti-pattern.** The schema-qualified-`@table` bug class retired matching FK *endpoints* and table *directive echoes* by bare SQL name, because table names collide across schemas and a bare-name compare silently picked the first-hit wrong schema. A *schema* qualifier is different in kind: schema names are unique in the catalog by construction (there is no "two schemas with the same name"), and a constraint name is unique *within* a schema. So `(schemaName, constraintName)` identifies exactly one FK. Once `findForeignKey` narrows to that one jOOQ `ForeignKey`, everything downstream (`findForeignKeyRef`, `synthesizeFkJoin`) is already reference-/class-identity based. The qualifier only narrows the candidate set; it never re-introduces a bare-name compare on a colliding namespace.

**Parsing: a typed carrier at the boundary, not a split buried in the lookup.** The `@table` precedent did not merely adopt the dot-grammar; it parsed the value *once* into a typed `QualifiedTableName(Optional<String> schema, String table)` and dispatched to a schema-*scoped* two-arg `findTable(schema, table)` that never re-parses and never scans across schemas. This Spec copies that shape, not just the grammar: add a sibling `QualifiedForeignKeyName(Optional<String> schema, String name)` record and a `parseQualifiedForeignKeyName(String)` parser (split on the first `.`; an empty half parses as unqualified, defensive, mirroring the table parser). A `.` cannot appear in a jOOQ constraint name and the Java-constant namespace uses `__`, so a dot unambiguously signals a schema qualifier in both namespaces. A distinct record (not reusing `QualifiedTableName`) documents intent and keeps the two grammars independently evolvable.

The reason a distinct carrier matters, not just style: burying the dot-split inside `findForeignKey(String name, …)` would silently widen that method's `name` contract from "an FK name" to "an optionally-qualified FK name" for **all** callers, including the two that never carry an author qualifier (the `IdReference` synthesis shim, which passes a real catalog constraint name from `buildQualifierMap`; and `qualifierForFk`, which passes an already-resolved name). It would happen to be safe only because FK names can't contain dots, a coincidence carrying the contract rather than the type.

**Lookup shape.** Parse the qualifier once, at the author-string entry points only (the `{key:}` branch of `parsePathElement` and the explicit record-FK site `resolveRecordFkTargetColumns`). Give `findForeignKey` an explicit schema-scope parameter (or a schema-scoped overload) that the parsed `QualifiedForeignKeyName.schema()` feeds; internal callers pass no schema and are visibly exempt from the grammar. Inside, candidates match the *bare* constraint name in both namespaces (unchanged), then the schema scope, when present, filters to FKs whose **holder schema** (`fk.getTable().getSchema().getName()`, matched case-insensitively exactly as the two-arg `findTable` matches its schema half) equals it. Outcomes:
- a qualified key that collides across schemas resolves to the one FK in the named schema (`Resolved`);
- a qualified key naming a schema with no such FK yields `NotInCatalog` (not a silent first-hit);
- unqualified keys skip the filter entirely, byte-for-byte unchanged.

**Scoping precedence: qualifier hard, source-scope soft.** The two facts have different provenance. Source-scoping is *derived context* ("I'm standing on table T") and is deliberately soft today (`if (!scoped.isEmpty()) candidates = scoped;`, narrows only when it leaves something). An author `schema.` qualifier is *stated intent* and is hard: if it eliminates every candidate, that is `NotInCatalog` for that schema, not a fallback. Apply the qualifier first (hard), then source-scope (soft) over the remainder. A qualified FK that resolves but does not touch the source then falls out naturally: it resolves to one FK, survives the soft source-scope, and hits the existing connection validator in `parsePathElement` (`foreignKeyTouchesTable`, the "key '…' does not connect to table '…'" arm).

**Enforcer symmetry across the author-facing sites.** Today the "does not connect" check lives only in the `{key:}` branch of `parsePathElement`; the explicit record-FK path (`resolveRecordFkTargetColumns`) has no equivalent touches-check and relies on later column reconciliation. This is safe *only* because a qualifier is not author-supplyable today, so a wrong-schema-but-real FK cannot reach these sites. Once `key:` can name an FK in a different schema than the record/source table, "qualified FK resolves but doesn't touch this table" becomes reachable at *both* sites, and asymmetric enforcement would make one reject cleanly while the other degrades to a worse column-reconciliation-miss message. The Spec lifts the touches-check so it fires uniformly at both author-facing key sites, ideally as part of the same schema-scoped lookup shape above, so the invariant has one enforcer rather than two half-copies.

**LSP mirror re-sources from the shared parser.** `Diagnostics.validateCatalogFk` matches the `key:` value against a flat list of bare FK names case-insensitively, and its own comment states it must not flag a name the generator would accept. A qualified `key:` (`opptak_samordna.some_fk`) equals no bare name, so it would red-squiggle a *valid* key. The mirror must split the qualifier the *same* way the generator does. Because the split now lives in the shared `parseQualifiedForeignKeyName` parse-boundary helper, the LSP re-sources from it rather than re-implementing the grammar (the "one model, many views" discipline: a view that re-derives the grammar drifts). The mandatory correctness fix is to strip the qualifier before the bare-name match, so a valid qualified `key:` is no longer red-squiggled: this alone satisfies the "must not flag a name the generator would accept" invariant, and it is genuinely cheap (the shared parser plus a bare-name compare on the stripped half).

Flagging a *bogus* schema (a qualifier whose `(schema, name)` pair is absent in the named schema, mirroring the generator's `NotInCatalog`) is **not** free and must not be filed under the strip fix: the LSP snapshot carries no per-FK schema. `CompletionData.Reference` holds only `targetTable`, `keyName`, `inverse`, and `keysClassFqn`, and `CompletionData.Table` holds a bare, unqualified table `name`, so there is no `(schema, fk)` pair to test against. The schema is recoverable only *approximately*, by parsing the package segment out of `keysClassFqn` (`…multischema_a.Keys`), and that segment is a jOOQ-naming-strategy transform of the SQL schema name, not the name itself. So the bogus-schema arm is either a `CatalogBuilder`/`CompletionData` snapshot extension that threads the FK holder schema through, or it stays deferred. This item picks one explicitly rather than assuming it falls out; the recommendation is to **defer** the bogus-schema arm (keeping this item's LSP change to the strip fix, which is the actual regression the qualifier introduces) and file the snapshot-schema extension as its own Backlog stub if wanted. Path-step refinement stays deferred, unchanged.

**Which endpoint the qualifier binds.** A cross-schema FK has endpoints in two schemas; the qualifier binds the **holder (child/referencing) side**, which is where jOOQ's generated `Keys` class declares the constraint. The docstring must state this so the author is not left guessing. A typo in the *schema* half currently collapses into the generic "could not be resolved in the jOOQ catalog" message, the same collapse the `@table` qualified path already accepts (`findTable(schema, table)` returns empty for either half), so this is a trade-off the precedent settled, not a new violation. Optional cheap win: since `Ambiguous` already carries the colliding schema names as typed data, a message that distinguishes "no FK 'X' in schema 'Y'" from "schema 'Y' is not in the catalog" is a small typed-rejection parity improvement; flagged as optional in implementation.

**Docs.** Extend the `ReferenceElement.key` doc string in `directives.graphqls` and the `@reference` / `@referenceFor` reference pages (`docs/manual/reference/directives/`) to document the `schema.constraint` form and when it is needed (cross-schema constraint-name collision), pointing at the `@table` qualified form as the sibling.

## Tests

- **Unit tier (the parse boundary + the resolution core):** add `parseQualifiedForeignKeyName` cases mirroring the `parseQualifiedTableName` battery (unqualified, qualified, multi-dot, blank/empty-half malformed); these pin the typed carrier at the boundary. Then extend `JooqCatalogMultiSchemaTest`, whose `note_event_fk` collision is the vehicle. Assert: (1) the unqualified lookup is `Ambiguous` naming both schemas (pins the pre-fix hazard as the baseline); (2) the qualified lookup resolves `multischema_a.note_event_fk` to the A-schema FK and `multischema_b.note_event_fk` to the B-schema FK, disambiguating the name collision; (3) a qualifier naming a schema with no such FK is `NotInCatalog`; (4) unqualified unique keys still resolve unchanged; (5) the qualifier-hard / source-scope-soft ordering: a qualified key whose FK does not touch a supplied source still resolves (`Resolved`), leaving the connection verdict to the caller. Note that `note_event_fk` cannot on its own pin *holder*-vs-*FK-target* schema binding: it is intra-schema in each schema (`multischema_a.note` → `multischema_a.event`), so holder and target coincide and a target-schema filter would pass the same assertions. The distinguishing case is the fixture's genuinely cross-schema FK, the one held on `multischema_b.gadget` referencing `multischema_a.widget`: assert (6) that qualifying it with its **holder** schema (`multischema_b`) resolves while qualifying it with its FK-*target* schema (`multischema_a`) is `NotInCatalog`. This is the only topology in the fixture where holder ≠ target, so it is the test that actually falsifies a target-schema implementation of the filter.
- **Pipeline tier (both author-facing key sites):** a `@reference(path: [{key: "multischema_a.note_event_fk"}])` fixture where the bare name would be ambiguous asserts the reference resolves and lands the join on the A-schema FK; a sibling with the wrong-schema qualifier asserts the "does not connect" rejection. Because the enforcer-symmetry change lifts the touches-check to the record-FK site too, add the parallel pair on an explicit record-FK `@reference(key:)` input (through `InputBeanResolver`), asserting the same resolve / wrong-schema-rejection outcomes there; this is the test that pins the invariant now has one uniform enforcer rather than one live arm and one gap.
- **LSP tier:** extend `DiagnosticsTest` so a qualified `key:` naming a real FK (`multischema_a.note_event_fk`) is *not* flagged (the regression the strip fix removes), the mirror re-sourcing the split from the shared `parseQualifiedForeignKeyName` rather than re-implementing the grammar. The bogus-schema-*is*-flagged assertion belongs with the bogus-schema arm and lands only if that arm is taken (see the LSP paragraph in Design); with the arm deferred, this tier asserts only the no-false-flag behaviour.
