= How-to guides
:description: Recipes for the day-to-day patterns Graphitron's directives unlock.
:!toc:

Task-indexed recipes for the day-to-day patterns Graphitron's directives unlock: mapping types to tables, joining via references, batch lookups, custom conditions, services, mutations, pagination and sorting, polymorphic types, error handling, global node identification, computed fields, Apollo federation, custom scalars, tenant scoping, and the recommended consumer test pattern.

== Status: in flight

Recipe pages are authored in R68 Phase 3, one at a time. Landed pages are listed below; until the rest land, the closest material is in xref:../../architecture/getting-started.adoc[Getting started with the rewrite runtime] (federation, custom scalars, tenant scoping, hello world) and the worked examples in the legacy https://github.com/sikt-no/graphitron/tree/main/graphitron-codegen-parent/graphitron-java-codegen/README.md[`graphitron-java-codegen` README] (the rewrite is directive-compatible with that material; recipes will be reframed and cross-linked into source-of-truth tests as they land).

== Available recipes

* xref:map-types-to-tables.adoc[Map types to tables]: bind types and fields with `@table` + `@field` across the four directive sites (object field, argument, input field, enum value). Covers text-column vs jOOQ-typed enum mapping, input-type binding, and the table-context cascade.
* xref:external-code.adoc[Wire external Java code]: the shared mechanics for `@service`, `@externalField`, `@tableMethod`, `@condition`, `@batchKeyLifter`, `@enum`, and `@record`. Plugin classpath, FQCN requirement, method-name defaults, `argMapping:` syntax.
* xref:polymorphic-types.adoc[Polymorphic types]: choose between single-table (`@discriminate` + `@discriminator`) and multi-table (no discriminator, two-stage `UNION ALL` fetcher) layouts for GraphQL interfaces and unions. Includes polymorphic child fields and pagination contract.
* xref:global-id.adoc[Global object IDs]: Relay Global Object Identification end to end. Stable-ID strategies, `keyColumns` migrations, composite-PK nodes, cross-type IDs from FK paths, decode-side argument and input contracts, and `Query.node`/`Query.nodes` dispatch.
* xref:sort-results.adoc[Sort results]: pick a column source (`index:`, `fields:`, `primaryKey:`), mix static `@defaultOrder` with dynamic `@orderBy`, build stable keyset cursors with explicit tie-breakers, and avoid the collation and index-volatility pitfalls.
