= How-to guides
:description: Recipes for the day-to-day patterns Graphitron's directives unlock.
:!toc:

Task-indexed recipes for the day-to-day patterns Graphitron's directives unlock: mapping types to tables, joining via references, batch lookups, custom conditions, services, mutations, pagination and sorting, polymorphic types, error handling, global node identification, computed fields, Apollo federation, custom scalars, tenant scoping, and the recommended consumer test pattern.

Landed pages are listed below; until the rest land, the closest material is in xref:../../architecture/getting-started.adoc[Getting started with the rewrite runtime] (federation, custom scalars, tenant scoping, hello world) and the worked examples in the legacy https://github.com/sikt-no/graphitron/tree/main/graphitron-codegen-parent/graphitron-java-codegen/README.md[`graphitron-java-codegen` README] (the rewrite is directive-compatible with that material; recipes will be reframed and cross-linked into source-of-truth tests as they land).

== Available recipes

* xref:map-types-to-tables.adoc[Map types to tables]: bind types and fields with `@table` + `@field` across the four directive sites (object field, argument, input field, enum value). Covers text-column vs jOOQ-typed enum mapping, input-type binding, and the table-context cascade.
* xref:external-code.adoc[Wire external Java code]: the shared mechanics for `@service`, `@externalField`, `@tableMethod`, `@condition`, `@batchKeyLifter`, `@enum`, and `@record`. Plugin classpath, FQCN requirement, method-name defaults, `argMapping:` syntax.
* xref:polymorphic-types.adoc[Polymorphic types]: choose between single-table (`@discriminate` + `@discriminator`) and multi-table (no discriminator, two-stage `UNION ALL` fetcher) layouts for GraphQL interfaces and unions. Includes polymorphic child fields and pagination contract.
* xref:global-id.adoc[Global object IDs]: Relay Global Object Identification end to end. Stable-ID strategies, `keyColumns` migrations, composite-PK nodes, cross-type IDs from FK paths, decode-side argument and input contracts, and `Query.node`/`Query.nodes` dispatch.
* xref:sort-results.adoc[Sort results]: pick a column source (`index:`, `fields:`, `primaryKey:`), mix static `@defaultOrder` with dynamic `@orderBy`, build stable keyset cursors with explicit tie-breakers, and avoid the collation and index-volatility pitfalls.
* xref:connections.adoc[Cursor-paginated connections]: synthesised vs declared connection types, the one-field-one-connection rule (and why `connectionName:` is deprecated), what `totalCount` actually costs (and when it returns `null`), and what the rewrite does (and does not) cap on page size.
* xref:condition-cascade.adoc[Stacking and overriding conditions]: when explicit `@condition` predicates stack with implicit ones, what `override: true` suppresses at argument vs field level, the `@table` cascade and the divergence from legacy "outer owns everything" semantics, the input-carrier rules for nested inputs, and how `contextArguments:` flow.
* xref:batch-lookups.adoc[Batch lookups]: drive a positional batch fetch with `@lookupKey` across the single-key, correlated multi-arg, `@table` composite, NodeId-encoded, `@splitQuery`-per-parent, and `@record`-parent shapes. Includes the failure-mode split between synthesised lookup-key NodeIds (`Throw`) and same-table `@nodeId` args (`Skip`), and the implicit-predicate exemption.
* xref:split-vs-inline.adoc[When to split queries]: choose between inline correlated subqueries and `@splitQuery`-batched fetches. Covers the round-trip-vs-fan-out trade-off, the three operational cases the classifier requires split for, the `DataLoader` cache key (path-scoped, request-scoped, tenant-prefixed), and the aliased-uses-don't-share-batches consequence.
* xref:result-types.adoc[Result-type variants]: pick a backing class for a `@record` parent (jOOQ `TableRecord`, jOOQ `Record`, Java record, plain POJO) and let accessor inference batch the children when the catalog can't. Covers the classifier's decision tree, the `AccessorKeyedSingle`/`AccessorKeyedMany` auto-derivation rules, when to reach for `@batchKeyLifter`, and the end-to-end fixtures for each shape.
* xref:error-channel.adoc[The errors channel]: route Java exceptions through a service payload's `errors:` field as typed data. Covers union vs single-type carriers, the source-order + cause-chain dispatch loop, why `description:` is captured but currently unused, what the synthesised `path:` slot actually contains, and the redact-with-correlation-id fallback for unmatched exceptions.
* xref:handle-services.adoc[Handle services]: bind GraphQL arguments to Java parameters (including nested input types and request-context values), pick a return shape the framework recognises (`Result<TableRecord>`, scalar, `@record` payload, or `Map<Key, Value>`), and wire `@service` onto a child field via the `@splitQuery` `Set<Key> ŌåÆ Map<Key, Value>` contract.
