= @referenceFor
:description: Declare an explicit per-participant join path for one participant of a multi-table interface or union child field.
:!toc:

`@referenceFor` declares an explicit join path for one participant of a multi-table interface or union child field. By default the rewrite auto-discovers each participant's join as the single foreign key from the participant's table to the parent's table; `@referenceFor` replaces that discovery for the named participant only. Use it when a participant has more than one foreign key to the parent (pick one with `{key:}`), is reached through an intermediate table (chain multiple elements), correlates by a non-foreign-key predicate (`{condition:}`), or shares the parent's own table (state the self-referencing `{key:}`).

The directive is repeatable on output field definitions, one application per participant. Applications are keyed by `type` and are independent; each `path` is that participant's complete path from the parent's table. This is the opposite of repeated xref:reference.adoc[`@reference`] applications, which concatenate into one running chain. Declaring the same `type` twice is a build error. Participants you do not name keep automatic discovery (override-merge): `@referenceFor` on the ambiguous participant is enough, its siblings still auto-discover.

== SDL signature

[source,graphql]
----
directive @referenceFor(
  type: String!
  path: [ReferenceElement!]!
) repeatable on FIELD_DEFINITION

input ReferenceElement {
  table: String,
  key: String,
  condition: ExternalCodeReference
}
----

== Parameters

[cols="1,1,1,3",options="header"]
|===
| Name | Type | Default | Description
| `type` | `String!` | (required) | Name of the participant type this path applies to. Case sensitive. Must be a table-bound participant of the field's return type.
| `path` | `[ReferenceElement!]!` | (required) | Join path from the parent type's table to this participant's table, read from parent toward participant. Same element grammar as `@reference`: `{table:}`, `{key:}`, `{condition:}`.
|===

`ReferenceElement` fields are identical to xref:reference.adoc[`@reference`]'s: `table` (jOOQ table identifier for the hop's destination), `key` (foreign-key constraint name), `condition` (external join condition).

== Example

`film` has two foreign keys to `language` (`language_id` and `original_language_id`), so auto-discovery from `language` to `film` cannot pick one. `@referenceFor` disambiguates:

[source,graphql]
----
type Language @table(name: "language") {
    dubbedFilm: FilmMedia
        @referenceFor(type: "Film", path: [{key: "film_original_language_id_fkey"}])
}
----

For a self-referencing (same-table) participant, state the self FK:

[source,graphql]
----
type Category @table(name: "category") {
    kin: CategoryKin
        @referenceFor(type: "CategorySelf", path: [{key: "category_parent_category_id_fkey"}])
}
----

== Contrast with @reference

Repeated `@reference` applications concatenate into one chain in authored order; repeated `@referenceFor` applications are independent, one per participant, and each `path` is that participant's complete path. Two directives on the same location share the `ReferenceElement` payload but have opposite repetition semantics, so keep them straight:

[cols="1,2,2",options="header"]
|===
| | `@reference` (repeated) | `@referenceFor` (repeated)
| Grain | One field's whole path | One (field, participant) pair
| Repetition | Concatenates in order | Independent, keyed by `type`
| Placement | Single-table field / argument / input field | Multi-table interface/union child field only
|===

A bare field-level `@reference` on a multi-table interface/union child field is rejected: a single stated path applies the same hops to every participant, so it can be terminal-correct for at most one. `@referenceFor` is the sanctioned per-participant surface.

== Constraints

* Legal only on a child field returning a multi-table interface or union (table-backed and record-backed parents alike). On single-table (discriminated) polymorphic fields, non-polymorphic fields, and root fields the directive is rejected: there is no participant set to bind a path to.
* `type` must name a table-bound participant of the field's return type, case-sensitively. An unknown, non-member, or unbound name is rejected with the list of valid participant names.
* Declaring the same `type` on two applications of one field is a build error (applications are independent, not concatenated).
* The path must terminate on the named participant's table; a terminal-target mismatch is rejected naming the participant.
* Every shape the path grammar can state is emittable: a single-hop foreign key (multi-FK disambiguation, same-table self-FK), a multi-hop key chain through intermediate join tables, and a `{condition:}` correlation. A route that joins real tables bridges each intermediate hop back toward the parent and, for a `{condition:}` correlation, joins the parent table (aliased, bound to the parent's key) and applies the two-arg condition between the parent and the participant.
