= @value
:description: Marks an `@mutation(typeName: UPDATE)` input field as an assignment column rather than a WHERE filter.
:!toc:

`@value` partitions the input fields of an `@mutation(typeName: UPDATE)` `@table` input into the WHERE-side filter set and the SET-clause assignment set. Input fields default to filter; adding `@value` opts a field into the SET clause instead.

== SDL signature

[source,graphql]
----
directive @value on INPUT_FIELD_DEFINITION
----

`@value` takes no parameters; presence on an input field is the entire signal.

== Canonical example

[source,graphql]
----
input FilmUpdateInput @table(name: "film") {
    filmId:      Int!    @field(name: "film_id")
    title:       String! @field(name: "title") @value
    description: String  @field(name: "description") @value
}

type Mutation {
    updateFilm(in: FilmUpdateInput!): Film @mutation(typeName: UPDATE)
}
----

`filmId` is a WHERE filter (by default); `title` and `description` are SET-clause assignment columns (marked `@value`).

== Where it applies

`@value` is only valid on input fields of `@mutation(typeName: UPDATE)` `@table` inputs:

* `@mutation(typeName: DELETE)`: rejected. DELETE has no assignment clause.
* `@mutation(typeName: INSERT)`: rejected. INSERT does not partition its input fields; every input field is a value column for the INSERT VALUES list.
* `@mutation(typeName: UPSERT)`: rejected under the current cardinality-safety regime. UPSERT generation is deferred.

`@value` is mutually exclusive with `@condition` on the same input field.

== UPDATE invariants

The classifier enforces two structural invariants on UPDATE inputs:

* At least one input field must carry `@value`. Without it, the SET clause would be empty and the UPDATE has nothing to write.
* At least one input field must remain unmarked (WHERE filter). With every field `@value`-marked, UPDATE would have no WHERE clause and would update every row in the table.

== Cardinality safety

The WHERE-side fields (every admissible input field not marked `@value`) must collectively cover the underlying `@table`'s primary key. Without PK coverage, a single input row could match multiple database rows and the UPDATE would silently broadcast. Opt out of the PK-coverage check by setting `multiRow: true` on the surrounding `@mutation` directive; the opt-out documents that the broadcast is intentional.
